# `Send` 与 `Sync`

## 1. 这一章是干什么的

当你开始写 Rust 并发代码后，很快就会碰到两类报错或术语：

- `Send`
- `Sync`

很多人第一次见到它们时，会有一种很强的陌生感：

- 这不是我自己定义的 trait
- 但编译器老是在并发报错里提到它
- 好像每次一跨线程，它们就出来了

这一章的目标，就是把这两个 trait 从“神秘报错背景板”变成你真正能理解的线程安全规则。

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 多线程基础
- `Arc<T>`、`Mutex<T>`
- `Rc<T>`、`RefCell<T>`
- 所有权与移动语义

最好已经看过：

- [多线程、共享状态与 `channel`](./04-多线程共享状态与channel.md)

## 3. 先记住最核心的两句话

你可以先把它们这样记：

1. `Send`：这个类型的值能不能安全地移动到另一个线程
2. `Sync`：这个类型的引用能不能安全地被多个线程共享

这是理解它们最重要的第一层。

## 4. 为什么 Rust 要有这两个 trait

因为并发的核心问题之一就是：

- 某个值能不能跨线程传
- 某个值能不能被多个线程同时看

在很多语言里，这些问题大部分只能靠：

- 文档
- 经验
- 运行时事故

而 Rust 选择把它们做进类型系统。

也就是说：

- 线程安全不是“约定俗成”
- 而是类型的一部分

## 5. `Send` 先怎么理解

一个类型实现了 `Send`，可以先理解成：

- 它的所有权可以安全地从一个线程移交给另一个线程

例如：

```rust
use std::thread;

fn main() {
    let message = String::from("rust");

    let handle = thread::spawn(move || {
        println!("{}", message);
    });

    handle.join().unwrap();
}
```

这里 `String` 能被安全地 move 到另一个线程，背后就和 `Send` 有关。

## 6. `Sync` 先怎么理解

一个类型实现了 `Sync`，可以先理解成：

- 多个线程可以安全地共享它的不可变引用

这听起来抽象一点，你可以先把它粗略翻译成：

- “这个类型在共享只读访问时是线程安全的”

如果某种类型的只读引用本身就不安全，那它通常就不会是 `Sync`。

## 7. 为什么 `Rc<T>` 不是 `Send`

这是最经典的例子之一。

`Rc<T>` 的核心是：

- 单线程引用计数

它的引用计数更新不是为多线程竞争设计的。

所以如果你把 `Rc<T>` 跨线程传来传去，就可能出现：

- 引用计数更新竞争
- 状态不一致

因此 Rust 不允许它实现 `Send`。

这也是为什么你在并发场景里更常看到：

- `Arc<T>`

而不是：

- `Rc<T>`

## 8. 为什么 `RefCell<T>` 也不是并发默认答案

`RefCell<T>` 的重点是：

- 单线程运行时借用检查

它不是线程同步工具。

所以当你把它带进多线程场景时，编译器通常也不会轻易放行。

这能帮助你建立一个更清楚的判断：

1. `Rc<T>` / `RefCell<T>`：单线程世界
2. `Arc<T>` / `Mutex<T>`：多线程世界

## 9. `Arc<T>` 为什么常常没问题

因为 `Arc<T>` 本来就是为跨线程共享所有权设计的。

它的引用计数更新是线程安全的。

所以当你看到：

```rust
let shared = Arc::new(data);
```

你就应该本能想到：

- 这东西是在为跨线程共享做准备

## 10. `Mutex<T>` 为什么经常和线程安全连在一起

因为它解决的是：

- 多线程共享状态下，如何串行化可变访问

也就是说：

- `Arc<T>` 解决谁能共享拥有
- `Mutex<T>` 解决谁能安全修改

这就是为什么并发代码里经典组合总是：

- `Arc<Mutex<T>>`

## 11. `Send` 和 `Sync` 不是你日常随便手写的 trait

这是很重要的一点。

初学时不要把它们理解成普通业务 trait。

例如：

- `Display`
- `Summary`
- `Reporter`

这些 trait 更像业务能力接口。

而 `Send` / `Sync` 更像：

- 编译器和类型系统用来判断线程安全边界的标记

也就是说，它们更底层，也更强约束。

## 12. 一个非常实用的阅读方法

以后你看到并发报错时，如果提到了：

- `T` cannot be sent between threads safely

你应该先想到：

- 这里在说 `Send`

如果提到了类似共享引用不安全的问题，就更可能和：

- `Sync`

有关。

这样你至少能把“报错在讨论哪类线程安全问题”先定位出来。

## 13. 一个直觉例子：为什么有些值能进线程，有些不行

例如 `String` 经常可以：

- 直接 move 进线程

而 `Rc<String>` 往往不行。

核心差别不是“它们是不是字符串”，而是：

- 外层所有权模型是不是线程安全

这点非常值钱。

因为以后你读复杂类型时，也要先看最外层包装。

## 14. 一层一层看类型，判断线程安全

比如：

```rust
Arc<Mutex<Vec<String>>>
```

你可以这样拆：

1. `Arc`：跨线程共享所有权
2. `Mutex`：共享状态下互斥修改
3. `Vec<String>`：真正的数据本体

这类组合通常就是在显式构建一个：

- 可跨线程共享的可变状态

而如果你看到：

```rust
Rc<RefCell<Vec<String>>>
```

你就应该立刻想到：

- 这更像单线程共享可变状态，不适合直接跨线程

## 15. `Send` 和所有权移动的关系

`Send` 和前面学过的 move 是天然连着的。

因为线程里最常见的动作之一就是：

- 把值 move 进另一个执行流

所以一个类型如果不支持安全跨线程移动，它就很难自然参与线程任务启动。

这也是为什么 `thread::spawn(move || { ... })` 这种代码经常会把 `Send` 问题暴露出来。

## 16. `Sync` 和共享引用的关系

一个非常实用的粗略记忆是：

- 如果 `&T` 能安全地在线程间共享，那么 `T` 通常是 `Sync`

你现在不用死抠完整形式化定义，先建立这个阅读直觉就够了。

这会帮你理解：

- 为什么有些类型能被多个线程同时只读访问
- 为什么有些类型哪怕只是共享引用也会出问题

## 17. 为什么 Rust 能靠这套 trait 提前拦住很多并发 bug

因为它不是等你把程序跑起来之后才观察行为，而是直接在类型层面问：

1. 这个值能不能安全跨线程传？
2. 这个值能不能安全跨线程共享引用？
3. 这套组合外壳是不是线程安全？

这会让很多并发 bug 在“写代码阶段”就直接暴露。

这也是 Rust 并发模型最核心的优势之一。

## 18. 什么时候你最容易碰到 `Send` / `Sync` 报错

最常见就是下面几类场景：

1. `thread::spawn`
2. 把闭包丢进并发执行环境
3. 想让多个线程共用某个状态
4. 后面学 `async` 时，某些任务执行器也会间接涉及这些约束

所以你现在学会 `Send` / `Sync`，后面对 async 也会更稳。

## 19. 一个最重要的实战判断

如果你碰到并发类型报错，可以先按这个顺序排查：

1. 外层是不是用了 `Rc<T>`？
2. 有没有把 `RefCell<T>` 直接带进多线程？
3. 是不是本该用 `Arc<T>` 却还停留在单线程共享模型？
4. 共享可变状态是不是应该补 `Mutex<T>`？

这比死读整段报错更容易落地。

## 20. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

JavaScript 很多并发问题主要出现在 Worker、消息传递或共享内存场景，但语言层面不会像 Rust 这样把线程安全做成 trait。

Rust 的风格是：

- 把“能不能跨线程”变成类型问题

### 和 Python 类比

Python 里你更多是运行时才感受到线程或进程之间对象传递、共享和同步的问题。

Rust 则会更早、更直接地在编译期拦住很多不安全组合。

### 和 Java 类比

Java 背景会让你对：

- 线程安全
- 共享对象
- 锁
- 不可变对象

这些概念并不陌生。

但 Rust 的区别是：

- 它不只靠文档和类库约束
- 它直接把线程安全边界刻进类型系统

## 21. 常见错误与修正方向

### 错误 1：看见 `Send` / `Sync` 报错就只想“怎么糊过去”

更好的做法是先定位：

- 我现在到底在做跨线程移动，还是共享引用

### 错误 2：单线程模型直接硬搬到多线程

典型就是：

- `Rc<T>`、`RefCell<T>` 直接想跨线程用

### 错误 3：只记住 `Arc<Mutex<T>>`，却不知道它为什么成立

你要能解释：

1. `Arc` 解决共享拥有
2. `Mutex` 解决共享修改
3. 两者组合才形成线程安全共享状态

### 错误 4：把所有并发都改成共享锁

有时更自然的方案其实是：

- 消息传递
- `channel`

## 22. 一个很值钱的第一轮记忆法

你可以先这样背：

1. `Send`：值能安全过线程边界
2. `Sync`：引用能安全过线程边界
3. `Rc<T>` / `RefCell<T>`：更像单线程模型
4. `Arc<T>` / `Mutex<T>`：更像多线程模型

这组记忆已经足够支撑你读懂大量并发代码。

## 23. 小练习

1. 写一个 `thread::spawn(move || ...)` 最小例子，观察为什么 move 很常见。
2. 对比 `Rc<String>` 和 `Arc<String>` 在并发场景下的适用性。
3. 用自己的话解释：`Send` 和 `Sync` 分别在保护什么。
4. 回头看你之前写的 `Arc<Mutex<T>>` 示例，自己解释为什么它适合多线程。

## 24. 挑战题

做一个“并发计数器模型分析”练习，要求：

1. 设计一个单线程共享计数器版本
2. 设计一个多线程共享计数器版本
3. 对比解释：为什么前者可以用 `Rc<RefCell<T>>`，后者更适合 `Arc<Mutex<T>>`
4. 用自己的话说明这里的 `Send` / `Sync` 分别是如何影响设计选择的

额外要求：

1. 至少解释一次：为什么 `Rc<T>` 不能简单跨线程
2. 至少解释一次：为什么共享可变状态不仅仅是“能传过去”这么简单

## 25. 小结

这一章最重要的不是让你背 trait 名字，而是让你开始读懂 Rust 的线程安全语言：

1. `Send` 讨论的是跨线程移动所有权
2. `Sync` 讨论的是跨线程共享引用
3. 多线程类型设计本质上是在回应这两类约束
4. `Rc<T>` / `RefCell<T>` 和 `Arc<T>` / `Mutex<T>` 不是风格差异，而是线程安全边界差异
