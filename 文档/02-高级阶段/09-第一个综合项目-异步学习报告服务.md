# 第一个综合项目：异步学习报告服务

## 1. 这一章是干什么的

到了高级阶段，如果你一直只看单点专题，会出现一种新的瓶颈：

- 单篇文档我都能看懂
- 但真让我把 async、并发、生态库、错误边界、项目结构组合起来，我还是会发懵

所以这一章的目标很明确：

- 用一个更像真实工程的项目，把高级阶段前半段真正收束起来

这次项目不再只是简单 CLI，也不只是单文件小工具，而是一个更接近真实业务的小型服务型工具：

- 异步学习报告服务

它会同时碰到：

1. `clap` 参数解析
2. `serde` 配置建模
3. `tokio` 运行时
4. `reqwest` 异步请求
5. `tracing` 日志
6. `thiserror` / `anyhow` 错误边界
7. 异步任务并发执行
8. 多模块工程结构

这正好是高级阶段最值得做的第一批综合项目类型。

## 2. 学这一章前，你需要知道什么

建议你已经看过：

- 对线程、channel、`Send` / `Sync` 有第一轮直觉

## 3. 这个项目要解决什么问题

先把项目目标说清楚。

我们设计一个：

- 异步学习报告服务

它做的事情可以理解成：

1. 从配置或命令行获取多个学习数据源
2. 并发拉取这些数据
3. 统一解析成结构化模型
4. 做汇总统计
5. 输出一份学习报告

你可以把它类比成：

- 一个轻量版“异步数据抓取 + 报告聚合”工具

它比前面的学习记录分析器更进一步的地方在于：

- 数据不再只是本地文本
- 项目不再只是同步流程
- 生态库不再只是点缀，而是真正进入边界设计

## 4. 为什么这个项目很适合高级阶段

因为它恰好逼你面对几个高级阶段的代表性问题：

### 4.1 async 不是装饰

你要真正把：

- `async fn`
- `.await`
- 并发请求

放进项目流程里理解。

### 4.2 生态库开始真正组合

这里的 `clap`、`serde`、`tokio`、`reqwest`、`tracing` 不再是分散知识点，而是同一个项目里的不同边界工具。

### 4.3 项目结构必须更清晰

如果你还把逻辑都塞进 `main.rs`，这个项目会立刻变得难读。

### 4.4 错误边界会变得更真实

因为你会同时面对：

- 参数错误
- 配置错误
- 网络错误
- 业务规则错误

## 5. 先定义一个可控范围

第一版项目，不要试图做成完整生产系统。

先把边界定清楚：

### 第一版目标

1. 支持命令行指定配置文件
2. 配置里包含多个数据源 URL
3. 并发请求所有数据源
4. 每个数据源返回 JSON 学习数据
5. 统一汇总总时长、课程数、平均分
6. 最终输出 Markdown 或终端文本报告

### 第一版先不做

1. 认证体系
2. 数据库存储
3. Web 后台界面
4. 完整重试和熔断

高级阶段项目也要先会控边界。

## 7. 每一层分别负责什么

### `interface`

负责：

- CLI 输入
- 最终输出展示

### `app`

负责：

- 把整个用例串起来

### `domain`

负责：

- 学习数据模型
- 汇总逻辑
- 报告构建逻辑

### `infra`

负责：

- 网络请求
- 远程数据获取

这个分层非常重要，因为它会决定：

- 生态库放在哪一层
- 错误边界放在哪一层

## 8. 配置应该怎么设计

先从一个足够实用的结构体开始：

```rust
use serde::Deserialize;

#[derive(Debug, Deserialize)]
pub struct Config {
    pub output_format: String,
    pub sources: Vec<SourceConfig>,
}

#[derive(Debug, Deserialize)]
pub struct SourceConfig {
    pub name: String,
    pub url: String,
}
```

这里 `serde` 的价值会非常直接：

- 外部配置可以自然映射成结构体

这比自己手写字符串解析稳很多。

## 9. CLI 边界为什么适合 `clap`

例如：

```rust
use clap::Parser;

#[derive(Debug, Parser)]
pub struct CliArgs {
    #[arg(short, long, default_value = "config.toml")]
    pub config: String,
}
```

这一层的职责非常清楚：

- 从进程参数里读配置入口

注意这里的 `clap` 应该主要留在：

- `interface/cli.rs`

而不是渗透进核心业务层。

## 10. 远程数据模型怎么设计

假设远程接口返回：

```json
{
  "subject": "rust",
  "score": 95,
  "hours": 3.5
}
```

那么 domain 层可以先有：

```rust
#[derive(Debug, Clone, serde::Deserialize)]
pub struct StudyItem {
    pub subject: String,
    pub score: u32,
    pub hours: f64,
}
```

这里要开始形成一个好习惯：

- 结构体不是只为“能跑起来”存在
- 它是项目数据边界的一部分

## 11. `infra/http_client.rs` 该干什么

它不该承担整个业务流程，只该负责：

- 发请求
- 拿响应
- 转成基础数据对象

例如：

```rust
use crate::domain::model::StudyItem;
use crate::error::InfraError;

pub async fn fetch_source(url: &str) -> Result<Vec<StudyItem>, InfraError> {
    let response = reqwest::get(url).await?;
    let items = response.json::<Vec<StudyItem>>().await?;
    Ok(items)
}
```

这段代码同时体现了：

1. `tokio` 运行时背景下的异步请求
2. `reqwest` HTTP 客户端
3. `serde` JSON 反序列化

## 12. 为什么这里的错误不要直接用字符串

因为网络层的失败信息很有技术来源。

例如：

- 连接失败
- 响应格式不对
- 远端超时

更合理的设计通常是：

```rust
#[derive(thiserror::Error, Debug)]
pub enum InfraError {
    #[error("http request failed: {0}")]
    Request(#[from] reqwest::Error),
}
```

这样错误来源就不会糊掉。

## 13. `app/run.rs` 才是异步主流程的核心

这里适合做的事情通常是：

1. 读取配置
2. 启动多个异步请求
3. 等待结果汇总
4. 交给 domain 做统计
5. 交给 interface 做输出

这层才真正像“一个用例的编排者”。

## 14. 并发请求怎么组织

初学高级 Rust 时，你不一定要一开始就背很多组合器。

先建立一个最核心的思路：

- 每个数据源产生一个异步任务
- 再把多个任务一起等待

例如伪代码层面可以先理解成：

```rust
for source in config.sources {
    // 为每个 source 构造一个 future
}

// 等待所有 future 完成
```

真实项目里，这一步经常会逐步接触：

- `join_all`
- `tokio::spawn`
- 异步 stream

但初学阶段先把“并发任务汇总”这个概念站住最重要。

## 15. domain 层该做什么

domain 层最该做的是：

- 不关心 CLI
- 不关心 HTTP
- 不关心具体配置来源

它只关心：

- 我拿到一批学习数据
- 怎么聚合成最终业务结果

例如：

```rust
pub struct StudySummary {
    pub total_subjects: usize,
    pub total_hours: f64,
    pub average_score: f64,
}
```

然后：

```rust
pub fn summarize(items: &[StudyItem]) -> StudySummary {
    // 纯逻辑聚合
}
```

这一层越纯，测试就越好补。

## 16. 输出层为什么最好独立

因为终端输出和 Markdown 输出，本质上是：

- 同一个业务结果的不同展示形式

这很像你前面学 trait object 时的一个天然应用点。

例如可以定义：

```rust
trait OutputFormatter {
    fn format(&self, summary: &StudySummary) -> String;
}
```

然后有：

1. `ConsoleFormatter`
2. `MarkdownFormatter`

这样就会把高级阶段前面的知识自然串起来。

## 17. `tracing` 应该放在哪

日志不应该到处乱打。

更稳的思路通常是：

1. 在入口层初始化日志系统
2. 在 app / infra 关键边界打结构化日志
3. 不要在每个纯逻辑函数里到处 `println!`

例如：

- 拉取某个 source 前打一条 debug/info
- 请求失败时记上下文
- 汇总完成时记结果概览

这会让项目更像真实服务。

## 18. 这个项目为什么能串起高级阶段主线

因为你会在同一个项目里真正同时碰到：

1. async/await
2. 生态库协作
3. 错误边界
4. 模块分层
5. 统一输出接口

也就是说，这不再是“会几个单点知识”，而是：

- 真正开始写像样的 Rust 工程

## 19. 哪些地方最值得写测试

高级阶段项目不是所有层都要同样测。

优先级最高的一般是：

### `domain` 层

- `summarize()`
- 报告构建逻辑

这部分最纯，测试收益最高。

### `config` 层

- 配置反序列化
- 参数边界

### `infra` 层

这里更适合：

- mock
- 本地测试服务
- 集成测试

而不是一上来就全打真网络请求。

## 20. 一个非常值钱的习惯：把第三方库包在边界里

例如：

- `clap` 放在 `interface/cli`
- `reqwest` 放在 `infra/http_client`
- `serde` 放在配置和 DTO 模型边界
- `tracing` 主要在入口和跨层调用点

这会让核心业务逻辑更干净，也更方便以后替换库或做测试。

## 21. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

这很像一个 Node.js 的异步数据抓取 CLI / 服务：

- 配置文件
- HTTP 客户端
- 并发请求
- 日志
- 输出格式化

Rust 的差别在于：

- async 模型、错误边界、类型建模会更显式

### 和 Python 类比

你可以把它想成一个异步数据清洗与聚合工具，只是 Rust 会强迫你把：

- 结构体
- 错误边界
- 模块边界

都设计得更清楚。

### 和 Java 类比

Java 背景会让你很快理解：

- app/service/infra 分层
- 配置对象
- DTO
- 客户端层
- 日志和错误边界

但 Rust 更偏：

- 小而清晰的模块组合
- `Result` 驱动的错误流

## 22. 常见错误与修正方向

### 错误 1：项目一上来全写在 `main.rs`

这会直接浪费你前面高级阶段学到的大量内容。

### 错误 2：把 `reqwest`、`serde`、`clap` 到处传染到所有模块

更好的做法是把它们留在边界层。

### 错误 3：错误类型只靠 `String`

异步和网络项目里，这会让排错成本很高。

### 错误 4：纯业务层也大量依赖 runtime 细节

domain 层应该尽量保持纯逻辑，不要被运行时和外部 I/O 绑死。

## 23. 一个推荐的实现顺序

如果你自己动手做这个项目，建议按这个顺序：

1. 先定义配置结构
2. 再定义业务模型
3. 再写 `infra` 请求层
4. 再写 `domain` 汇总逻辑
5. 再写输出格式化
6. 最后写 `app::run` 和 `main`

这个顺序能保证：

- 边界先站住
- 流程再串起来

## 25. 挑战题

把这个项目继续升级一版：

1. 支持多个远程数据源并发抓取
2. 支持失败源统计，但其余源继续完成
3. 输出最终成功源数、失败源数、平均分和总时长
4. 日志里记录每个数据源的抓取状态
5. 给 domain 层补至少 3 个测试

额外要求：

1. 至少解释一次：哪些错误适合中断整个流程，哪些错误更适合局部记录后继续
2. 至少解释一次：为什么某些生态库要包在边界层
3. 至少解释一次：为什么输出格式适合抽象成接口或策略
