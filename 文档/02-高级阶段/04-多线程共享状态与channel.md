# 多线程、共享状态与 `channel`

## 1. 这一章是干什么的

前面你已经开始进入 Rust 高级阶段，也已经碰过：

- 智能指针体系
- `Rc<T>` / `Arc<T>`
- `RefCell<T>` / `Mutex<T>`
- trait object

接下来最自然的下一步，就是把这些能力真正带进并发场景。

因为很多高级 Rust 代码的复杂度，往往不是来自单线程逻辑，而是来自：

1. 多个线程如何协作
2. 数据到底该共享还是传递
3. 共享状态应该用什么模型保护
4. 为什么有时 `channel` 比锁更自然

这一章的目标，就是建立你对 Rust 多线程编程的第一轮整体直觉。

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 所有权与借用
- `Arc<T>`、`Mutex<T>`
- `Rc<T>`、`RefCell<T>`
- 多模块项目的基本结构

最好已经看过：

- [智能指针体系](./01-智能指针体系.md)
- [内部可变性与 `RefCell<T>`](./02-内部可变性与RefCellT.md)
- [trait object 与动态分发](./03-trait-object与动态分发.md)

## 3. Rust 的并发直觉和很多语言不一样

很多语言里的并发体验通常是：

- 先把线程跑起来
- 再在运行时慢慢发现共享状态、锁竞争、数据竞争的问题

Rust 则更偏向：

- 先把线程边界上的数据所有权和线程安全讲清楚
- 编译器再允许你真正跑起来

所以学 Rust 并发时，第一件事不是先背 API，而是先接受一个思维变化：

- 并发代码不是“普通代码 + 开线程”
- 而是“所有权模型进入了跨线程场景”

## 4. 先从最小线程例子开始

```rust
use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        println!("hello from thread");
    });

    handle.join().unwrap();
}
```

这段代码里有两个最重要的点：

1. `thread::spawn(...)` 开启了一个新线程
2. `join()` 表示等待这个线程执行完

如果你只先记一件事，那就是：

- 线程启动后就有了新的执行流

## 5. `join()` 为什么重要

因为主线程不会天然等你所有子线程自己结束。

例如：

```rust
use std::thread;

fn main() {
    thread::spawn(|| {
        println!("hello from thread");
    });
}
```

这类代码可能会出现：

- 主线程很快结束
- 子线程还没来得及完整执行

所以 `join()` 的第一轮直觉可以先记成：

- 把线程结果收回来，并等待它完成

## 6. 线程闭包为什么经常会遇到 `move`

看一个常见写法：

```rust
use std::thread;

fn main() {
    let message = String::from("rust");

    let handle = thread::spawn(move || {
        println!("{}", message);
    });

    handle.join().unwrap();
}
```

这里的 `move` 很关键。

因为线程闭包一旦启动，就可能比当前作用域活得更久。

编译器需要确保：

- 子线程里用到的数据不会指向已经失效的外部变量

所以它经常要求你：

- 把需要的数据所有权移动进线程

这和你前面学闭包时的 `move` 是同一件事，只是现在场景变成了多线程。

## 7. 为什么不能随便把普通借用传进线程

因为线程是一个更独立的执行边界。

假设你只是借用了外部变量：

- 主线程可能先结束
- 外部变量可能先失效
- 子线程却还在尝试访问它

Rust 在编译期就会尽量防住这种问题。

所以你可以先记住这个并发直觉：

- 线程边界上，编译器会比单线程场景更强烈地要求所有权清晰

## 8. 两类最核心的并发模型

Rust 入门并发时，先把模型分成两大类就够了：

### 8.1 共享状态模型

几个线程共同访问一份数据。

典型工具：

- `Arc<T>`
- `Mutex<T>`

### 8.2 消息传递模型

线程之间不直接共享数据，而是互相发送消息。

典型工具：

- `channel`

这两类模型在 Rust 里都非常重要。

## 9. 共享状态模型的最小例子

例如计数器：

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..5 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut value = counter.lock().unwrap();
            *value += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("{}", *counter.lock().unwrap());
}
```

这里其实同时发生了两层事：

1. `Arc<T>` 负责跨线程共享所有权
2. `Mutex<T>` 负责保证一次只有一个线程能改值

## 10. 为什么这里不能用 `Rc<RefCell<T>>`

这是你必须建立的关键判断。

原因不是“语法不支持”，而是：

- `Rc<T>` 不是线程安全的
- `RefCell<T>` 也不是为多线程同步设计的

它们更适合：

- 单线程共享可变状态

而一旦进入多线程，Rust 更自然的共享可变状态模型通常就是：

- `Arc<Mutex<T>>`

## 11. `Mutex<T>` 里的 `lock()` 在做什么

你可以先把它理解成：

- 申请一次独占访问权

例如：

```rust
let mut guard = counter.lock().unwrap();
*guard += 1;
```

这里 `guard` 很像一把临时的“钥匙”：

- 钥匙拿到手，才能访问里面的值
- 钥匙离开作用域，锁自动释放

这也是为什么作用域在并发代码里会变得更重要。

## 12. 为什么锁的作用域应该尽量短

因为锁住共享状态的时间越长：

1. 其他线程等待越久
2. 代码越容易绕
3. 死锁和性能问题越容易出现

所以一个很实用的习惯是：

- 只在真正需要读写共享状态的那小段时间持有锁

例如：

```rust
{
    let mut value = counter.lock().unwrap();
    *value += 1;
}

// 锁已经释放
```

## 13. 消息传递模型：`channel`

Rust 另一条非常重要的并发主线是：

- 不共享数据，而是传递数据

最常见的入口就是：

- `std::sync::mpsc`

这里的 `mpsc` 可以先粗略记成：

- 多生产者，单消费者

## 14. `channel` 的最小例子

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        tx.send(String::from("hello")).unwrap();
    });

    let received = rx.recv().unwrap();
    println!("{}", received);
}
```

这里最重要的角色是：

1. `tx`：发送端
2. `rx`：接收端

你可以先把它想成：

- 一个线程把数据塞进通道
- 另一个线程从通道里取出来

## 15. `send()` 和所有权关系

这是 `channel` 特别值得注意的一点。

例如：

```rust
tx.send(String::from("hello")).unwrap();
```

这里发送的往往不是“借用”，而是：

- 把值所有权发送过去

也就是说，消息传递模型很符合 Rust 的一条核心理念：

- 与其共享并争抢同一份数据，不如把数据所有权明确地交给下游线程

这也是为什么很多 Rust 并发设计会显得比别的语言更“清楚”。

## 16. 多条消息的接收

你可以连续 `send()` 多次：

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        tx.send(String::from("a")).unwrap();
        tx.send(String::from("b")).unwrap();
        tx.send(String::from("c")).unwrap();
    });

    for message in rx {
        println!("{}", message);
    }
}
```

这里的一个很值钱的直觉是：

- `rx` 也可以像迭代器一样不断接收消息

这会让你感觉 Rust 的并发模型和它的数据流、迭代器风格其实是连着的。

## 17. 为什么 `channel` 往往比锁更容易讲清边界

因为 `channel` 更偏向：

- 数据所有权流动

而不是：

- 多个人一起抓着一份共享状态反复改

所以一个很实用的判断是：

### 更适合共享状态的场景

- 多个线程必须持续看到和修改同一份状态
- 共享状态本身是核心模型

### 更适合消息传递的场景

- 线程更像流水线上的不同处理阶段
- 数据天然就是“产出后交给下游”
- 你想降低共享状态复杂度

## 18. 一个更真实的小例子：任务分发

假设你有一批字符串任务要处理。

```rust
use std::sync::mpsc;
use std::thread;

fn main() {
    let tasks = vec!["rust", "python", "java"];
    let (tx, rx) = mpsc::channel();

    for task in tasks {
        let tx = tx.clone();
        thread::spawn(move || {
            let result = format!("done: {}", task);
            tx.send(result).unwrap();
        });
    }

    drop(tx);

    for message in rx {
        println!("{}", message);
    }
}
```

这里有几个很重要的点：

1. `tx.clone()` 允许多个线程都发送消息
2. 每个线程各自处理一个任务
3. 主线程统一收集结果
4. `drop(tx)` 用来显式关闭主发送端，避免接收端永远等下去

## 19. `drop(tx)` 为什么有时很关键

因为接收端会认为：

- 只要还有发送端活着，后面就可能还有新消息

如果主线程自己还留着一个不用的 `tx`，那 `for message in rx` 这种循环可能就不会自然结束。

所以在多发送端场景里，常见做法是：

- 把不再需要的发送端显式丢掉

这是一种非常 Rust 风格的“生命周期边界显式化”。

## 20. 共享状态 vs 消息传递，真正怎么选

很多并发设计难点不在语法，而在选模型。

你可以先按下面顺序判断：

1. 数据是必须被多个线程持续共享吗？
   如果是，考虑共享状态模型
2. 数据更像一个阶段性结果，要从 A 流向 B 吗？
   如果是，优先考虑 `channel`
3. 如果能少共享状态，就尽量少共享状态

这不是绝对规则，但非常实用。

因为共享状态一多，锁、竞争、边界、死锁风险都会跟着上来。

## 21. 多线程里最常见的几类错误

### 错误 1：一上来就把所有东西都塞进 `Arc<Mutex<T>>`

这通常会让模型变得又重又乱。

先想一想：

- 这个问题能不能改成消息传递

### 错误 2：锁作用域太长

这会让并发性能变差，也会增加代码复杂度。

### 错误 3：忘了线程边界上的所有权

很多编译报错本质上都是：

- 你想把借用带进线程
- 但线程要求更清楚的拥有关系

### 错误 4：发送端没关，接收端一直等

这在 `channel` 场景里非常常见。

## 22. 为什么 Rust 的并发会显得“啰嗦但靠谱”

因为它几乎在强迫你做这些事：

1. 先想清楚谁拥有数据
2. 先想清楚是共享还是传递
3. 先想清楚线程结束时边界在哪
4. 先想清楚修改权限怎么控制

在 JavaScript、Python 里，这些问题常常是运行起来后才暴露。

Rust 则更希望你在设计时就把它们讲清楚。

## 23. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

JavaScript 主流并发更偏事件循环、消息队列、Worker 之类模型。

Rust 的 `channel` 会让你感觉有点像更底层、更显式的消息传递。

### 和 Python 类比

Python 里多线程和多进程经常会让人陷入：

- 共享对象怎么同步
- 队列怎么传数据

Rust 也面对同样问题，但会在类型层面更强地约束你。

### 和 Java 类比

Java 背景会让你对：

- `Thread`
- 锁
- 共享状态
- 生产者消费者模型

这些东西比较熟。

但 Rust 会比 Java 更强调：

- 线程边界上的所有权
- 哪些类型真的线程安全

## 24. 一个很实用的最小模板

如果你现在做一个基础并发工具，可以先这样分层：

1. 主线程负责准备输入
2. 工作线程负责处理任务
3. 结果通过 `channel` 汇总回来
4. 只有真的必须共享的状态，才放进 `Arc<Mutex<T>>`

这套模板比“一开始到处上共享锁”稳很多。

## 25. 小练习

1. 写一个最小线程例子，用 `join()` 等待它完成。
2. 写一个 `Arc<Mutex<i32>>` 计数器，启动多个线程累计计数。
3. 写一个 `channel` 例子，让子线程发送三条消息给主线程。
4. 比较一下：同一个计数任务，为什么共享状态模型和消息传递模型写法不同。

## 26. 挑战题

做一个“并发任务处理器”练习，要求：

1. 主线程准备一组字符串任务
2. 多个子线程分别处理任务
3. 每个线程把处理结果通过 `channel` 发回主线程
4. 主线程最终汇总所有结果
5. 额外再设计一个需要共享统计计数器的版本

额外要求：

1. 至少使用一次 `Arc<Mutex<T>>`
2. 至少使用一次 `channel`
3. 至少解释一次：为什么这个部分更适合共享状态
4. 至少解释一次：为什么这个部分更适合消息传递

## 27. 小结

这一章最重要的不是记住几个 API，而是建立并发模型直觉：

1. 线程边界会放大所有权问题
2. 共享状态模型和消息传递模型是两条核心主线
3. `Arc<Mutex<T>>` 适合共享可变状态
4. `channel` 适合在线程间转移数据所有权
5. 如果能少共享状态，就尽量少共享状态
