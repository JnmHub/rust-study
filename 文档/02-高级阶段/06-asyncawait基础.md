# async/await 基础

## 1. 这一章是干什么的

很多人学 Rust 到并发阶段后，会碰到另一类新的陌生区：

- `async`
- `await`
- `Future`
- 执行器

而且很容易产生一个误解：

- async 不就是“线程的另一种写法”吗？

这其实不对。

这一章的目标，就是把 Rust 里 async/await 的第一轮心智模型建立起来，让你先真正分清：

1. async 和线程不是一回事
2. `async fn` 到底返回了什么
3. `.await` 在做什么
4. 为什么 async 代码的设计方式和同步代码不一样
5. 它和 `Send` / `Sync`、所有权、借用为什么又会重新连起来

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 基础函数与返回值
- 多线程和 `channel`
- `Send` / `Sync`
- 所有权、借用、生命周期的基础直觉

最好已经看过：

- [多线程、共享状态与 `channel`](./04-多线程共享状态与channel.md)
- [`Send` 与 `Sync`](./05-Send与Sync.md)

## 3. 先把 async 和线程彻底分开

这是学 async 最重要的第一步。

### 线程

你可以先理解成：

- 操作系统级或更重量级的并发执行流

### async

你可以先理解成：

- 一种“任务可以暂停、让出、以后再继续”的协作式并发模型

也就是说：

- 线程更像“真正同时跑的执行流”
- async 更像“一个任务在合适的时候挂起，等条件满足后再恢复”

所以 async 不等于“开更多线程”。

## 4. `async fn` 到底返回了什么

看一个最小例子：

```rust
async fn hello() -> String {
    String::from("hello")
}
```

这段代码最容易被误解成：

- 调用 `hello()` 就立刻开始执行，并直接返回 `String`

其实更准确的理解是：

- `hello()` 返回的是一个 future

也就是说，它返回的不是最终结果本身，而是：

- “一个未来会产出这个结果的异步任务描述”

## 5. 什么是 `Future`

初学时你不用立刻啃标准库 trait 定义。

你可以先把 `Future` 粗略理解成：

- 一种代表“未来某个时刻会完成并产出结果”的值

这和你前面学过的很多 Rust 值很不一样。

因为它不是“现在就有答案”，而是：

- 以后会有答案

## 6. `.await` 在做什么

例如：

```rust
async fn hello() -> String {
    String::from("hello")
}

async fn run() {
    let result = hello().await;
    println!("{}", result);
}
```

初学时你可以把 `.await` 先粗略理解成：

- “等待这个 future 继续推进，直到拿到结果”

更具体一点的说法是：

- 这里允许当前异步任务在等待期间挂起
- 等 future 准备好后，再继续往下执行

## 7. 为什么 `.await` 不是“阻塞一整条线程”的同义词

这点非常关键。

同步世界里的等待常常意味着：

- 线程卡住
- 什么都不干

而 async 世界里的等待，更像：

- 这个任务先暂停
- 让执行器去跑别的就绪任务
- 以后条件满足了再回来继续跑

也就是说，async 的核心优势通常来自：

- 更高效地利用等待时间

尤其是在 I/O 场景里更明显。

## 8. 为什么 async 特别适合 I/O 密集场景

例如：

- 网络请求
- 文件读取
- 数据库访问
- 定时等待

这类任务经常不是 CPU 算不过来，而是：

- 大量时间都在等外部资源

如果每次等待都占着一条线程不放，成本会更高。

async 的思路则是：

- 等的时候先挂起
- 让执行器去调度别的任务

所以 async 的强项通常不是：

- 更快地做纯 CPU 运算

而是：

- 更有效率地处理大量等待型任务

## 9. 为什么 Rust 里的 async 看起来比别的语言更“显式”

因为 Rust 不太喜欢隐藏代价。

你会明显看到：

1. 哪个函数是 `async fn`
2. 哪个调用点用了 `.await`
3. 哪些值跨过了 await 边界
4. 哪些任务需要执行器来跑

这和很多语言那种“看起来很像普通代码，但底层已经切换模型”的风格不太一样。

## 10. 执行器是什么

如果你现在只想先建立直觉，可以把执行器理解成：

- “负责真正驱动 async 任务往前跑的运行环境”

也就是说：

- `async fn` 只是生成 future
- 不是自己就能神奇跑起来

某个执行器需要去 poll / 调度这些 future。

在真实项目里，这通常就是：

- `tokio`
- `async-std`

等运行时生态的一部分。

## 11. 为什么标准库 async 例子常和运行时库绑在一起

因为：

- Rust 语言层面支持 async/await
- 但真正跑大量异步任务，通常要依赖运行时生态

这也是为什么你后面学 async 时，很自然会碰到：

- `tokio`

所以 async 不只是语法点，它也会把你带进生态层。

## 12. 一个非常重要的区别：同步函数 vs async 函数

### 同步函数

```rust
fn read_data() -> String {
    String::from("data")
}
```

调用时：

- 会立刻执行
- 直接返回结果

### async 函数

```rust
async fn read_data() -> String {
    String::from("data")
}
```

调用时：

- 返回 future
- 不是立刻把最终结果给你

这个区别必须牢牢记住。

## 13. 为什么 async 会重新引出所有权问题

因为 future 不是“立刻跑完”的。

它可能：

1. 执行一部分
2. 在 `.await` 处挂起
3. 稍后再恢复

这意味着编译器必须非常认真地判断：

- 哪些值会跨过 await 边界继续存活
- 哪些借用在挂起期间是否仍然有效

所以 async 不是绕开所有权，而是：

- 把所有权和生命周期问题带进了新的执行模型

## 14. 为什么 async 里会再次碰到 `Send`

这是后面非常常见的报错来源。

如果某个异步任务可能被调度到不同线程的执行器上，那运行时通常会要求：

- 这个 future 是 `Send`

也就是说：

- future 内部捕获的状态
- 也得能安全跨线程移动

所以你前一篇学的 `Send` / `Sync`，在 async 里并不是过去式，而是会重新出现。

## 15. 一个典型直觉：不是所有 async 都天然线程安全

这点非常重要。

很多人会误以为：

- async 比线程“轻”
- 所以它就没那么多线程安全约束

不对。

如果你的异步任务会在线程池执行器里跑，那里面捕获的内容一样可能需要：

- `Send`

所以 async 不是线程安全规则的豁免区。

## 16. async 和 `channel` 的关系

前一篇你已经见过线程间消息传递。

async 世界里也同样非常强调：

- 用消息传递组织任务边界

因为 async 场景里你也会经常面对：

1. 任务 A 产出数据
2. 任务 B 消费数据
3. 多个任务之间需要解耦

所以：

- “共享状态”
- “消息传递”

这两条主线在 async 世界里仍然成立，只是工具生态会稍有不同。

## 17. async 和多线程是什么关系

你可以先建立一个粗略但实用的判断：

1. 多线程关注的是“执行流如何分布在多个线程上”
2. async 关注的是“任务如何在等待时挂起和恢复”

它们不是互斥关系。

真实项目里经常是：

- 一个多线程执行器
- 调度很多异步任务

所以 async 和线程不是“二选一”的关系，而是：

- 可能叠加

## 18. 一个很常见的设计误区

### 误区：只要是并发，就应该立刻上 async

不对。

如果你的任务是：

- CPU 密集型
- 算法型
- 没太多 I/O 等待

那 async 可能就不是最自然的第一选择。

反过来，如果你的任务是：

- 网络 I/O
- 高并发请求
- 等待很多外部资源

那 async 才更容易体现优势。

## 19. 为什么 async 代码的风格和同步代码不一样

因为你必须开始认真面对：

1. 哪些函数需要 async 化
2. 哪些边界要传播 future
3. 哪些逻辑最好保持同步纯函数
4. 哪些 I/O 或等待操作要交给 async 世界

一个很实用的习惯是：

- 纯计算逻辑尽量保持同步、纯函数化
- 真正需要等待外部资源的边界再做 async

这会让代码结构更稳。

## 20. 一个很好的 async 学习切入点

你现在还不需要一上来就写大型异步服务。

更好的切入点通常是：

1. 先理解 `async fn` 返回 future
2. 理解 `.await`
3. 理解为什么要执行器
4. 再做一个小的异步 I/O 工具

这个学习顺序会比一头扎进框架更不容易崩。

## 21. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过：

- `async function`
- `await`
- Promise

那你会很快接受“future 会在未来完成”的思路。

但 Rust 的差别是：

- 所有权和生命周期会更显式地影响 async 代码设计

### 和 Python 类比

Python 的 `asyncio` 也会让你接触：

- 协程
- await
- 事件循环

Rust 的不同点在于：

- 类型更严格
- 任务边界更显式
- 线程安全约束更可能直接出现在类型报错里

### 和 Java 类比

Java 里你可能见过：

- `CompletableFuture`
- 回调链
- Reactor / 异步框架

Rust async 的体验也带有“未来结果”的味道，但它会更直接地把所有权、借用和任务边界连接起来。

## 22. 常见错误与修正方向

### 错误 1：把 async 当“线程的轻量替代品”

它们解决的问题重叠但不相同，别混为一谈。

### 错误 2：一看到 `.await` 就以为等于同步阻塞

async 等待的关键价值恰恰在于：

- 任务能挂起，让出执行机会

### 错误 3：纯计算逻辑一股脑 async 化

这会让代码复杂度上升，但未必带来收益。

### 错误 4：忽略 future 内部状态也会受到 `Send` / `Sync` 影响

这会让你后面在真实运行时中遇到并发约束报错时完全没方向。

## 23. 一个很实用的第一轮记忆法

你可以先这样记：

1. `async fn` 返回 future
2. `.await` 用来等待 future 推进到结果
3. async 主要擅长大量等待型任务
4. async 不等于线程
5. async 也会重新碰到所有权与线程安全问题

## 24. 小练习

1. 写一个最小 `async fn`，返回一个字符串。
2. 用自己的话解释：为什么调用 async 函数不会立刻得到最终结果。
3. 写下你对线程和 async 的三条差别理解。
4. 回想你之前的 CLI 项目，想一想里面哪些操作如果变成网络请求，会更适合 async。

## 25. 挑战题

做一个“异步日志抓取器”设计练习，要求：

1. 列出它会有哪些等待型操作
2. 分清哪些逻辑适合同步纯函数，哪些边界适合 async
3. 解释如果它运行在多线程执行器里，为什么 future 里的状态可能还会受到 `Send` 约束
4. 比较：如果改成纯线程模型，设计会发生什么变化

额外要求：

1. 至少解释一次：为什么 async 更适合 I/O 密集任务
2. 至少解释一次：为什么 async 不是自动更快
3. 至少解释一次：为什么 async 会把你带进运行时生态

## 26. 小结

这一章最重要的不是会写多少 async 语法，而是先把模型分清：

1. async 不是线程
2. `async fn` 返回的是 future
3. `.await` 是等待 future 推进
4. async 特别适合等待型任务
5. 所有权、`Send`、任务边界会在 async 世界里重新出现
