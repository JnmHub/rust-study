# 内部可变性与 `RefCell<T>`

## 1. 这一章是干什么的

前一篇你已经建立了智能指针体系的第一轮地图，也已经知道：

- `Rc<T>` 解决单线程共享所有权
- `Arc<T>` 解决跨线程共享所有权
- `RefCell<T>` 和 `Mutex<T>` 解决“共享状态下怎么修改”的问题

但如果你只停在这个层面，后面一碰到 `RefCell<T>` 还是会容易发怵。

因为它最让人不安的地方在于：

- Rust 不是一直强调编译期借用检查吗？
- 为什么这里突然允许把一部分检查放到运行时？

这一章就是要把这个问题真正讲清楚。

它的核心目标是：

1. 让你明白什么叫“内部可变性”
2. 让你看清 `RefCell<T>` 和 `&mut T` 的本质差别
3. 让你知道它为什么经常和 `Rc<T>` 一起出现
4. 让你知道什么时候该想到它，什么时候不该滥用它

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 所有权与借用
- `&T` 和 `&mut T`
- `Rc<T>`、`Arc<T>`、`Mutex<T>` 的基本定位
- `Result<T, E>` 与作用域

最好已经看过：

- [高级阶段导读](./高级阶段导读.md)
- [智能指针体系](./01-智能指针体系.md)

## 3. 什么叫“内部可变性”

先别急着看 `RefCell<T>` 的语法。

先记住一句话：

- 内部可变性 = 外部看起来是不可变持有，内部仍然允许修改

这听起来有点反直觉，因为你前面学 Rust 时已经习惯了：

- 想改值，就要有 `mut`
- 想改借来的值，就要拿到 `&mut T`

内部可变性则是在说：

- 有些类型会把“是否允许修改”的控制逻辑包进自己内部

也就是说，“能不能改”不再只由外部绑定的 `mut` 决定，还会由这个类型自己的规则决定。

## 4. 为什么 Rust 会需要内部可变性

因为有些场景在逻辑上完全合理，但编译期借用规则不容易直接表达。

典型情况包括：

1. 多个地方共享一份数据，但你只在非常短的时刻修改它
2. 某些对象需要记录内部状态，比如缓存、计数器、测试桩
3. 你的 API 对外想保持不可变接口，但内部需要做一些受控更新

换句话说，内部可变性不是为了绕过 Rust，而是为了处理一些：

- 编译期规则太严格
- 但逻辑上仍然安全

的场景。

## 5. 先和 `&mut T` 做一个最关键的对比

### `&mut T`

你可以先把它理解成：

- 编译期已经明确拿到了独占可变访问权

也就是说，编译器在编译期就能判断：

- 这里只有一个可变借用
- 不会和别的借用冲突

### `RefCell<T>`

你可以先把它理解成：

- 编译期先放宽一点
- 运行时再检查“当前是不是只存在一个可变借用，或者多个只读借用”

所以两者最大的差别不是“都能改值”，而是：

- 借用规则是谁来检查

也就是：

1. `&mut T`：编译期检查
2. `RefCell<T>`：运行时检查

## 6. `RefCell<T>` 的最小直觉例子

```rust
use std::cell::RefCell;

fn main() {
    let counter = RefCell::new(0);

    *counter.borrow_mut() += 1;

    println!("{}", counter.borrow());
}
```

这段代码里最值得先抓住 3 个动作：

1. `RefCell::new(0)`：把值包进 `RefCell`
2. `borrow_mut()`：运行时申请一次可变借用
3. `borrow()`：运行时申请一次不可变借用

这里已经能看到一个非常重要的信号：

- 你不再直接写 `&mut counter`
- 而是通过 `RefCell` 提供的方法拿到访问权限

## 7. `borrow()` 和 `borrow_mut()` 分别在做什么

### `borrow()`

作用是：

- 申请一次只读借用

你可以同时有多个 `borrow()`。

### `borrow_mut()`

作用是：

- 申请一次可变借用

但运行时规则仍然和 Rust 借用模型同向：

1. 可变借用只能同时存在一个
2. 可变借用和只读借用不能同时并存

也就是说，`RefCell<T>` 不是把规则取消了，而是把规则从编译期搬到了运行时。

## 8. `RefCell<T>` 为什么会 panic

这是必须正面讲清楚的一点。

看一个会出问题的例子：

```rust
use std::cell::RefCell;

fn main() {
    let value = RefCell::new(10);

    let a = value.borrow_mut();
    let b = value.borrow_mut();

    println!("{} {}", a, b);
}
```

这类代码通常会在运行时 panic。

原因很简单：

- 第一次 `borrow_mut()` 已经拿到了可变借用
- 第二次又想同时拿一个新的可变借用
- 这违反了借用规则

所以 `RefCell<T>` 的代价非常明确：

- 你把一部分安全检查从编译期搬到了运行时
- 如果你在运行时用错了，它不会编译报错，而会直接 panic

## 9. 一个更准确的心智模型

初学时你可以先这样记：

1. `RefCell<T>` 不是“无限制可变”
2. 它只是“延迟借用检查”
3. 借用规则仍然存在
4. 只是违反规则时，从“编译不过”变成了“运行时炸掉”

这个差别非常大。

所以它从来不是“更省事的默认选择”，而是一种权衡工具。

## 10. 为什么 `RefCell<T>` 常和 `Rc<T>` 一起出现

单独看：

- `Rc<T>` 解决的是单线程共享所有权
- `RefCell<T>` 解决的是单线程内部可变性

组合起来：

- `Rc<RefCell<T>>`

就变成了：

- 单线程共享可变状态

这是它最经典的使用方式。

例如：

```rust
use std::cell::RefCell;
use std::rc::Rc;

fn main() {
    let shared = Rc::new(RefCell::new(vec!["rust".to_string()]));

    let a = Rc::clone(&shared);
    let b = Rc::clone(&shared);

    a.borrow_mut().push("python".to_string());
    b.borrow_mut().push("java".to_string());

    println!("{:?}", shared.borrow());
}
```

这里的分工非常清楚：

1. `Rc<T>` 让多个地方拥有同一份数据
2. `RefCell<T>` 让这份共享数据还能被修改

## 11. 为什么不能直接用 `Rc<Vec<T>>`

因为 `Rc<T>` 只解决：

- 共享所有权

它不解决：

- 共享状态下如何修改

也就是说，`Rc<Vec<T>>` 会让你有多个拥有者，但这份 `Vec<T>` 仍然不是你想改就能改的。

如果你还想改它，就要再补上：

- `RefCell<T>`

所以：

1. `Rc<T>`：共享拥有
2. `RefCell<T>`：内部可变

组合在一起才完整。

## 12. `RefCell<T>` 和 `Mutex<T>` 的差别是什么

这也是非常容易混的一组。

### `RefCell<T>`

- 单线程
- 运行时借用检查
- 不负责线程同步

### `Mutex<T>`

- 多线程
- 通过锁保证同一时刻只有一个线程能改
- 适用于跨线程共享可变状态

一个很实用的判断是：

1. 单线程共享可变状态：优先想到 `Rc<RefCell<T>>`
2. 多线程共享可变状态：优先想到 `Arc<Mutex<T>>`

## 13. 一个很容易误用的地方：把它当“懒人可变性”

很多人刚学到 `RefCell<T>` 后，第一反应会是：

- 既然它能绕开一部分编译期限制，那我以后卡借用时报错就包一层 `RefCell<T>` 好了

这是非常危险的思路。

因为这会带来两个问题：

1. 你可能只是把设计问题推迟到了运行时
2. 你会失去编译期帮你兜底的大量安全价值

更稳的原则应该是：

- 先考虑普通所有权与借用能不能把设计讲清楚
- 只有当需求本身确实更适合内部可变性时，再上 `RefCell<T>`

## 14. 什么时候 `RefCell<T>` 比较合理

下面这些场景通常更合理。

### 14.1 共享结构里的局部可变状态

例如：

- 树节点
- 图结构
- 某些 UI 状态树

这类结构很容易出现：

- 多个地方共享同一个节点
- 节点内部又需要调整某些状态

### 14.2 缓存或惰性计算

例如：

- 对外暴露不可变方法
- 但内部第一次访问时要把缓存算出来

### 14.3 测试桩 / mock 对象

这是 `RefCell<T>` 非常经典的教学场景。

因为：

- 你可能有一个看起来只读的接口
- 但测试时你想记录“它到底被调用了几次”

这时内部可变性就非常合适。

## 15. 一个很典型的 mock 场景

假设你定义了一个发送消息的 trait：

```rust
trait Messenger {
    fn send(&self, message: &str);
}
```

注意这里是：

- `&self`

也就是说，从接口角度它看起来是只读调用。

但测试时你可能希望：

- 每调用一次 `send`，就把消息记录到列表里

这时测试桩就很适合用：

```rust
use std::cell::RefCell;

struct MockMessenger {
    messages: RefCell<Vec<String>>,
}

impl MockMessenger {
    fn new() -> Self {
        Self {
            messages: RefCell::new(vec![]),
        }
    }
}

impl Messenger for MockMessenger {
    fn send(&self, message: &str) {
        self.messages.borrow_mut().push(message.to_string());
    }
}
```

这就是内部可变性的一个非常正当的理由：

- 接口语义是只读的
- 但对象内部要记录额外状态

## 16. `Cell<T>` 要不要知道

这一章主角是 `RefCell<T>`，但你至少可以先知道：

- 标准库里还有 `Cell<T>`

它和 `RefCell<T>` 一样，也属于内部可变性工具。

但初学时你可以先记一个粗略区别：

1. `Cell<T>` 更适合小而可复制的值
2. `RefCell<T>` 更适合需要借用访问的更一般场景

如果你现在还没把 `RefCell<T>` 稳住，不用急着深挖 `Cell<T>` 的全部细节。

## 17. 一个很实用的作用域技巧

既然 `RefCell<T>` 的借用冲突会在运行时发生，那你就更需要注意借用的持续范围。

例如：

```rust
use std::cell::RefCell;

fn main() {
    let value = RefCell::new(vec![1, 2, 3]);

    {
        let mut data = value.borrow_mut();
        data.push(4);
    }

    println!("{:?}", value.borrow());
}
```

这里用块作用域把 `borrow_mut()` 提前结束掉，就是一个很常见、也很重要的写法。

因为很多运行时冲突，实际上只是：

- 借用活得比你以为的更久

## 18. 和普通借用规则的真正关系

这点值得你再巩固一次。

Rust 的借用规则核心并没有变：

1. 同时可以有多个只读借用
2. 或者只能有一个可变借用
3. 可变借用和只读借用不能并存

`RefCell<T>` 做的不是推翻这 3 条，而是：

- 把这 3 条的检查推迟到运行时

所以你学 `RefCell<T>` 时，越懂普通借用规则，越容易理解它。

## 19. 一个很重要的设计判断

以后你写代码时，可以先按这个顺序判断：

1. 普通 `&mut T` 能不能自然表达？
2. 如果不能，是不是因为需要共享所有权？
3. 如果是单线程共享可变状态，是否真的适合 `Rc<RefCell<T>>`？
4. 如果是多线程场景，是否应该改成 `Arc<Mutex<T>>`？

这个判断顺序会比“我卡住了，先包一层 `RefCell<T>`”稳很多。

## 20. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

JavaScript 里对象内部改状态通常非常自然，语言不会强制你把“谁拥有它、谁借用它”讲清楚。

Rust 则默认不允许这种随意性。

`RefCell<T>` 有点像：

- 在非常有限的范围内，把一部分灵活性重新引入回来

### 和 Python 类比

Python 里对象可变状态的记录很常见，比如缓存、计数、测试桩。

Rust 也有同样需求，只是它会更明确地把这种能力标成：

- 内部可变性

### 和 Java 类比

Java 里你很容易写一个对象，在 `final` 引用下仍然修改对象内部字段。

Rust 默认不会这么宽松。

`RefCell<T>` 某种程度上让你拿回了“引用不变但内部状态可变”的能力，但它的边界和代价都更显式。

## 21. 常见错误与修正方向

### 错误 1：一碰到借用报错就上 `RefCell<T>`

这通常是逃避设计问题，不是解决问题。

### 错误 2：忘了 `borrow_mut()` 的作用域

很多运行时 panic，并不是逻辑本身很复杂，而是借用持续时间比你想的长。

### 错误 3：单线程和多线程场景不分

单线程内部可变性和多线程共享可变状态是不同问题。

### 错误 4：以为 `RefCell<T>` 没有借用规则

有，而且规则没变，只是检查时机变了。

## 22. 一个非常实用的简化记忆

你可以先这样背：

1. `&mut T`：编译期可变借用
2. `RefCell<T>`：运行时可变借用检查
3. `Rc<RefCell<T>>`：单线程共享可变状态
4. `Arc<Mutex<T>>`：多线程共享可变状态

这个层次记忆会比单独背定义更稳。

## 24. 挑战题

做一个“学习通知记录器”设计练习，要求：

1. 定义一个 trait，方法签名使用 `&self`
2. 做一个 mock 实现，内部记录每次调用的消息
3. 用 `RefCell<Vec<String>>` 保存这些消息
4. 解释为什么这里不能只靠普通 `&mut self` 解决

这个挑战的重点不是代码多少，而是你要练会：

- 什么时候内部可变性是合理抽象

## 25. 小结

这一章最重要的不是把 `RefCell<T>` 神秘化，而是把它放回正确的位置：

1. `RefCell<T>` 是内部可变性工具
2. 它没有取消借用规则，只是把部分检查推迟到运行时
3. 它最常见的组合是 `Rc<RefCell<T>>`
4. 它适合单线程共享可变状态，不适合乱用来逃避设计问题
5. 作用域管理对它尤其重要
