# 工程化错误处理与项目结构设计

## 1. 这一章是干什么的

到了高级阶段，错误处理和项目结构设计已经不该再分开看了。

原因很简单：

- 错误边界本身就是项目边界的一部分

你在前面已经学过：

- `Result<T, E>`
- `?`
- `AppError`
- `thiserror`
- `anyhow`
- 多模块项目结构

但当项目继续变大时，真正的问题会变成：

1. 哪一层应该定义自己的错误类型
2. 哪一层该保留底层错误，哪一层该转换
3. 哪一层可以用更通用的错误汇总
4. 项目目录应该怎样拆，才能让错误边界和模块边界对齐

这一章的目标，就是让你从“会写错误处理”走向“会把错误处理嵌进项目结构设计”。

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- `Result<T, E>`
- `?`
- 自定义错误枚举
- 多模块 CLI 结构
- 基础生态库角色

最好已经看过：

- [错误处理与项目边界设计](../01-初级阶段/04-错误处理与项目边界设计.md)
- [常见生态库与选型直觉](./07-常见生态库与选型直觉.md)

## 3. 为什么高级阶段要重新看错误处理

因为初级阶段你主要在学：

- 怎么把错误表达出来
- 怎么用 `?`
- 怎么写一个基础 `AppError`

而高级阶段真正开始面对的是：

- 多层结构之间的错误传播和边界收束

也就是说，问题从：

- “这段代码失败了怎么办”

升级成了：

- “这个项目的哪一层该知道什么错误，哪一层不该知道什么错误”

## 4. 先建立一个非常实用的分层模型

你可以先把一个 Rust 项目粗略分成 4 层：

1. 输入与适配层
2. 应用流程层
3. 领域/业务层
4. 基础设施层

这不是唯一正确分层，但很适合先建立工程直觉。

### 输入与适配层

例如：

- CLI 参数
- HTTP 请求
- 配置文件

### 应用流程层

例如：

- 一个完整用例的编排逻辑

### 领域/业务层

例如：

- 核心规则
- 统计逻辑
- 校验逻辑

### 基础设施层

例如：

- 文件
- 网络
- 数据库
- 外部服务

## 5. 为什么错误类型应该跟层级一起设计

因为每一层关注的问题并不一样。

例如：

- 基础设施层关心“文件不存在”“网络超时”
- 业务层关心“配置不合法”“状态转换不允许”
- 应用层关心“当前用例最终要怎么失败给上层”

如果你把这些错误全部混成一个大杂烩字符串，项目很快就会乱。

所以你可以先建立一个总原则：

- 错误类型和模块职责应该相互对齐

## 6. 一个非常典型的坏味道

例如：

```rust
fn run() -> Result<(), String> {
    // ...
}
```

在小练习里这不一定错。

但只要项目一复杂，`String` 就会带来问题：

1. 错误来源不清楚
2. 调用方很难做分类处理
3. 业务错误和底层错误完全混在一起

所以高级阶段你要慢慢习惯：

- 错误不是“打印一句话就完”
- 错误是系统边界的一部分

## 7. 一个更像工程代码的结构例子

假设你在做一个异步抓取服务，可以先把目录粗略设计成：

```text
src/
├─ main.rs
├─ lib.rs
├─ app/
│  ├─ mod.rs
│  └─ run.rs
├─ domain/
│  ├─ mod.rs
│  ├─ report.rs
│  └─ error.rs
├─ infra/
│  ├─ mod.rs
│  ├─ http_client.rs
│  └─ file_store.rs
└─ interface/
   ├─ mod.rs
   └─ cli.rs
```

你不用机械照抄这种结构，但它能帮助你看到一个很关键的点：

- 错误和模块边界应该一起设计，而不是事后拼接

## 8. 基础设施层错误应该长什么样

基础设施层通常最接近：

- `std::io::Error`
- `reqwest::Error`
- JSON 解析错误

这一层的特点通常是：

- 更贴近技术实现细节

例如：

```rust
#[derive(thiserror::Error, Debug)]
pub enum HttpClientError {
    #[error("request failed: {0}")]
    Request(#[from] reqwest::Error),
}
```

这一层的错误不一定直接给最终用户看，但它很重要，因为它保留了：

- 技术来源
- 失败上下文

## 9. 业务层错误应该更像规则错误

例如：

```rust
#[derive(thiserror::Error, Debug)]
pub enum ReportError {
    #[error("report data is empty")]
    EmptyData,

    #[error("invalid score: {0}")]
    InvalidScore(i32),
}
```

这类错误和底层 I/O 错误不同。

它更像：

- 你的程序规则不允许当前状态继续走下去

所以业务层错误往往不应该直接依赖太多底层 crate 细节。

## 10. 应用层应该做什么

应用层最常做的事通常是：

1. 串联多个子模块
2. 决定是否做错误转换
3. 决定暴露给上层的错误边界

例如：

```rust
pub async fn run(config: Config) -> Result<(), AppError> {
    // 编排多个步骤
    Ok(())
}
```

这里的 `AppError` 往往更像：

- 当前应用对外暴露的“统一失败边界”

## 11. 为什么 `thiserror` 和 `anyhow` 经常要配合理解

这是高级 Rust 工程里非常高频的实际选择。

### `thiserror`

更适合：

- 定义清晰、稳定、可枚举的错误类型

### `anyhow`

更适合：

- 应用层快速汇总错误
- 脚本、小工具、最终入口整合

你可以先记一个很实用的粗略分工：

1. 库层、模块层、业务边界：更偏 `thiserror`
2. 最外层应用编排：更偏 `anyhow`

## 12. 一个很常见的工程组合

例如：

### 底层模块

```rust
#[derive(thiserror::Error, Debug)]
pub enum FileStoreError {
    #[error("io error: {0}")]
    Io(#[from] std::io::Error),
}
```

### 上层应用入口

```rust
use anyhow::Result;

pub async fn run() -> Result<()> {
    // ...
    Ok(())
}
```

这不是唯一方案，但它非常常见。

因为它实现了：

1. 模块内部边界清楚
2. 应用层汇总方便

## 13. 为什么不要所有层都直接用 `anyhow`

因为那样会失去一部分边界清晰度。

如果所有函数都直接：

- `Result<T, anyhow::Error>`

短期当然很快。

但项目一旦变大，就会开始出现：

1. 哪些错误是业务错误，不清楚
2. 哪些错误该被调用方区分处理，不清楚
3. 哪些模块真正承诺了什么错误边界，不清楚

所以一个更稳的判断通常是：

- 最外层可以更宽
- 越靠近核心边界，错误类型越该明确

## 14. 为什么也不要所有层都强行自定义超细错误

这又是另一个极端。

如果项目还很小，却在每个文件里都做非常复杂的错误层级，会带来：

1. 设计成本过高
2. 维护负担过重
3. 边界比需求还复杂

所以真正成熟的做法不是：

- 永远最细

而是：

- 边界清楚，复杂度跟项目规模匹配

## 15. 一个实用判断：这一层错误要不要被“识别”

你可以先问自己：

- 调用方需不需要根据这个错误类型做分支处理？

如果答案是：

- 需要

那更值得保留清晰类型。

如果答案是：

- 不需要，只需要最终统一打印或上抛

那更宽一点的汇总方式就可能更自然。

## 16. 项目结构设计为什么会和错误边界直接连起来

因为模块结构本质上就是：

- 谁可以依赖谁
- 谁知道谁的细节

而错误类型也在表达：

- 谁需要知道哪一层失败了

例如：

1. `infra/http_client.rs` 不该知道整个应用的 CLI 错误
2. `domain/report.rs` 不该依赖底层 HTTP 实现细节
3. `main.rs` 可以统一决定最终打印给用户的文案

你会发现：

- 项目结构清晰时，错误传播路径通常也更清晰

## 17. 一个很实用的模块边界习惯

每个模块可以尽量问自己这三个问题：

1. 我对外暴露的主入口函数是什么？
2. 我对外暴露的错误边界是什么？
3. 我内部依赖的第三方库细节，有没有必要泄漏给外面？

这个习惯会非常值钱。

因为它会逼你从“写功能”升级到“设计接口”。

## 18. `main.rs` 为什么应该非常薄

高级阶段项目里，`main.rs` 更应该像：

- 组装入口
- 初始化环境
- 调用应用层
- 打印最终错误

例如：

```rust
fn main() {
    if let Err(error) = real_main() {
        eprintln!("{:#}", error);
        std::process::exit(1);
    }
}
```

真正的流程编排应该更可能放到：

- `app::run`
- `lib.rs`

里。

这会让：

- 应用逻辑
- 进程入口逻辑

分离得更干净。

## 19. 高级阶段项目结构，不是目录越深越高级

这是一个非常重要的提醒。

项目结构设计的目标从来不是：

- 文件夹越多越专业

真正重要的是：

1. 模块职责有没有分开
2. 错误边界有没有分开
3. 第三方依赖有没有被包在合理层次里
4. 调用链是否容易理解

所以：

- 清楚 > 花哨

## 20. 一个小而稳的高级项目结构模板

如果你现在做一个中型 CLI/服务工具，可以先从这样一套相对稳的思路开始：

```text
src/
├─ main.rs
├─ lib.rs
├─ config.rs
├─ error.rs
├─ app/
├─ domain/
├─ infra/
└─ interface/
```

你不用照抄，但这套分层有几个核心意思：

1. `interface` 负责面对用户输入输出
2. `app` 负责流程编排
3. `domain` 负责核心规则
4. `infra` 负责文件、网络、数据库等外部依赖

这会让项目非常容易继续长。

## 21. 为什么第三方库尽量放在边界层

例如：

- `reqwest` 更适合包在 `infra/http_client.rs`
- `clap` 更适合包在 `interface/cli.rs`
- `serde` 可能主要出现在配置、DTO、边界模型

这样做的价值是：

1. 核心业务不必到处直接依赖第三方库
2. 替换库时影响范围更小
3. 测试时更容易 mock 边界

这和 Java 里的“适配器层”、Python/JS 里的“边界封装”本质上是同一个工程思路。

## 22. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过稍大一点的 Node.js 项目，你应该见过这种演进：

- 一开始所有逻辑都在一个文件
- 后来开始拆 service、client、config、logger
- 错误处理也开始从 `throw new Error()` 变成分层

Rust 会更强地逼你把这些事做得显式。

### 和 Python 类比

Python 工程里也会逐步出现：

- 配置层
- 业务层
- 外部依赖层
- 日志和错误层

Rust 的差别在于：

- 错误边界更容易被类型系统固定下来

### 和 Java 类比

Java 背景会让你对：

- 分层
- service
- infrastructure
- exception boundary

这些东西不陌生。

Rust 的差别是：

- 它不靠异常体系主导流程
- 更靠 `Result`、错误类型和模块边界主导流程

## 23. 常见错误与修正方向

### 错误 1：所有模块都直接返回 `anyhow::Error`

小项目能跑，但边界会越来越糊。

### 错误 2：所有模块都定义超复杂细粒度错误

边界复杂度可能比业务本身还高。

### 错误 3：第三方库直接渗透到核心业务的每个角落

后面想替换或测试时会更麻烦。

### 错误 4：目录结构看起来很多层，但职责没真正分开

这会让项目看起来“像工程”，但其实还是一团。

## 24. 一个很实用的设计顺序

如果你现在要搭一个高级阶段项目，建议按这个顺序做：

1. 先画出输入边界、业务边界、外部依赖边界
2. 再决定每层暴露哪些主函数
3. 再决定每层错误边界
4. 最后再引入具体生态库

这个顺序会让你少走很多弯路。

因为它把“项目结构”放在“库选择”前面了。

## 25. 小练习

1. 给你之前做过的一个 CLI 项目重新画一次分层图。
2. 指出其中哪些模块属于 interface、app、domain、infra。
3. 指出其中哪些错误更适合用明确枚举，哪些更适合汇总。
4. 想一想 `clap`、`serde`、`reqwest`、`tracing` 应该分别更适合放在哪一层。

## 26. 挑战题

做一个“异步学习报告服务”架构练习，要求：

1. 支持 CLI 启动参数
2. 支持配置文件
3. 支持异步拉取远程数据
4. 支持结构化日志
5. 支持清晰错误输出

额外要求：

1. 画出你会拆出的主要模块
2. 说明每个模块依赖哪些生态库
3. 说明每个模块最主要的错误边界是什么
4. 解释你会在哪一层使用 `thiserror`，在哪一层可能使用 `anyhow`

## 27. 小结

这一章最重要的不是给你一套唯一目录模板，而是建立两个真正值钱的直觉：

1. 错误边界和项目边界应该一起设计
2. 第三方生态库应该尽量被包在清晰的边界层里

如果这两点开始站住，你的 Rust 工程代码就会越来越像“可维护项目”，而不是“堆功能脚本”。
