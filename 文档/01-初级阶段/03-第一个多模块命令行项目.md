# 第一个多模块命令行项目

## 1. 这一章是干什么的

前面你已经做过入门阶段的综合 CLI 小项目。

现在要继续往前走一步：

- 不只是“能跑”
- 而是开始写得更像一个小工程

这一章会带你搭一个更完整的多模块命令行项目，并重点解决下面这些问题：

- `main.rs` 和 `lib.rs` 分别干什么
- 参数解析、文件读取、核心逻辑、输出格式怎么拆
- 错误怎么逐层返回
- 哪些函数更适合补测试

## 2. 学这一章前，你需要知道什么

建议你已经掌握：

- 基础模块系统
- `Result<T, E>`
- `?`
- `Vec<T>`、字符串、迭代器基础
- 所有权与借用的基础直觉

如果你对这些内容还不完全稳定，也没关系。项目本来就是拿来把知识点重新串起来的。

## 3. 这次项目做什么

这次我们设计一个：

- 日志关键词统计器

它的目标很简单：

1. 从命令行读取文件路径和关键词
2. 读取日志文件
3. 找出包含关键词的行
4. 输出匹配数量和匹配内容
5. 给出一个简短统计摘要

这个项目很适合初级阶段，因为它会同时碰到：

- 参数解析
- 文件读写
- 字符串处理
- 模块拆分
- 测试
- 错误传播

## 4. 为什么这个项目比入门项目更进一步

入门项目通常更强调：

- 从零把流程跑通

这一章的项目则更强调：

1. 每个模块的职责是否单一
2. 参数和返回值是否设计合理
3. 哪部分是纯逻辑，哪部分在碰 I/O
4. 哪些地方方便写测试

也就是说，这一次你不只是练“会写”，而是在练“怎么拆得更顺”。

## 5. 先看项目结构

建议把项目结构拆成这样：

```text
日志关键词统计器/
├─ Cargo.toml
├─ src/
│  ├─ main.rs
│  ├─ lib.rs
│  ├─ config.rs
│  ├─ reader.rs
│  ├─ analyzer.rs
│  └─ report.rs
└─ tests/
   └─ 集成测试.rs
```

这里有一个很实用的现实建议：

- 仓库里的教程文档可以用中文文件名
- Rust 源码模块在真实项目里通常仍然更建议用英文或稳定的拼音命名

原因不是技术限制，而是生态约定和团队协作习惯。

## 6. 每个模块各自负责什么

### `main.rs`

职责尽量轻：

- 接程序入口
- 调用配置解析
- 调用业务主流程
- 统一打印错误

### `lib.rs`

把库层入口放在这里：

- 暴露模块
- 组织 `run(config)` 这样的主流程函数

### `config.rs`

只负责：

- 解析命令行参数
- 生成 `Config`

### `reader.rs`

只负责：

- 读取文件内容
- 拆成行列表

### `analyzer.rs`

只负责：

- 过滤关键词
- 统计匹配数量
- 组织核心分析逻辑

### `report.rs`

只负责：

- 把分析结果格式化成可打印文本

## 7. `Config` 应该长什么样

可以先从一个最小结构体开始：

```rust
pub struct Config {
    pub file_path: String,
    pub keyword: String,
}
```

这里直接用拥有型字段是很自然的，因为：

- 参数解析之后，这份配置要跨多个模块传递
- 如果全都写成借用，生命周期关系会更绕

这也是前面所有权章节讲过的一个实战体现：

- 长期保存、跨边界传递的数据，优先考虑拥有型字段

## 8. `config.rs` 的一个最小思路

```rust
use crate::AppError;

pub struct Config {
    pub file_path: String,
    pub keyword: String,
}

impl Config {
    pub fn from_args(args: &[String]) -> Result<Self, AppError> {
        if args.len() < 3 {
            return Err(AppError::MissingArgument);
        }

        Ok(Self {
            file_path: args[1].clone(),
            keyword: args[2].clone(),
        })
    }
}
```

这段代码的重点不是“必须这样写”，而是让你看到：

- 参数解析是一个单独职责
- 错误在这里就可以被明确表达

## 9. `reader.rs` 尽量做纯粹一点

```rust
use std::fs;

pub fn read_lines(path: &str) -> Result<Vec<String>, std::io::Error> {
    let content = fs::read_to_string(path)?;

    Ok(content.lines().map(|line| line.to_string()).collect())
}
```

这里有两个设计点值得注意：

1. 参数只读，所以接 `&str`
2. 返回新的 `Vec<String>`，因为这些行数据后面要继续被分析模块使用

## 10. `analyzer.rs` 最适合放纯逻辑

```rust
pub fn filter_lines<'a>(lines: &'a [String], keyword: &str) -> Vec<&'a str> {
    lines
        .iter()
        .filter(|line| line.contains(keyword))
        .map(|line| line.as_str())
        .collect()
}

pub fn count_matches(lines: &[&str]) -> usize {
    lines.len()
}
```

这里可以看到几件事：

1. 读取行列表时不需要拿所有权，只借用切片
2. 返回的是借用出来的匹配结果，所以用了生命周期
3. `count_matches` 是纯逻辑函数，非常容易写测试

## 11. `report.rs` 负责把结果组织成输出

```rust
pub fn build_report(keyword: &str, matches: &[&str]) -> String {
    let mut output = String::new();
    output.push_str(&format!("关键词: {}\n", keyword));
    output.push_str(&format!("匹配数量: {}\n", matches.len()));

    for line in matches {
        output.push_str("- ");
        output.push_str(line);
        output.push('\n');
    }

    output
}
```

这类函数非常适合返回 `String`，因为：

- 它本来就是在构建新的输出文本
- 返回拥有型字符串最自然

## 12. `lib.rs` 负责把流程串起来

可以写成这样：

```rust
pub mod analyzer;
pub mod config;
pub mod reader;
pub mod report;

use config::Config;

#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    Io(std::io::Error),
}

impl From<std::io::Error> for AppError {
    fn from(value: std::io::Error) -> Self {
        Self::Io(value)
    }
}

pub fn run(config: &Config) -> Result<String, AppError> {
    let lines = reader::read_lines(&config.file_path)?;
    let matches = analyzer::filter_lines(&lines, &config.keyword);
    Ok(report::build_report(&config.keyword, &matches))
}
```

这里的重点是分层：

1. `reader` 负责读文件
2. `analyzer` 负责分析
3. `report` 负责拼输出
4. `run` 负责把这些步骤串起来

## 13. `main.rs` 应该尽量薄

```rust
use std::env;

use log_keyword_tool::config::Config;
use log_keyword_tool::run;

fn main() {
    let args: Vec<String> = env::args().collect();

    match Config::from_args(&args).and_then(|config| run(&config)) {
        Ok(report) => println!("{}", report),
        Err(error) => eprintln!("程序执行失败: {:?}", error),
    }
}
```

这里你要刻意形成一个工程习惯：

- 入口文件负责启动
- 真正业务逻辑别都塞在入口里

这和 Java 里把逻辑拆到 service 层、和 Python 里把脚本入口和业务函数拆开，是同一个方向。

## 14. 哪些地方最适合写测试

项目里不是所有地方都一样适合测。

初级阶段优先测试这些“纯逻辑”部分：

1. `filter_lines`
2. `count_matches`
3. `build_report`
4. `Config::from_args`

例如：

```rust
#[cfg(test)]
mod tests {
    use super::filter_lines;

    #[test]
    fn should_find_lines_containing_keyword() {
        let lines = vec![
            String::from("error: disk full"),
            String::from("info: started"),
            String::from("error: retry"),
        ];

        let matches = filter_lines(&lines, "error");
        assert_eq!(matches.len(), 2);
    }
}
```

## 15. 为什么这个结构更好维护

因为它把不同变化点分开了：

- 参数变化，改 `config`
- 文件来源变化，改 `reader`
- 分析规则变化，改 `analyzer`
- 输出样式变化，改 `report`

这比把所有逻辑都塞进 `main.rs` 好很多。

## 16. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

这有点像你在 Node.js CLI 里把：

- 参数读取
- 文件读取
- 业务分析
- 输出格式化

拆成不同模块，而不是全写在一个脚本里。

### 和 Python 类比

很像把脚本拆成：

- `main`
- `config`
- `service`
- `formatter`

这样的分层，只是 Rust 会额外要求你把类型边界和错误边界说清楚。

### 和 Java 类比

如果你有 Java 背景，可以把它粗略想成：

- `main` 是入口
- `Config` 是配置对象
- `analyzer` / `reader` / `report` 类似更轻量的职责模块

## 17. 常见错误与修正方向

### 错误 1：所有逻辑都塞到 `main.rs`

程序能跑不代表结构合理。入口文件应该越来越薄。

### 错误 2：读取、分析、打印混在一个函数里

这会让测试变难，也会让错误定位变难。

### 错误 3：为了避免生命周期，所有地方都整体 `clone()`

先想一想是不是可以通过借用切片和借用字符串解决。

### 错误 4：一开始就做过度复杂的错误体系

初级阶段先把：

- 参数错误
- I/O 错误

两类处理清楚，就很够用了。

## 18. 小练习

1. 先只完成 `Config::from_args`。
2. 再单独实现 `read_lines`。
3. 再单独实现 `filter_lines` 和 `build_report`。
4. 最后再写 `run` 把它们串起来。
5. 给 `filter_lines` 至少补一个测试。

## 19. 挑战题

把这个项目升级一版：

1. 增加大小写不敏感搜索
2. 增加“只输出匹配数量，不输出明细”的模式
3. 为参数解析增加更清晰的错误信息
4. 给 `report` 模块增加至少 2 个测试
5. 自己解释每个模块为什么放在当前位置

如果你还能再往前一步，可以继续做：

1. 引入 `clap` 做参数解析
2. 用 `thiserror` 整理错误类型

## 20. 小结

这一章最重要的不是项目本身，而是你要开始形成这些工程习惯：

1. 入口要薄
2. 配置、I/O、分析、输出要分层
3. 纯逻辑函数优先设计成容易测试
4. 先把借用和拥有关系设计清楚，再写代码会顺很多
5. 小项目也值得认真做结构

到这里，你就已经正式开始从“能写 Rust 练习题”，走向“能组织 Rust 小项目”了。
