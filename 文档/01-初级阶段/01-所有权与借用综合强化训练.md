# 所有权与借用综合强化训练

## 1. 这一章是干什么的

前面你已经分别学过：

- 所有权初识
- 引用与借用
- `String` 和 `&str`

但很多人真正卡住，不是在概念定义上，而是在真实代码里不会判断：

- 这个参数该不该拿所有权
- 这个函数能不能返回引用
- 这个结构体字段该不该存借用
- 这个集合一遍历为什么值就没了

这一章不再单独重复规则，而是把所有权和借用放进更真实的使用场景里，帮你建立“遇到代码时怎么判断”的能力。

## 2. 学这一章前，你需要知道什么

你最好已经理解下面这些基本概念：

- move
- `Copy`
- `&T`
- `&mut T`
- `String` 与 `&str`

如果这些词你已经见过，但还不够稳，正好适合继续看这一章。

## 3. 先建立一个统一判断框架

以后你每次遇到所有权问题，都先问自己 3 个问题：

1. 这个值最后应该归谁拥有？
2. 我现在只是读取，还是要修改？
3. 这个值要活多久？

很多 Rust 报错看起来复杂，本质上都在逼你把这 3 个问题回答清楚。

## 4. 场景一：函数参数到底传什么

这是初级阶段最常见的问题。

### 4.1 什么时候直接拿所有权

如果函数会消费这个值，或者函数执行完后调用方本来也不再需要它，那么直接拿所有权是合理的。

```rust
fn save_name(name: String) {
    println!("保存名字: {}", name);
}
```

这里的直觉是：

- 调用方把 `name` 交给你
- 你拿到之后，这个值的归属已经转移

如果你把它类比到 JavaScript / Python，可以粗略理解成“把对象交给下游处理”，但 Rust 会把这个转移显式写进类型和编译规则里。

### 4.2 什么时候只借用 `&str`

如果函数只是读字符串内容，不需要拥有它，最常见的选择通常是：

```rust
fn print_name(name: &str) {
    println!("名字是: {}", name);
}
```

这种写法比 `&String` 更灵活，因为它既能接：

- `String`
- `&String`
- `&str`

也就是说，读字符串时优先考虑：

- `&str`

### 4.3 什么时候用 `&mut String`

如果函数要原地修改调用方持有的数据，才考虑可变借用：

```rust
fn add_suffix(name: &mut String) {
    name.push_str("_rust");
}
```

这时函数不是拿走值，而是：

- 临时借来修改
- 修改完再还回去

### 4.4 一个实用判断表

可以先这样记：

1. 只读：优先 `&T`
2. 要改但不接管：优先 `&mut T`
3. 要消费、要存起来、要转交：考虑 `T`

## 5. 场景二：返回值为什么经常直接返回拥有的数据

很多初学者容易写出这样的代码：

```rust
fn build_name() -> &str {
    let name = String::from("rust");
    &name
}
```

这个思路会出问题，因为：

- `name` 是函数内部创建的局部变量
- 函数结束后它就被释放了
- 你不能返回一个指向已经失效内容的引用

更常见的正确写法是：

```rust
fn build_name() -> String {
    let name = String::from("rust");
    name
}
```

这时返回的是所有权本身，不是悬空引用。

### 一个重要直觉

如果数据是在函数内部新建出来的，返回拥有的数据通常比返回引用自然得多。

## 6. 场景三：结构体字段该放 `String` 还是 `&str`

初学者很容易被 `&str` 看起来“更轻量”这件事吸引，于是想在结构体里到处放 `&str`。

例如：

```rust
struct User<'a> {
    name: &'a str,
}
```

这不是不能写，但它会立刻把生命周期关系带进来。

如果这个结构体要在项目里长期存活、跨函数传递、被多个地方管理，那么直接拥有数据往往更省心：

```rust
struct User {
    name: String,
}
```

### 初学阶段的实用建议

如果一个结构体要“保存数据”，优先考虑存拥有的数据：

- `String`
- `Vec<T>`
- 其他拥有型字段

如果一个结构体只是临时“观察”外部数据，再考虑借用字段。

## 7. 场景四：遍历集合时为什么值会突然被 move

先看这段代码：

```rust
fn main() {
    let names = vec![String::from("a"), String::from("b")];

    for name in names {
        println!("{}", name);
    }

    println!("{:?}", names);
}
```

这会有问题，因为：

- `for name in names` 会拿走集合里的元素
- 更准确地说，这里会消费 `names`
- 循环结束后，`names` 已经不能再用了

### 7.1 只读遍历：`iter()`

```rust
fn main() {
    let names = vec![String::from("a"), String::from("b")];

    for name in names.iter() {
        println!("{}", name);
    }

    println!("{:?}", names);
}
```

这里拿到的是：

- `&String`

所以集合仍然归原来的变量所有。

### 7.2 要修改元素：`iter_mut()`

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("b")];

    for name in names.iter_mut() {
        name.push_str("_x");
    }

    println!("{:?}", names);
}
```

这里拿到的是：

- `&mut String`

### 7.3 就是要消费集合：`into_iter()`

```rust
fn main() {
    let names = vec![String::from("a"), String::from("b")];

    for name in names.into_iter() {
        println!("{}", name);
    }
}
```

这表示：

- 你明确接受“遍历后集合就不再使用”

## 8. 场景五：借用了集合元素后，为什么不能立刻改集合本身

看一个常见例子：

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("b")];
    let first = &names[0];

    names.push(String::from("c"));

    println!("{}", first);
}
```

这类代码经常报错，本质原因是：

- `first` 正在借用 `names` 内部的元素
- `push` 可能导致底层内存重新分配
- 那样一来，之前借出去的引用可能失效

Rust 在编译期就把这种风险拦下来了。

### 常见修正思路

#### 方案 1：缩短借用作用域

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("b")];

    {
        let first = &names[0];
        println!("{}", first);
    }

    names.push(String::from("c"));
}
```

#### 方案 2：先复制出需要的数据

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("b")];
    let first = names[0].clone();

    names.push(String::from("c"));

    println!("{}", first);
}
```

这里用 `clone()` 是有成本的，但如果你确实需要保留一个独立副本，它是合理的。

## 9. 场景六：到底什么时候该 `clone()`

很多初学者有两个极端：

1. 一报借用错误就疯狂 `clone()`
2. 死也不 `clone()`，结果代码越来越拧巴

更实用的判断方式是：

### 9.1 可以优先借用的场景

- 只是读数据
- 数据还要继续被原拥有者使用
- 接口可以自然表达成 `&T`

### 9.2 `clone()` 可能合理的场景

- 你确实需要一个独立副本
- 数据量不大，成本可接受
- 借用关系会把代码复杂度明显拉高

### 9.3 更应该重构而不是 `clone()` 的场景

- 你只是为了压过编译器才复制
- 同一个函数里到处克隆同一批数据
- 数据本来应该只读传递，却被设计成反复移动

### 一个简化结论

不要把 `clone()` 当“止痛药”。

先问自己：

- 我是真的需要一份新数据
- 还是只是接口设计不合理

## 10. 一个很常见的错误代码到修正过程

### 错误版本

```rust
fn count_long_names(names: Vec<String>) -> usize {
    let mut total = 0;

    for name in names {
        if name.len() > 3 {
            total += 1;
        }
    }

    println!("{:?}", names);
    total
}
```

这里的问题是：

- `for name in names` 消费了 `names`
- 后面又想继续使用 `names`

### 修正版本

```rust
fn count_long_names(names: &[String]) -> usize {
    let mut total = 0;

    for name in names.iter() {
        if name.len() > 3 {
            total += 1;
        }
    }

    println!("{:?}", names);
    total
}
```

这里的升级点有两个：

1. 参数改成切片借用 `&[String]`
2. 遍历改成 `iter()`

这比“为了能打印就先整体 `clone()` 一份”更像合理设计。

## 11. 和 JavaScript、Python、Java 的再类比

### JavaScript / Python

你经常传的是对象引用，但语言不会强迫你把“当前谁负责它”说清楚。很多副作用问题只有运行时才暴露。

### Java

Java 里对象变量本质上也更像引用，但因为有 GC，你很少需要直接思考“这个对象会不会因为离开作用域而被编译器禁止借出”。

### Rust

Rust 把“数据怎么流动、什么时候还能借、什么时候必须归还”都提前做成了规则。

所以你可以把 Rust 的所有权训练理解成：

- 让隐藏的数据流成本，变成显式设计问题

## 12. 常见错误与修正方向

### 错误 1：只读函数写成拿所有权

如果函数只是打印、判断、统计，优先想一想能不能改成借用。

### 错误 2：结构体过早使用 `&str`

如果结构体要长期保存数据，优先考虑 `String`。

### 错误 3：遍历后还想继续用原集合

回头检查你是不是无意中用了消费式遍历。

### 错误 4：一报错就到处 `clone()`

先检查接口和数据流设计，别把复制当默认答案。

## 14. 挑战题

做一个“学员名单分析器”，要求：

1. 输入一组 `Vec<String>`
2. 统计总人数
3. 找出名字长度最长的那一项
4. 生成一个新的 `Vec<String>`，内容是全部大写版本
5. 原始输入在统计结束后仍然能继续使用

额外要求：

1. 至少有一个函数接收 `&[String]`
2. 至少有一个函数返回新的拥有型数据
3. 自己解释每个函数为什么选择了当前参数类型

## 15. 小结

这一章最重要的不是再背一遍规则，而是建立几个实用判断：

1. 只读优先借用，修改优先可变借用，消费才拿所有权
2. 函数内部新建的数据，通常直接返回拥有权更自然
3. 结构体如果要保存数据，优先存拥有型字段
4. 集合遍历时要明确自己是在借用、修改，还是消费
5. `clone()` 可以用，但应该带着成本意识用
