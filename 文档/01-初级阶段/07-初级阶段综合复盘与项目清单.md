# 初级阶段综合复盘与项目清单

## 1. 这一章是干什么的

这一章不是继续塞新语法，而是把整个初级阶段真正收口。

你在初级阶段已经系统碰过这些内容：

- 所有权与借用的真实判断
- trait、泛型、生命周期的组合使用
- 多模块命令行项目结构
- 错误处理与项目边界
- 迭代器、闭包与集合组合
- 第二个更完整的综合项目

如果这时候不回头复盘，你很容易出现一种典型状态：

- 每篇单独看都能理解
- 但一到自己设计项目，还是不知道从哪下手

这一章要做的，就是把这种“分散会一点”的状态，压成比较稳定的初级能力。

## 2. 到这个阶段，你已经不再是“纯入门 Rust 学习者”

初级阶段和入门阶段的差别很大。

入门阶段主要是：

- 建立语言直觉
- 接受 Rust 的规则感
- 开始理解所有权和类型系统

初级阶段则更接近：

- 真正开始能稳定组织一个 Rust 小项目

也就是说，你现在最该关心的问题已经不是：

- 这个语法怎么写

而更像是：

- 这个值该怎么流动
- 这个模块该怎么拆
- 这个错误该在哪一层处理
- 这个数据处理逻辑该用循环还是迭代器链

## 3. 你现在应该已经掌握的核心能力

如果初级阶段学得比较顺，理想状态下你至少应该具备下面这些能力。

### 3.1 所有权与借用层面

- 能比较稳定地区分 `T`、`&T`、`&mut T`
- 知道什么时候该返回拥有型数据，什么时候可以返回借用
- 遇到集合遍历、字段设计、参数传递时，不会完全失去判断
- 不会再把 `clone()` 当成默认止痛药

### 3.2 抽象能力层面

- 能读懂常见 trait 约束和泛型函数签名
- 知道泛型解决类型复用，trait 解决能力约束，生命周期解决借用关系
- 能写出第一批真正有抽象味道的函数和结构体

### 3.3 工程组织层面

- 会把配置、读取、解析、分析、输出拆成不同模块
- 知道 `main.rs` 应该尽量薄
- 知道 `lib.rs` 更适合承接项目主流程
- 知道哪些逻辑适合单元测试，哪些更适合集成测试

### 3.4 错误处理层面

- 知道底层、中层、顶层在错误处理里的不同职责
- 会写基础 `AppError`
- 能用 `?` 做错误传播
- 知道什么时候该 `filter_map(...ok())`，什么时候该 `collect::<Result<Vec<_>, _>>()`

### 3.5 数据处理层面

- 能用迭代器链表达线性数据流
- 能用 `map`、`filter`、`filter_map`、`find`、`fold` 做组合处理
- 知道闭包为什么会和外部变量、所有权、借用连起来

## 4. 如果你现在还会卡，最常见卡点在哪里

这很正常。

初级阶段常见的卡点，通常已经不是“没看过”，而是下面这些更真实的问题：

1. 一写到返回引用就不稳
2. 一看到稍长的泛型签名就有压力
3. 项目一拆模块就开始不知道错误类型放哪
4. 一写迭代器链就容易被引用层级绕住
5. 一遇到“整体失败”和“忽略失败项”的选择就拿不准
6. 一做综合项目就容易重新把逻辑塞回一个大函数

这些卡点本质上都不是单点语法问题，而是：

- 你还在过渡到 Rust 的工程性写法

## 6. 一套非常实用的自测问题

如果你能把下面这些问题用自己的话答清楚，说明初级阶段已经不是“表面看过”。

1. 什么时候函数参数该接 `String`，什么时候更适合接 `&str`？
2. 为什么很多结构体字段更适合直接存 `String`，而不是存 `&str`？
3. 泛型、trait、生命周期分别在解决什么问题？
4. `main.rs`、`lib.rs`、`error.rs`、`parser.rs`、`report.rs` 这类模块各自更适合承担什么职责？
5. 为什么 `?` 不只是简化语法，而是在表达边界？
6. 什么场景适合 `filter_map(...ok())`，什么场景更适合 `collect::<Result<Vec<_>, _>>()`？
7. 什么时候一条迭代器链是在表达清晰数据流，什么时候反而该退回普通 `for`？

如果这些问题你现在答不出来，别急着往高级阶段冲，先回头把相关章节重新敲一遍。

## 7. 你现在最值得完成的项目清单

如果你想把初级阶段练扎实，下面这组项目最有代表性。

### 第一组：基础综合 CLI

1. 日志关键词统计器
2. 词频统计器
3. 配置文件读取器

目标：

- 巩固模块拆分、错误传播、字符串处理

### 第二组：结构化数据处理项目

1. 学习记录分析器
2. 成绩统计与报告生成器
3. CSV 风格文本清洗器

目标：

- 巩固建模、解析、业务校验、统计汇总

### 第三组：项目升级训练

1. 给现有 CLI 增加严格模式和宽松模式
2. 增加测试
3. 引入 `clap` 或 `thiserror`
4. 重构模块边界

目标：

- 从“写出第一版”升级到“能把第一版写顺”

## 8. 一个很值得做的复盘动作

同一个项目，不要只做一次。

一个更好的升级顺序通常是：

1. 先写最小可运行版
2. 再补错误处理
3. 再拆模块
4. 再补测试
5. 再把明显拧巴的部分重构掉

这一套动作特别重要。

因为 Rust 不是一门只靠“第一次写出来”就能真正学会的语言。

它更像一门：

- 你必须反复重构，才能把规则和直觉写顺

## 9. 你什么时候可以认为自己“完成了初级阶段”

不是把文档都看完就算完成。

更现实的判断标准通常是：

1. 你能独立做一个结构清晰的多模块 CLI
2. 你能解释自己为什么这样拆模块
3. 你能给纯逻辑层补一批基础测试
4. 你遇到常见借用报错时，不会立刻完全无从下手
5. 你能比较自然地写出 `Result`、`AppError`、`?`
6. 你能把一组文本输入转换成结构化数据，再进一步做统计或分析
7. 你不会再把所有项目逻辑都本能塞进 `main.rs`

如果这些能力已经站住，说明你已经开始跨进 Rust 的“稳定初级工程能力段”。

## 10. 初级阶段完成后，最自然的下一步是什么

下一阶段最值得进入的，就是高级阶段。

高级阶段和初级阶段最大的不同在于：

- 初级阶段主要在练“把规则用顺”
- 高级阶段则开始进入“为什么要选这套抽象和这套并发模型”

也就是说，接下来你会开始认真面对：

1. 智能指针体系
2. 内部可变性
3. trait object 与动态分发
4. 多线程、共享状态、channel
5. `Send` / `Sync`
6. async/await

这已经比初级阶段更接近真实中型项目和开源代码阅读了。

## 11. 一个建议的阶段切换方式

不要一完成初级阶段，就立刻把全部注意力切到高级概念上。

更稳的切法通常是：

1. 先做一次初级阶段综合复盘
2. 再完整做出一个你自己的 CLI 项目
3. 给这个项目补测试
4. 然后再进入高级阶段第一批主题

这样你带着项目经验去看高级阶段，会更容易理解：

- 为什么会需要 `Rc<RefCell<T>>`
- 为什么会需要 `Arc<Mutex<T>>`
- 为什么某些抽象要用 trait object

## 12. 建议你现在立刻做的 4 件事

如果你想把这一阶段收得更稳，建议你立刻做下面 4 件事：

1. 回头重做一次你最不稳的那篇所有权或迭代器练习
2. 把“学习记录分析器”亲手完整实现一遍
3. 给 `parser` 和 `analyzer` 至少各补 3 个测试
4. 用自己的话写一份“我现在如何理解 Rust 初级阶段的工程边界”

这些动作非常值，因为它们会让你从“看文档理解”过渡到“自己真的会写”。

## 13. 小结

这一章最重要的定位不是新知识，而是确认你现在站在哪里：

1. 你已经从 Rust 入门阶段跨进了真正的初级工程能力段
2. 你现在最关键的差距，不再主要是“有没有看过”，而是“能不能自己组合这些能力”
3. 你已经可以开始接触更真实的项目组织和抽象选择问题
4. 高级阶段最自然的入口，应该建立在你已经做过至少一两个完整小项目的基础上
