## 3. 这个项目解决什么真实问题

我们先给项目一个非常清楚的目标：

- 从一个文本文件里读取学习记录
- 解析每一行
- 统计总数、有效数、总学习时长、平均分
- 找出分数最高的一条记录
- 输出一份清晰报告

假设输入文件内容像这样：

```text
rust,90,2.5
javascript,82,1.5
python,95,3.0
bad_line
java,88,2.0
```

每一行理想格式是：

```text
科目,分数,学习时长
```

例如：

- `rust,90,2.5`

这类项目比前一个更进一步的地方在于：

- 它不是只做字符串搜索
- 它开始有“解析 -> 建模 -> 统计 -> 输出”这一整条业务链

## 4. 为什么这个项目很适合初级阶段

因为它刚好把前面最重要的几个点集中在一起了。

这次项目更适合认真拆分：

- 读文件

## 8. 先设计核心数据模型

### 8.1 学习记录

```rust
pub struct StudyRecord {
    pub subject: String,
    pub score: u32,
    pub hours: f64,
}
```

这里几个字段都直接用拥有型数据或简单值类型，非常自然：

- `subject` 要跨模块使用，所以用 `String`
- `score` 用 `u32`
- `hours` 用 `f64`

### 8.2 统计结果

```rust
pub struct StudySummary {
    pub total_lines: usize,
    pub valid_records: usize,
    pub total_hours: f64,
    pub average_score: f64,
    pub best_subject: Option<String>,
    pub best_score: Option<u32>,
}
```

这里用 `Option` 是合理的，因为：

- 如果没有任何有效记录，最高分项目就可能不存在

这类“可能没有”的状态，正适合用 `Option` 表达。

## 9. 配置对象怎么设计

第一版可以非常简单：

```rust
pub struct Config {
    pub file_path: String,
}
```

如果你想再加一点扩展空间，也可以设计成：

```rust
pub struct Config {
    pub file_path: String,
    pub ignore_invalid_lines: bool,

```

这样后面就能引出两种很常见的策略：

1. 遇到坏行就整体失败
2. 遇到坏行就跳过，只统计合法数据

这本身就是一个非常好的工程思考点。

## 10. `AppError` 应该怎么长

这次项目已经足够值得有一个独立的 `error.rs`。

例如：

```rust
#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    Io(std::io::Error),
    InvalidLineFormat { line_number: usize, content: String },
    InvalidScore { line_number: usize, value: String },
    InvalidHours { line_number: usize, value: String },
}
```

这里有几个值得注意的点：

1. 参数缺失是业务错误
2. 文件读取失败是 I/O 错误
3. 行解析错误带上行号和原始内容，会非常利于排查

这已经开始接近真实工具里的错误设计了。

## 11. `Display` 在这个项目里很有必要

因为这类 CLI 最后是直接面向用户输出的。

例如：

```rust
use std::fmt;

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::MissingArgument => write!(f, "请提供学习记录文件路径"),
            AppError::Io(error) => write!(f, "读取文件失败: {}", error),
            AppError::InvalidLineFormat { line_number, content } => {
                write!(f, "第 {} 行格式不正确: {}", line_number, content)
            }
            AppError::InvalidScore { line_number, value } => {
                write!(f, "第 {} 行分数无法解析: {}", line_number, value)
            }
            AppError::InvalidHours { line_number, value } => {
                write!(f, "第 {} 行学习时长无法解析: {}", line_number, value)
            }
        }
    }
}
```

这样顶层 `main()` 就能直接用：

```rust
eprintln!("{}", error);
```

## 12. `reader.rs` 应该只负责读取，不负责理解内容

一个很常见的坏味道是：

- 读文件的同时就开始解析业务格式

更好的边界是：

```rust
pub fn read_lines(path: &str) -> Result<Vec<String>, std::io::Error> {
    let content = std::fs::read_to_string(path)?;
    Ok(content.lines().map(|line| line.to_string()).collect())
}
```

这层只负责：

- 把文件变成原始行列表

它不应该在这一层决定：

- 某一行是不是合法学习记录

那是 `parser.rs` 的职责。

## 13. `parser.rs` 才是这个项目的第一核心

因为这一步真正把“字符串”变成“业务数据”。

先看最小思路：

```rust
use crate::error::AppError;
use crate::model::StudyRecord;

pub fn parse_record(line: &str, line_number: usize) -> Result<StudyRecord, AppError> {
    let parts: Vec<&str> = line.split(',').collect();

    if parts.len() != 3 {
        return Err(AppError::InvalidLineFormat {
            line_number,
            content: line.to_string(),
        });
    }

    let subject = parts[0].trim().to_string();
    let score_raw = parts[1].trim();
    let hours_raw = parts[2].trim();

    let score = score_raw.parse::<u32>().map_err(|_| AppError::InvalidScore {
        line_number,
        value: score_raw.to_string(),
    })?;

    let hours = hours_raw.parse::<f64>().map_err(|_| AppError::InvalidHours {
        line_number,
        value: hours_raw.to_string(),
    })?;

    Ok(StudyRecord { subject, score, hours })
}
```

这段代码非常值得你多看几遍，因为它把很多初级阶段知识点都串起来了：

1. `split(',')` 拆字段
2. `trim()` 清理空白
3. `parse()` 做类型解析
4. `map_err()` 转换底层错误
5. `?` 传播错误
6. 返回结构体

## 14. 批量解析时有两种路线

这是这个项目最有教学价值的地方之一。

### 14.1 严格模式：任何一行坏掉都整体失败

```rust
pub fn parse_all(lines: &[String]) -> Result<Vec<StudyRecord>, AppError> {
    lines
        .iter()
        .enumerate()
        .map(|(index, line)| parse_record(line, index + 1))
        .collect()
}
```

这段代码非常漂亮。

因为它直接用：

- `enumerate()`
- `map()`
- `collect::<Result<Vec<_>, _>>()`

把“逐行解析，任何一行失败就整体失败”表达出来了。

### 14.2 宽松模式：坏行跳过，只保留合法数据

```rust
pub fn parse_valid_only(lines: &[String]) -> Vec<StudyRecord> {
    lines
        .iter()
        .enumerate()
        .filter_map(|(index, line)| parse_record(line, index + 1).ok())
        .collect()
}
```

这时你就能直观看到：

- `collect(Result<...>)` 代表整体严格失败
- `filter_map(...ok())` 代表忽略失败项

这正是前一篇迭代器文档讲的内容在项目里的落地。

## 15. `analyzer.rs` 是第二核心

这一层不应该再关心原始字符串。

它应该只处理已经合法的 `StudyRecord`。

例如：

```rust
use crate::model::{StudyRecord, StudySummary};

pub fn summarize(records: &[StudyRecord], total_lines: usize) -> StudySummary {
    let valid_records = records.len();
    let total_hours: f64 = records.iter().map(|record| record.hours).sum();
    let total_score: u32 = records.iter().map(|record| record.score).sum();

    let average_score = if valid_records == 0 {
        0.0
    } else {
        total_score as f64 / valid_records as f64
    };

    let best = records.iter().max_by_key(|record| record.score);

    StudySummary {
        total_lines,
        valid_records,
        total_hours,
        average_score,
        best_subject: best.map(|record| record.subject.clone()),
        best_score: best.map(|record| record.score),
    }
}
```

这段代码里有几个重要点：

1. `map(...).sum()` 做聚合
2. `max_by_key()` 找最高分
3. `Option` 处理“可能没有最佳项”

## 16. `fold` 在这里也很适合练

如果你不想只用 `sum()`，也可以拿 `fold()` 做更综合的累计：

```rust
let (count, total_hours, total_score) = records.iter().fold(
    (0usize, 0.0f64, 0u32),
    |(count, hours, score), record| {
        (count + 1, hours + record.hours, score + record.score)
    },
);
```

这很适合当作练习，因为它会帮你更牢地理解：

- 迭代器不只是“变换元素”
- 也能把多个统计量一次性收束出来

## 17. `report.rs` 负责最后一公里

这一层最适合做的事是：

- 把统计结果变成人能读的文本

例如：

```rust
use crate::model::StudySummary;

pub fn build_report(summary: &StudySummary) -> String {
    let mut output = String::new();
    output.push_str("学习记录分析结果\n");
    output.push_str("====================\n");
    output.push_str(&format!("总行数: {}\n", summary.total_lines));
    output.push_str(&format!("有效记录数: {}\n", summary.valid_records));
    output.push_str(&format!("总学习时长: {:.1}\n", summary.total_hours));
    output.push_str(&format!("平均分: {:.2}\n", summary.average_score));

    match (&summary.best_subject, summary.best_score) {
        (Some(subject), Some(score)) => {
            output.push_str(&format!("最高分科目: {} ({})\n", subject, score));
        }
        _ => output.push_str("最高分科目: 无\n"),
    }

    output
}
```

这段代码也体现了一个很好的边界：

- 统计逻辑不应该写在 `report`
- 输出拼接逻辑不应该写在 `analyzer`

## 18. `lib.rs` 才是总流程编排者

可以把主流程组织成这样：

```rust
pub fn run(config: &Config) -> Result<String, AppError> {
    let lines = reader::read_lines(&config.file_path)?;
    let records = parser::parse_all(&lines)?;
    let summary = analyzer::summarize(&records, lines.len());
    Ok(report::build_report(&summary))
}
```

这个函数非常像一条压缩后的项目主链：

1. 读文件
2. 解析记录
3. 生成统计
4. 构建报告

你会发现，前面那几篇文档里的知识点，在这里都自然连起来了。

## 19. `main.rs` 继续保持很薄

例如：

```rust
use std::env;

use study_analyzer::config::Config;
use study_analyzer::run;

fn main() {
    if let Err(error) = real_main() {
        eprintln!("{}", error);
        std::process::exit(1);
    }
}

fn real_main() -> Result<(), Box<dyn std::error::Error>> {
    let args: Vec<String> = env::args().collect();
    let config = Config::from_args(&args)?;
    let report = run(&config)?;
    println!("{}", report);
    Ok(())
}
```

如果你现在还不想引入 `Box<dyn std::error::Error>`，也完全可以让 `real_main()` 直接返回 `Result<(), AppError>`。

关键不是哪种写法更“高级”，而是：

- 顶层负责展示
- 业务流程别和入口混在一起

## 23. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

这有点像你在 Node.js 里做一个 CSV 风格的小 CLI：

- 读文件
- `split` 每行
- 转对象
- `map/filter/reduce`
- 输出报告

Rust 的关键差别是：

- 解析失败路径必须明确
- 每层职责边界要更清楚

### 和 Python 类比

很像一个小的数据清洗脚本，只是 Rust 会更强迫你：

- 把数据模型写出来
- 把失败路径写出来
- 把模块边界写清楚

### 和 Java 类比

如果你熟悉 Java，这个项目大概能类比成：

- `StudyRecord` / `StudySummary` 像模型类
- `parser` / `analyzer` / `report` 像服务职责切分
- `AppError` 像业务异常模型

但 Rust 的表达方式更偏值和返回类型，而不是面向异常体系本身。

## 24. 常见错误与修正方向

### 错误 1：读取、解析、统计全塞进一个函数

这样最容易让项目失控，也最难测。

### 错误 2：所有失败都统一写成 `String`

短期方便，长期会让错误来源模糊掉。

### 错误 3：坏行该整体失败还是该跳过，没有先做决策

这会导致逻辑来回改。先定策略，再写流程。

### 错误 4：结构体里过早塞借用字段

像 `StudyRecord` 这种要跨模块传递的数据，直接用拥有型字段更稳。

### 错误 5：统计逻辑和输出逻辑混在一起

`analyzer` 负责算结果，`report` 负责展示结果。

## 25. 一个推荐的开发顺序

如果你自己动手做这个项目，建议按这个顺序推进：

1. 先写 `StudyRecord` 和 `StudySummary`
2. 再写 `AppError`
3. 再写 `parse_record()`
4. 再写 `parse_all()`
5. 再写 `summarize()`
6. 再写 `build_report()`
7. 最后把 `run()` 和 `main()` 串起来

这个顺序的好处是：

- 先把数据模型和错误模型站住
- 再搭主流程

这比一上来就在 `main.rs` 里堆逻辑更稳。

3. 在宽松模式下，额外统计“无效行数”
4. 输出时同时展示“总行数 / 有效记录数 / 无效记录数”
5. 给 `parser` 和 `analyzer` 各补至少 3 个测试

额外要求：

1. 至少用一次 `collect::<Result<Vec<_>, _>>()`
2. 至少用一次 `filter_map`
3. 至少解释一次：为什么这一层该返回错误，而不是打印错误
4. 至少解释一次：为什么某个字段选择拥有型，而不是借用型
