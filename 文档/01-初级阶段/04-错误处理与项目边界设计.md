# 错误处理与项目边界设计

## 1. 这一章是干什么的

前面你已经学过：

- `Result<T, E>`
- `?`
- 多模块 CLI 项目

但真实项目一旦稍微复杂一点，你很快就会遇到这类问题：

- 配置解析失败，应该谁来报错
- 文件读取失败，应该在底层打印，还是交给上层处理
- 业务规则不合法，应该继续复用 `std::io::Error`，还是单独建业务错误
- 每一层都 `println!` 一次错误，会不会反而更乱

这一章要解决的重点不是“再学几个错误处理语法”，而是：

- 把错误处理放回到项目边界里理解

也就是：

1. 谁负责发现错误
2. 谁负责把错误往上带
3. 谁负责决定最后怎么展示给用户

这三个问题一旦理顺，你写的 Rust 项目会稳定很多。

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- `Option<T>`
- `Result<T, E>`
- `match`
- `?`
- 基础模块系统

最好已经看过：

- [错误处理进阶](../00-入门阶段/19-错误处理进阶.md)
- [第一个多模块命令行项目](./03-第一个多模块命令行项目.md)

## 3. 先建立一个总原则

错误处理最常见的混乱，通常不是因为语法不会，而是因为职责混了。

先记一个非常实用的原则：

1. 底层函数负责发现错误
2. 中间层函数负责传播或转换错误
3. 顶层入口负责决定怎么展示错误

你可以把这条原则理解成：

- 错误是沿着模块边界流动的

## 4. 什么叫“项目边界”

这里的“边界”，你可以先理解成不同职责层之间的分界线。

例如一个 CLI 小项目里，常见边界至少有这些：

1. 参数输入边界
2. 文件或网络 I/O 边界
3. 业务规则边界
4. 输出展示边界

我们用一个简单例子看：

- `config.rs` 负责把命令行参数变成配置
- `reader.rs` 负责读文件
- `analyzer.rs` 负责做分析
- `main.rs` 负责启动和打印

这些模块就天然构成了边界。

## 5. 为什么“边界不清”会让错误处理变差

看一个常见的反面例子：

```rust
fn read_file(path: &str) -> String {
    match std::fs::read_to_string(path) {
        Ok(content) => content,
        Err(error) => {
            println!("读取失败: {}", error);
            String::new()
        }
    }
}
```

这段代码的问题不只是“返回空字符串不优雅”，更深一层的问题是：

1. 底层函数已经擅自决定了输出方式
2. 上层拿到空字符串后，很难区分“文件真的是空的”还是“读取失败了”
3. 错误信息已经被打印了，但程序可能还会继续走，状态变得模糊

这类代码在 JavaScript、Python 里也会出问题，只是 Rust 更鼓励你从类型层面把这种混乱压掉。

## 6. 更合理的第一步：把错误返回出去

同样逻辑，更合理的第一步通常是：

```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

这里的意思非常清楚：

- 我负责读文件
- 成功返回内容
- 失败返回错误
- 至于怎么展示错误，不是我这一层决定的

这就是边界清晰的开始。

## 7. 底层、中层、顶层各自该做什么

### 7.1 底层

底层通常最接近真实失败来源。

例如：

- 文件不存在
- 权限不足
- 字符串解析失败

底层最常做的事是：

- 返回标准库错误或第三方库错误

### 7.2 中层

中层最常做两类事：

1. 直接传播错误
2. 把底层错误转换成更贴近业务的错误

### 7.3 顶层

顶层通常是：

- `main`
- HTTP 入口
- 命令处理入口

顶层最该做的是：

- 决定给用户看到什么信息
- 决定退出码、日志级别或响应格式

## 8. 一个最小分层例子

先看一套很常见的结构：

```rust
pub fn read_lines(path: &str) -> Result<Vec<String>, std::io::Error> {
    let content = std::fs::read_to_string(path)?;
    Ok(content.lines().map(|line| line.to_string()).collect())
}

pub fn run(path: &str) -> Result<usize, std::io::Error> {
    let lines = read_lines(path)?;
    Ok(lines.len())
}

fn main() {
    match run("log.txt") {
        Ok(total) => println!("总行数: {}", total),
        Err(error) => eprintln!("程序执行失败: {}", error),
    }
}
```

这段代码虽然简单，但已经体现了：

1. 底层不打印错误
2. 中层只传播
3. 顶层统一展示

## 9. 什么时候需要自定义错误

不是所有错误都应该继续用 `std::io::Error`。

一个实用判断是：

如果失败原因已经进入了“你的程序规则”，通常就该开始考虑业务错误。

例如：

- 参数缺失
- 关键词为空
- 文件能读，但内容格式不合法
- 匹配结果为空，而你的程序要求至少有一条结果

这些就不只是 I/O 失败了，而是：

- 业务规则失败

## 10. 自定义错误为什么常用 `enum`

因为错误本来就经常是“几种失败情况之一”。

例如：

```rust
#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    EmptyKeyword,
    Io(std::io::Error),
}
```

这和前面讲 `enum` 的思路完全一致：

- 程序当前处于哪一种失败状态

## 11. `From` 在这里有什么价值

如果你希望底层 `std::io::Error` 能很自然地往上变成你的 `AppError`，可以实现：

```rust
#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    EmptyKeyword,
    Io(std::io::Error),
}

impl From<std::io::Error> for AppError {
    fn from(value: std::io::Error) -> Self {
        Self::Io(value)
    }
}
```

这样当函数返回 `Result<_, AppError>` 时，很多 `std::io::Error` 就能通过 `?` 自动转换上来。

这会让你的中层代码清爽很多。

## 12. `?` 不只是简化语法，它还在维护边界

很多初学者只把 `?` 看成“少写一点 `match`”。

这个理解不算错，但还不够。

更重要的是，`?` 会让代码自然表达出：

- 这层不处理它
- 这层把错误交给上一层

例如：

```rust
pub fn load_config(path: &str) -> Result<String, AppError> {
    let content = std::fs::read_to_string(path)?;
    Ok(content)
}
```

这里的 `?` 不是偷懒，而是在明确说：

- 读取失败时，当前函数不做展示决策

## 13. `Option` 到 `Result` 的转换也很重要

项目里不只是 I/O 会失败。

有时是“值不存在”，但在当前业务里这其实已经是错误。

例如：

```rust
#[derive(Debug)]
pub enum AppError {
    MissingKeyword,
}

fn parse_keyword(args: &[String]) -> Result<&str, AppError> {
    args.get(1)
        .map(|value| value.as_str())
        .ok_or(AppError::MissingKeyword)
}
```

这里的关键点是：

- `Option` 表示“可能没有”
- `Result` 表示“这里没有已经构成错误”

这也是边界设计的一部分。

## 14. 一个更像真实项目的错误边界示例

假设你继续扩展前面的“日志关键词统计器”。

你可能会有这样的模块分工：

### `config.rs`

- 负责参数解析
- 报 `MissingArgument`
- 报 `EmptyKeyword`

### `reader.rs`

- 负责文件读取
- 报 `std::io::Error`

### `analyzer.rs`

- 负责分析内容
- 报业务校验错误，比如 `NoMatchFound`

### `main.rs`

- 统一打印用户可理解的错误信息

这时你就会发现：

- 错误类型和模块职责会自然对齐

## 15. 不同层看到的错误，不一定长得一样

这是一个非常重要的点。

底层错误往往更技术化：

- `No such file or directory`
- `Permission denied`

上层错误往往更偏用户视角：

- 找不到你指定的日志文件
- 关键词不能为空

所以真实项目里很常见的一种做法是：

1. 底层保留原始错误
2. 中层组合成更大的错误类型
3. 顶层再决定展示成什么文本

## 16. 为什么不要每一层都打印错误

看一个常见坏味道：

```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    match std::fs::read_to_string(path) {
        Ok(content) => Ok(content),
        Err(error) => {
            eprintln!("reader error: {}", error);
            Err(error)
        }
    }
}

fn run(path: &str) -> Result<(), std::io::Error> {
    match read_file(path) {
        Ok(_) => Ok(()),
        Err(error) => {
            eprintln!("run error: {}", error);
            Err(error)
        }
    }
}
```

这会导致：

1. 同一个错误被打印多次
2. 输出层级混乱
3. 测试时也容易被多余输出污染

更稳的做法通常是：

- 只在真正的顶层统一打印

## 17. `main` 为什么经常应该很薄

你可以把 `main` 先粗略理解成：

- 程序的展示边界

一个很常见的结构是：

```rust
fn main() {
    if let Err(error) = real_main() {
        eprintln!("程序执行失败: {}", error);
        std::process::exit(1);
    }
}
```

再把真正业务逻辑放进：

```rust
fn real_main() -> Result<(), AppError> {
    // 这里做参数解析、业务调用
    Ok(())
}
```

这会让你的入口更清楚：

- `main` 负责展示和退出
- `real_main` 负责流程编排

## 18. `Display` 和 `Debug` 该怎么理解

初级阶段你至少要先建立这个区分：

### `Debug`

更偏开发者视角，常见形式像：

```rust
{:?}
```

### `Display`

更偏用户视角，常见形式像：

```rust
{}
```

如果你的错误是要直接展示给用户看的，最终通常更希望它支持 `Display`。

例如：

```rust
use std::fmt;

#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    EmptyKeyword,
}

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::MissingArgument => write!(f, "缺少必要参数"),
            AppError::EmptyKeyword => write!(f, "关键词不能为空"),
        }
    }
}
```

## 19. 一个更完整的组合示例

```rust
use std::fmt;

#[derive(Debug)]
pub enum AppError {
    MissingArgument,
    EmptyKeyword,
    Io(std::io::Error),
}

impl From<std::io::Error> for AppError {
    fn from(value: std::io::Error) -> Self {
        Self::Io(value)
    }
}

impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::MissingArgument => write!(f, "请提供日志文件路径和关键词"),
            AppError::EmptyKeyword => write!(f, "关键词不能为空"),
            AppError::Io(error) => write!(f, "读取文件失败: {}", error),
        }
    }
}

pub fn validate_keyword(keyword: &str) -> Result<(), AppError> {
    if keyword.trim().is_empty() {
        Err(AppError::EmptyKeyword)
    } else {
        Ok(())
    }
}
```

这个例子里你能同时看到：

1. 业务错误
2. 底层错误封装
3. 顶层展示文本

## 20. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

JavaScript 常见是 `throw` / `try...catch`。

Rust 更偏向：

- 把失败作为返回值显式带出来

这意味着错误传播链在函数签名上更清楚。

### 和 Python 类比

Python 经常在底层直接抛异常，上层再捕获。

Rust 的不同点是：

- 你更容易一眼从签名看出哪一层可能失败

### 和 Java 类比

Java 里你可能见过：

- checked exception
- runtime exception
- service 层抛，controller 层处理

Rust 在设计味道上有点像“显式失败路径 + 分层处理”，只是它不是主打异常语义，而是主打 `Result` 和类型边界。

## 21. 常见错误与修正方向

### 错误 1：底层直接打印错误然后吞掉

这会让上层失去处理机会，也会让状态变模糊。

### 错误 2：所有错误都强行塞成 `String`

短期看简单，长期会让错误类型信息丢失。

### 错误 3：一开始就设计特别复杂的错误层级

初级阶段先把：

- 参数错误
- I/O 错误
- 业务校验错误

三类分开，就已经很够用了。

### 错误 4：业务层继续暴露太底层的错误细节

有时候你应该保留原始错误，但对用户展示的文案不一定要完全照抄底层错误。

## 22. 一个实用的最小设计模板

如果你现在做一个中小型 CLI，可以先按下面思路设计：

1. `config` 模块解析参数并返回 `Result<Config, AppError>`
2. `reader` 模块做 I/O，必要时返回底层错误
3. `service` 或 `analyzer` 模块做业务规则校验
4. `lib.rs` 组织 `run()` 主流程
5. `main.rs` 统一 `eprintln!` 和退出码

这套结构已经足够支撑很多初级项目。

## 23. 小练习

1. 把一个会返回空字符串代表失败的函数，改成返回 `Result`。
2. 写一个 `AppError`，至少包含“参数缺失”和 “I/O 错误”两种情况。
3. 写一个 `validate_keyword`，当关键词为空时返回业务错误。
4. 把 `Option` 用 `ok_or` 或 `ok_or_else` 转成 `Result`。

## 24. 挑战题

把前一篇“日志关键词统计器”升级一版：

1. 新增 `AppError` 枚举
2. 配置解析失败时返回业务错误
3. 文件读取失败时保留底层 I/O 错误
4. `run()` 只负责传播或组合错误，不直接打印
5. `main()` 统一展示错误并返回非零退出码

额外要求：

1. 至少使用两次 `?`
2. 至少实现一个 `From`
3. 至少解释一次：为什么当前层选择“传播”，而不是“处理”

## 25. 小结

这一章最重要的不是多会几种写法，而是把错误处理重新理解成边界设计：

1. 底层发现错误
2. 中层传播或转换错误
3. 顶层决定展示方式
4. `?` 不只是省代码，它也在表达“这一层不处理它”
5. 错误类型最好和模块职责一起设计
