# 迭代器、闭包与集合组合训练

## 1. 这一章是干什么的

入门阶段你已经见过：

- `iter()`
- `map`
- `filter`
- `collect`
- 最基础的闭包写法

这些已经足够让你看懂很多 Rust 代码。

但一旦你开始写更真实的逻辑，很快就会遇到新的卡点：

- 什么时候该用普通 `for`，什么时候该用迭代器链
- 为什么 `filter` 里参数看起来总像一层又一层引用
- `map`、`filter_map`、`fold`、`find` 到底怎么分工
- 闭包为什么有时能直接用外部变量，有时又会和所有权冲突
- 为什么有些迭代器链写出来很顺，有些却越写越绕

这一章的目标，就是把“会看基础迭代器”推进到“能比较稳定地写组合式数据处理”。

## 3. 初级阶段要升级的，不是“再背几个 API”

这篇最重要的升级点不是记更多方法名，而是建立一个新的写作视角：

- 你处理的不是“一个元素”
- 而是一整条数据流

也就是说，你要开始习惯这样思考：

1. 数据从哪里来
2. 中间经过哪些变换
3. 哪些数据被保留
4. 哪些数据被丢弃
5. 最后收束成什么结果

Rust 的迭代器链，本质上就是在表达这条数据流。

## 4. 什么时候优先用 `for`，什么时候优先用迭代器链

这是一个非常现实的问题。

### 更适合 `for` 的场景

- 逻辑有很多分支
- 中间状态较多
- 需要多次修改外部变量
- 写成链式后反而不直观

### 更适合迭代器链的场景

- 流程很线性
- 是“过滤 -> 变换 -> 汇总”这类组合
- 中间状态不多
- 你想把“怎么处理每个元素”表达得更集中

### 一个实用原则

如果迭代器链能让你更清楚地看到数据流，就用它。

如果链式写法开始遮住逻辑本身，就退回 `for`。

Rust 并不要求你把所有循环都写成一条很长的链。

## 5. 先从一段普通循环改写开始

先看一个你肯定能读懂的版本：

```rust
fn collect_long_words(words: &[String]) -> Vec<String> {
    let mut result = Vec::new();

    for word in words {
        if word.len() > 4 {
            result.push(word.to_uppercase());
        }
    }

    result
}
```

这段代码本身没问题。

它做了两件事：

1. 筛出长度大于 4 的单词
2. 把它们转成大写

用迭代器链改写：

```rust
fn collect_long_words(words: &[String]) -> Vec<String> {
    words
        .iter()
        .filter(|word| word.len() > 4)
        .map(|word| word.to_uppercase())
        .collect()
}
```

这里的优势是：

- 数据流更集中
- 不需要手动建 `result`
- “筛选”和“变换”被拆成了两个非常清楚的步骤

## 6. 读一条迭代器链的正确方式

很多初学者看到长一点的链就容易慌。

其实更稳的读法是：

1. 先找数据源
2. 再看中间每一步在做什么
3. 最后看收束结果

例如这段：

```rust
let result: Vec<String> = words
    .iter()
    .filter(|word| word.len() > 4)
    .map(|word| word.to_uppercase())
    .collect();
```

可以拆成：

1. `words.iter()`：从 `words` 借用出一个只读迭代流
2. `filter(...)`：保留长度大于 4 的元素
3. `map(...)`：把保留下来的元素变成大写字符串
4. `collect()`：收集成 `Vec<String>`

你以后读复杂链时，尽量也按这个顺序拆。

## 7. `map`、`filter`、`filter_map` 到底怎么分

### 7.1 `map`

`map` 负责：

- 每个元素都变一下

```rust
let numbers = vec![1, 2, 3];
let doubled: Vec<i32> = numbers.iter().map(|n| n * 2).collect();
```

### 7.2 `filter`

`filter` 负责：

- 按条件保留或丢弃

```rust
let numbers = vec![1, 2, 3, 4];
let evens: Vec<&i32> = numbers.iter().filter(|n| **n % 2 == 0).collect();
```

### 7.3 `filter_map`

`filter_map` 负责：

- 同时做“筛掉一部分”和“变换留下的部分”

例如一组字符串里，有些能解析成数字，有些不能：

```rust
let raw = vec!["10", "abc", "30"];

let parsed: Vec<i32> = raw
    .iter()
    .filter_map(|item| item.parse::<i32>().ok())
    .collect();
```

这里发生了两件事：

1. 能解析成功的保留
2. 解析失败的直接丢掉

这就是 `filter_map` 的典型价值。

## 8. `filter_map` 为什么在 Rust 里特别常见

因为 Rust 里很多“尝试某事”的结果天然就是：

- `Option<T>`
- `Result<T, E>`

而真实数据处理经常会遇到：

- 某些值合法
- 某些值不合法
- 我只想拿合法那部分继续算

这时 `filter_map` 会非常顺手。

例如：

```rust
fn parse_scores(items: &[String]) -> Vec<u32> {
    items
        .iter()
        .filter_map(|item| item.parse::<u32>().ok())
        .collect()
}
```

它比“先 `map` 成 `Result`，再额外处理一次”更紧凑。

## 9. `find`、`any`、`all` 解决的是另一类问题

不是所有数据流都要收集成新集合。

有时你只是想判断或查找。

### `find`

- 找到第一个满足条件的元素

```rust
let numbers = vec![3, 5, 8, 10];
let first_even = numbers.iter().find(|n| **n % 2 == 0);
```

### `any`

- 是否至少有一个满足条件

```rust
let has_even = numbers.iter().any(|n| n % 2 == 0);
```

### `all`

- 是否全部满足条件

```rust
let all_positive = numbers.iter().all(|n| *n > 0);
```

这一类方法的好处是：

- 一旦结论已经够了，就不必再把全部数据走完并收集起来

## 10. `fold` 是“自己定义收束规则”

`collect()` 是把结果收集成标准容器。

`fold` 则更通用：

- 你自己定义怎么把一串元素压缩成一个最终值

例如求和：

```rust
let numbers = vec![1, 2, 3, 4];
let sum = numbers.iter().fold(0, |acc, n| acc + n);
```

这里：

- `0` 是初始累加器
- `acc` 是当前累计结果
- `n` 是当前元素

### 一个更有代表性的例子

```rust
fn join_non_empty(lines: &[String]) -> String {
    lines
        .iter()
        .filter(|line| !line.trim().is_empty())
        .fold(String::new(), |mut acc, line| {
            if !acc.is_empty() {
                acc.push_str(" | ");
            }
            acc.push_str(line);
            acc
        })
}
```

这个例子说明：

- `fold` 不只是求和
- 它可以把你自己的收束逻辑写得非常明确

## 11. `enumerate` 和 `zip` 是非常实用的组合工具

### `enumerate`

如果你想同时拿到索引和值：

```rust
for (index, item) in items.iter().enumerate() {
    println!("{}: {}", index, item);
}
```

### `zip`

如果你想把两条迭代流一一配对：

```rust
let names = vec!["Alice", "Bob"];
let scores = vec![90, 82];

let pairs: Vec<(&str, i32)> = names
    .into_iter()
    .zip(scores.into_iter())
    .collect();
```

这在“标签和数据对应”“字段和值配对”这类场景里很常见。

## 12. 为什么闭包一和迭代器放在一起就显得重要

因为迭代器方法本质上一直在问你：

- 这个元素要不要保留
- 这个元素要怎么变
- 这个元素怎么参与汇总

这些问题都很适合用短小局部逻辑回答。

而闭包正是最适合承载这种局部逻辑的东西。

例如：

```rust
let keyword = "error";

let matched: Vec<&String> = lines
    .iter()
    .filter(|line| line.contains(keyword))
    .collect();
```

这里闭包直接用到了外部的 `keyword`。

这就是闭包最常见、也最有价值的地方之一。

## 13. 闭包“捕获外部变量”先建立第一轮直觉

初级阶段你不需要一上来死磕 `Fn`、`FnMut`、`FnOnce` 的全部细节。

但你至少要先有这个直觉：

- 闭包可以使用外部作用域里的变量

例如：

```rust
fn main() {
    let limit = 10;
    let bigger_than_limit = |n: &i32| *n > limit;

    let items = vec![8, 10, 12, 20];
    let result: Vec<&i32> = items.iter().filter(|n| bigger_than_limit(n)).collect();

    println!("{:?}", result);
}
```

这里的 `limit` 并不是闭包参数，而是从外部“带进来”的。

## 14. 闭包捕获为什么会和所有权发生关系

因为外部变量一旦被闭包使用，编译器就要判断：

- 它是借用外部变量
- 还是修改外部变量
- 还是把外部变量直接拿走

例如：

```rust
fn main() {
    let prefix = String::from("item:");

    let build = |name: &str| format!("{} {}", prefix, name);

    println!("{}", build("rust"));
    println!("{}", prefix);
}
```

像这种情况，闭包通常只是借用 `prefix` 来读，所以后面还能继续用。

但如果你在闭包里显式把一个值 move 进去，语义就不一样了。

## 15. `move` 闭包的第一轮理解

例如：

```rust
fn main() {
    let keyword = String::from("error");

    let contains_keyword = move |line: &str| line.contains(&keyword);
}
```

这里的 `move` 可以先粗略理解成：

- 把外部捕获到的值所有权搬进闭包环境里

它常见于：

- 需要让闭包拥有自己的数据副本
- 线程、异步、回调之类更独立的执行环境

初级阶段你先记住：

- `move` 不是“更高级的装饰”
- 它会直接改变捕获方式

## 16. 为什么有时链式写法会和借用冲突

看一个常见例子：

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("bb")];

    let short_names: Vec<&String> = names
        .iter()
        .filter(|name| name.len() < 2)
        .collect();

    names.push(String::from("ccc"));

    println!("{:?}", short_names);
}
```

这里容易报错，本质原因是：

- `short_names` 里借用了 `names` 内部元素
- 你随后又想修改 `names`
- `push` 可能导致底层存储变化

这和前面所有权章节的本质是一样的，只是场景换成了迭代器链。

### 两种常见修正方向

#### 方案 1：先把借用作用域用完

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("bb")];

    {
        let short_names: Vec<&String> = names
            .iter()
            .filter(|name| name.len() < 2)
            .collect();

        println!("{:?}", short_names);
    }

    names.push(String::from("ccc"));
}
```

#### 方案 2：需要独立结果时就生成拥有型数据

```rust
fn main() {
    let mut names = vec![String::from("a"), String::from("bb")];

    let short_names: Vec<String> = names
        .iter()
        .filter(|name| name.len() < 2)
        .map(|name| name.clone())
        .collect();

    names.push(String::from("ccc"));

    println!("{:?}", short_names);
}
```

这里虽然有 `clone()` 成本，但换来了独立结果。

## 17. 为什么有些闭包参数看起来有很多 `&`

这通常不是闭包本身在故意复杂，而是因为：

- 迭代器本来就在借用元素
- 某些方法又把“对元素的引用”再传给闭包

例如：

```rust
let evens: Vec<&i32> = numbers.iter().filter(|n| **n % 2 == 0).collect();
```

这里看起来有 `**n`，是因为：

- `iter()` 产出 `&i32`
- `filter` 给闭包的又是“对这个元素项的引用”

初学时你不要过度纠缠外形，可以先抓语义：

- 当前数据到底是值、借用值，还是借用的借用

如果写起来总觉得绕，也可以先拆开写。

## 18. 一个很实用的技巧：链太长就分段

这非常重要。

不要把“会写迭代器”理解成“必须一口气写出十几行链条”。

例如你可以这样分段：

```rust
let non_empty = lines.iter().filter(|line| !line.trim().is_empty());
let errors = non_empty.filter(|line| line.contains("error"));
let result: Vec<&String> = errors.collect();
```

或者这样：

```rust
let trimmed: Vec<&str> = lines.iter().map(|line| line.trim()).collect();
let useful: Vec<&str> = trimmed.into_iter().filter(|line| !line.is_empty()).collect();
```

只要更好读，就是好写法。

## 19. 组合处理 `Option` 和 `Result` 的典型写法

### 19.1 把一组字符串解析成数字，忽略失败项

```rust
fn parse_valid_numbers(items: &[String]) -> Vec<i32> {
    items
        .iter()
        .filter_map(|item| item.parse::<i32>().ok())
        .collect()
}
```

### 19.2 把一组字符串都解析成数字，任何一个失败就整体失败

```rust
fn parse_all_numbers(items: &[String]) -> Result<Vec<i32>, std::num::ParseIntError> {
    items
        .iter()
        .map(|item| item.parse::<i32>())
        .collect()
}
```

这段非常值得你记住。

因为它体现了一个 Rust 里非常漂亮的组合特性：

- 如果迭代器元素是 `Result<T, E>`
- 那么 `collect()` 可以直接把它收束成 `Result<Vec<T>, E>`

这在真实项目里特别有用。

## 20. 为什么 `collect::<Result<Vec<_>, _>>()` 很值得掌握

看一个更显式的版本：

```rust
fn parse_all_numbers(items: &[String]) -> Result<Vec<i32>, std::num::ParseIntError> {
    items
        .iter()
        .map(|item| item.parse::<i32>())
        .collect::<Result<Vec<i32>, _>>()
}
```

这里表达的是：

- 每个元素都尝试解析
- 只要有一个失败，整个结果就是失败
- 全部成功，才得到 `Vec<i32>`

这很像把“逐个 `?`”压缩成了一种集合级别的写法。

## 21. 一个更贴近项目的小例子

假设你有一批日志行：

```rust
fn extract_error_codes(lines: &[String]) -> Vec<u32> {
    lines
        .iter()
        .filter(|line| line.contains("ERROR"))
        .filter_map(|line| line.split('#').nth(1))
        .filter_map(|part| part.trim().parse::<u32>().ok())
        .collect()
}
```

这段链做了几件事：

1. 只保留包含 `ERROR` 的行
2. 取出 `#` 后面的部分
3. 尝试解析成 `u32`
4. 只留下成功解析的错误码

这已经很接近真实 CLI 或日志分析工具里的代码风格了。

## 22. 迭代器链不是为了“短”，而是为了“清楚”

这是一个很关键的判断标准。

不要误以为：

- 行数越少越好
- 链越长越高级

真正更好的标准是：

- 数据流是不是清楚
- 所有权是不是清楚
- 失败路径是不是清楚

如果一条链把这三件事都说明白了，它就是好代码。

## 23. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过：

- `array.map(...)`
- `array.filter(...)`
- `array.some(...)`
- `array.find(...)`

你会很快熟悉这种“数据流式处理”的感觉。

但 Rust 多出来的一层关键约束是：

- 元素到底是被借用、被修改，还是被消费

### 和 Python 类比

Python 里列表推导式、`map`、`filter` 也能完成很多类似任务。

Rust 的不同点是：

- 类型更明确
- 借用关系更明确
- 很多组合会直接和 `Option` / `Result` 连起来

### 和 Java 类比

如果你接触过 Java Stream，会觉得有熟悉感。

但 Rust 的迭代器通常更贴近零成本抽象的风格，而且所有权模型会比 Java Stream 更值得你注意。

## 24. 常见错误与修正方向

### 错误 1：明明只是过滤和变换，却手写很多临时变量

可以尝试先改成简单的链式处理。

### 错误 2：链太长，读者已经看不出每一步在干什么

这时应该分段，或者退回普通循环。

### 错误 3：为了省事，到处 `clone()`

先检查你是不是真的需要拥有型结果。

### 错误 4：该整体失败的场景，却误写成 `filter_map(...ok())`

如果业务要求“任何一项失败都算失败”，更适合用：

- `map(...parse...)`
- `collect::<Result<Vec<_>, _>>()`

### 错误 5：把 `move` 当装饰语法乱加

`move` 会改变捕获方式。加之前先问自己：闭包是不是确实要拥有外部值。

## 25. 一个实用的写作顺序

如果你自己在写一段集合处理逻辑，推荐按这个顺序下手：

1. 先写最普通的 `for`
2. 确认逻辑正确
3. 看看是不是“过滤 -> 变换 -> 汇总”这类线性流程
4. 再把它压成迭代器链
5. 最后检查所有权和返回类型是不是更合理了

这个顺序很适合初级阶段。

因为它能避免你一上来就在语法和逻辑里同时迷路。

## 26. 小练习

1. 写一个函数，接收 `&[String]`，返回所有非空字符串的大写版本。
2. 写一个函数，接收 `&[String]`，解析出其中所有合法的 `u32`。
3. 写一个函数，接收 `&[String]`，如果有任何一个元素不能解析成 `u32` 就整体返回错误。
4. 用 `fold` 手写一个把字符串列表拼成单行文本的函数。
5. 用 `find`、`any`、`all` 分别写三个最小示例。

## 27. 挑战题

做一个“学习记录分析器”数据处理模块，要求：

1. 输入是一组 `Vec<String>`，每行格式类似 `rust,90`、`js,88`、`bad_line`
2. 先筛出格式合法的记录
3. 把分数解析成数字
4. 统计总记录数、有效记录数、平均分
5. 找出分数最高的一条记录

额外要求：

1. 至少用一次 `filter_map`
2. 至少用一次 `fold` 或 `collect::<Result<Vec<_>, _>>()`
3. 至少解释一次：哪些地方选择了忽略失败项，哪些地方选择了整体失败
4. 至少解释一次：哪些结果是借用型，哪些结果是拥有型

## 28. 小结

这一章最重要的升级点是：

1. 迭代器链是在表达一条数据流，不是在炫技巧
2. `map`、`filter`、`filter_map`、`find`、`fold` 各自解决的问题不同
3. 闭包最大的价值之一，是把局部处理规则嵌进数据流
4. 迭代器组合会直接和借用、拥有、`Option`、`Result` 连在一起
5. 写复杂链时，清楚永远比短更重要
