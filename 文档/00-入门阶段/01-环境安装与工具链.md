# Rust 环境安装与工具链

## 1. 这一章是干什么的

这一章解决一个很现实的问题：你要先知道 Rust 到底怎么安装、怎么验证、怎么更新，以及日常开发时到底应该用哪个命令。

很多初学者一开始就混淆这 3 个名字：

- `rustup`
- `rustc`
- `cargo`

如果这 3 个东西的职责你不分清，后面看教程和报错会一直糊。

## 2. 学这一章前，你需要知道什么

只要你知道“命令行里可以执行程序”就够了。

如果你学过这些语言，可以这样类比：

- Python：你可以把 `python` 想成解释器，把 `pip` 想成包管理工具
- Java：你可以把 `javac` 想成编译器，把 Maven/Gradle 想成工程工具
- JavaScript：你可以把 `node` 想成运行时，把 `npm`/`pnpm` 想成包管理和脚本入口

Rust 也有类似分工，但它的整合度更高。

## 3. 核心概念

### 3.1 `rustup` 是工具链管理器

`rustup` 负责安装和管理 Rust 工具链。

你可以先把它理解成：

- 安装 Rust 的入口
- 切换版本的入口
- 管理不同 toolchain 的入口

它更像“总管家”，而不是你每天直接拿来编译项目的主命令。

常见职责：

- 安装 stable/nightly
- 更新 Rust
- 查看当前默认 toolchain

### 3.2 `rustc` 是编译器

`rustc` 负责把 Rust 源码编译成可执行文件或其他目标产物。

如果你只写一个单文件实验程序，也可以直接调用 `rustc`：

```powershell
rustc main.rs
```

但在真实项目里，你通常不会长期手动直接使用 `rustc` 管理整个工程，因为工程会涉及：

- 依赖
- 配置
- 构建模式
- 测试
- 发布

这些事情单靠 `rustc` 不方便统一管理。

### 3.3 `cargo` 是项目入口

`cargo` 是 Rust 最重要的日常命令。

它既是：

- 构建工具
- 依赖管理工具
- 项目脚手架工具
- 测试入口
- 运行入口

你后面大部分时间真正打交道的是 `cargo`，而不是裸 `rustc`。

最常见命令：

```powershell
cargo new hello_rust
cargo run
cargo build
cargo test
```

如果只记一个结论，那就是：

在真实学习和项目开发里，优先把 `cargo` 当主入口。

## 4. 安装 Rust

### 4.1 官方推荐方式

Rust 官方推荐通过 `rustup` 安装工具链。

Windows PowerShell 常见安装方式：

```powershell
winget install Rustlang.Rustup
```

安装完成后，重新打开终端，再检查版本：

```powershell
rustup --version
rustc --version
cargo --version
```

### 4.2 当前这台机器的状态

我已经在当前环境里检查过，当前机器已经安装好 Rust 工具链，结果是：

```text
rustup 1.29.0
rustc 1.95.0
cargo 1.95.0
```

这意味着你在这个仓库里可以直接开始学，不需要先补装。

### 4.3 安装后你应该看到什么

如果安装正常，至少应满足这 3 件事：

1. `rustup --version` 有输出
2. `rustc --version` 有输出
3. `cargo --version` 有输出

如果其中某个命令找不到，通常不是 Rust 本身坏了，而是：

- 终端没有重开
- PATH 没刷新
- 安装没有完成

## 5. 为什么 Rust 这么强调工具链

很多语言也有工具链，但 Rust 的工程体验高度依赖工具链的一致性。

原因在于 Rust 把很多事情做成了统一入口：

- 建项目
- 加依赖
- 编译
- 运行
- 测试
- 文档
- 格式化
- 静态检查

如果类比你已有语言经验：

- Python 生态自由度很高，但工具链组合很多
- JavaScript 工具很多，灵活但也容易碎片化
- Java 工程规范强，但配置和构建体系通常更重
- Rust 倾向于默认就给你一套比较统一的路线

这也是为什么 Rust 初学阶段很值得先把工具链摸熟。

## 6. 你现在最该记住的命令

下面这些命令，后面会反复出现：

```powershell
rustup --version
rustc --version
cargo --version
cargo new demo_project
cargo run
cargo build
cargo check
cargo test
```

这里特别解释两个容易混淆的命令：

### `cargo build`

真正执行编译，生成构建产物。

### `cargo check`

只做类型检查和语义检查，不生成完整可执行产物，通常更快。

初学时你会频繁改代码、看报错，所以 `cargo check` 很有用。

## 7. 和 JavaScript、Python、Java 的类比

### 和 Python 类比

你可以粗略地把：

- `rustc` 类比成某种“底层编译器角色”
- `cargo` 类比成“解释器入口 + 包管理 + 项目命令入口”的组合

但这个类比只能帮助上手，不能完全等价，因为 Python 默认不是编译型工作流。

### 和 Java 类比

你可以把：

- `rustc` 类比成 `javac`
- `cargo` 类比成一个更一体化的 Gradle/Maven + run/test 入口

Rust 的一体化体验通常比 Java 初学体验更轻。

### 和 JavaScript 类比

你可以把：

- `cargo new` 类比成创建项目脚手架
- `cargo run` 类比成某种统一的项目脚本入口

但 Rust 不像 Node.js 项目那样默认依赖一个解释型运行时，它最终要生成二进制程序。

## 8. 初学者常见错误

### 错误 1：用 `rustc` 手动管理整个项目

刚开始可以用 `rustc` 编译单文件实验，但一旦进入真实项目，应该尽快转向 `cargo`。

### 错误 2：看到版本输出就以为一切都懂了

工具装好只是起点，不等于你已经理解了项目结构、构建过程和依赖管理。

### 错误 3：一出错就怀疑安装坏了

很多时候只是命令用错了、路径不对，或者在错误目录执行了 `cargo` 命令。

例如，在没有 `Cargo.toml` 的目录里执行：

```powershell
cargo run
```

通常就会报错，因为 Cargo 找不到项目根。

## 9. 如何排查环境问题

如果你后面遇到“命令找不到”或“版本不对”，按这个顺序检查：

1. 先执行 `rustup --version`
2. 再执行 `rustc --version`
3. 再执行 `cargo --version`
4. 确认是不是重开了终端
5. 确认当前目录是不是正确项目目录

你可以把这个思路理解成：先确认工具存在，再确认工具可用，最后确认你站的位置对不对。

## 10. 小练习

1. 解释一下：`rustup`、`rustc`、`cargo` 三者谁更像“安装管理者”，谁更像“编译器”，谁更像“项目主入口”。
2. 在你自己的话里说明：为什么实际开发里更应该优先使用 `cargo`。
3. 试着记忆 `cargo build` 和 `cargo check` 的差别。

## 11. 挑战题

在当前终端里依次执行下面命令，并把输出记到你自己的学习笔记里：

```powershell
rustup --version
rustc --version
cargo --version
```

然后再用自己的话写 5 行总结：

1. 我的机器当前是否已经安装 Rust
2. 当前版本大概是什么
3. 我以后最常用的命令会是谁
4. `rustc` 和 `cargo` 的关系是什么
5. 为什么我不应该把三者混为一谈

## 12. 小结

这一章你最需要拿走的结论只有 4 个：

1. `rustup` 负责管理工具链
2. `rustc` 是编译器
3. `cargo` 是日常开发主入口
4. 学 Rust，先熟悉工具链，会让后面所有学习成本明显下降
