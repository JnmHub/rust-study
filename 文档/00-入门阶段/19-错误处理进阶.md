# 错误处理进阶

## 1. 这一章是干什么的

前面你已经学过：

- `Option<T>`
- `Result<T, E>`
- `match`
- `expect()` / `unwrap()`

这些已经足够让你开始认真面对“成功”和“失败”。

这一章则往前再走一步，解决更真实的问题：

- 每一层都手写 `match` 会不会太啰嗦？
- 错误怎么向上返回？
- `?` 到底在干什么？
- 自定义错误类型最小应该怎么理解？

这一章的目标不是把你一下带进完整工程化错误体系，而是让你从“我知道有 `Result`”走到“我开始能写像样的错误传播代码”。

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- `Result<T, E>`
- 函数返回值
- 文件读取与命令行参数

这里先带着一个总印象往下看：

Rust 的错误处理不只是“打印错误然后结束”，它更强调：

- 哪一层负责发现错误
- 哪一层负责决定怎么处理错误

## 3. 手写 `match` 为什么会越来越啰嗦

先看一个你已经能读懂的写法：

```rust
use std::fs;

fn read_file(path: &str) -> Result<String, std::io::Error> {
    match fs::read_to_string(path) {
        Ok(content) => Ok(content),
        Err(error) => Err(error),
    }
}
```

这当然没错。

但如果你仔细看，会发现这里本质上是在做：

- 成功就原样返回成功
- 失败就原样返回失败

这类写法一多，代码会很快变得重复。

## 4. `?` 是什么

Rust 提供了一个非常重要的简化工具：

- `?`

看同样逻辑的更常见写法：

```rust
use std::fs;

fn read_file(path: &str) -> Result<String, std::io::Error> {
    let content = fs::read_to_string(path)?;
    Ok(content)
}
```

这可以先这样理解：

- 如果成功，就把内部值拿出来继续往下走
- 如果失败，就立刻把错误返回给上层

也就是说，`?` 在帮你做“失败就提前返回”的动作。

## 5. `?` 的第一轮心智模型

初学时你可以先这样记：

```rust
let value = some_result?;
```

大致相当于：

```rust
let value = match some_result {
    Ok(v) => v,
    Err(e) => return Err(e),
};
```

这不是它的全部细节，但已经足够支撑你读懂大量入门到中级代码。

## 6. `?` 只能用在什么地方

一个非常关键的限制是：

- `?` 只能用在“当前函数本身也能返回失败”的上下文里

例如：

```rust
use std::fs;

fn read_file(path: &str) -> Result<String, std::io::Error> {
    let content = fs::read_to_string(path)?;
    Ok(content)
}
```

这里没问题，因为函数返回的是 `Result<..., ...>`。

但如果你写：

```rust
use std::fs;

fn read_file(path: &str) -> String {
    let content = fs::read_to_string(path)?;
    content
}
```

这通常就不行，因为这个函数签名根本没说自己能返回错误。

所以你要建立一个直觉：

- 想用 `?`
- 你的函数就得承认“我也可能失败”

## 7. 多步错误传播

真实代码里，经常不只一步可能失败。

例如：

```rust
use std::fs;

fn read_and_trim(path: &str) -> Result<String, std::io::Error> {
    let content = fs::read_to_string(path)?;
    Ok(content.trim().to_string())
}
```

这里的流程很自然：

1. 读取文件
2. 如果失败，直接返回错误
3. 如果成功，拿到内容继续处理
4. 最后返回处理后的成功值

这就是 `?` 最常见的日常价值。

## 8. `main` 和错误处理

前面你经常在 `main` 里这样写：

```rust
match read_file(path) {
    Ok(content) => println!("{}", content),
    Err(error) => println!("read failed: {}", error),
}
```

这仍然是完全合理的，因为：

- 某一层总要真正决定怎么展示错误

你可以先这样分工理解：

- 底层函数：尽量返回 `Result`
- 上层流程：决定是提示用户、重试，还是直接结束

## 9. 自定义错误的第一轮理解

当项目复杂一点时，你可能不想只返回标准库错误，而是想表达：

- 参数不够
- 文件为空
- 内容格式不对

这时就需要自定义错误类型。

最小直觉可以先从枚举开始：

```rust
enum AppError {
    MissingArgument,
    EmptyFile,
}
```

这很符合你已经学过的 Rust 风格：

- 错误本身也是一种状态
- 状态适合用 `enum`

## 10. 一个最小自定义错误例子

```rust
#[derive(Debug)]
enum AppError {
    MissingArgument,
    EmptyFile,
}

fn validate_content(content: &str) -> Result<(), AppError> {
    if content.trim().is_empty() {
        Err(AppError::EmptyFile)
    } else {
        Ok(())
    }
}
```

这里重点先不是完整错误生态，而是先学会：

- 你可以用自己的 `enum` 表达业务错误

## 11. 错误类型为什么常写成 `enum`

因为一个程序常见的失败情况本来就是“几种可能之一”。

例如：

- 参数缺失
- 文件读取失败
- 内容不合法
- 解析失败

这天然就很适合枚举建模。

## 12. 标准库错误和业务错误怎么区分

你可以先建立一个简单区分：

### 标准库错误

例如：

- 文件不存在
- 权限不足
- 解析失败

这类错误往往来自标准库或第三方库。

### 业务错误

例如：

- 用户没传参数
- 文件虽然能读，但内容不符合你的程序要求

这类更像“你的程序规则”。

后面你会慢慢学到怎么把它们整理到同一个错误体系里。

## 13. 一个稍微完整一点的思路

假设你做文件统计工具，可以先这样分层：

```rust
#[derive(Debug)]
enum AppError {
    MissingArgument,
    EmptyFile,
    Io(std::io::Error),
}
```

这表示：

- 有自己的业务错误
- 也可能包一层底层 `io` 错误

你现在先不用急着掌握更高级写法，只要先知道：

- 错误不一定只有一种来源

## 14. 为什么 `unwrap()` 不能成为默认方案

前面练习里用 `unwrap()` 没问题，但如果你开始写真实工具，就要慢慢升级思维：

- `unwrap()` = “失败就直接崩”

这在学习时方便，在真实工具里通常不够友好。

更成熟一点的思路通常是：

1. 底层函数返回 `Result`
2. 中间层用 `?` 传播
3. 顶层决定怎么展示错误

## 15. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

JavaScript 常见是 `throw` / `try...catch`。Rust 不默认走异常流，而更偏向：

- 用返回值把失败显式带出来

### 和 Python 类比

Python 的异常体系很强，但 Rust 更喜欢在函数签名层面让你一眼看出“这里可能失败”。

### 和 Java 类比

Java 也有显式异常和抛出传播的感觉，Rust 的不同点是它更多通过 `Result` 和 `?` 这种值级别机制来表达，而不是主要依赖异常。

## 16. 常见错误与如何调试

### 错误 1：函数返回普通值，却硬用 `?`

先回头检查函数签名是不是该改成 `Result<..., ...>`。

### 错误 2：所有地方都手写大段 `match`

如果只是“出错就往上抛”，优先想一想是不是可以用 `?`。

### 错误 3：一上来自定义超复杂错误体系

入门阶段先会：

- 返回 `Result`
- 用 `?`
- 写一个简单错误枚举

就已经很够用了。

### 错误 4：把 `unwrap()` 当长期默认方案

短期方便，长期会让错误体验和代码边界都变差。

## 17. 小练习

1. 把一个手写 `match` 返回 `Result` 的函数改成 `?` 写法。
2. 写一个函数，读取文件后做一次简单校验，并继续返回 `Result`。
3. 定义一个最小 `AppError` 枚举。
4. 写一个返回 `Result<(), AppError>` 的校验函数。

## 18. 挑战题

把你前面设计的 CLI 小工具升级一下：

1. 参数缺失时返回自定义错误
2. 文件读取失败时返回底层错误
3. 文件为空时返回业务错误
4. 中间层尽量用 `?` 做传播
5. 在最顶层统一打印错误提示

额外要求：

1. 至少定义一个错误枚举
2. 至少用两次 `?`
3. 至少解释一次：为什么这里适合传播，而不是当场 `unwrap()`

## 19. 小结

这一章最关键的理解是：

1. `?` 用来简化错误传播
2. `?` 的核心直觉是：成功继续，失败提前返回
3. 想用 `?`，函数通常也要返回 `Result`
4. 自定义错误常适合用 `enum`
5. 更成熟的错误处理通常是“底层返回，中层传播，顶层决定怎么展示”
