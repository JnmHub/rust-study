# 模块、包与文件组织

## 1. 这一章是干什么的

到这里，你已经不再只是写几行单文件代码了。

接下来很自然会遇到一个问题：

- 代码一多，应该怎么拆？
- 一个 Rust 项目里，包、crate、模块到底是什么关系？
- 为什么有时候写 `mod`，有时候写 `use`，有时候还要写 `pub`？

这一章的目标就是把 Rust 的基础组织方式讲清楚，让你从“能写小例子”过渡到“能把代码放进合理结构里”。

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- Cargo 项目的基本结构
- 函数、结构体、方法的基础写法
- 路径和文件的基本概念

这里先带着一个总印象往下看：

Rust 的模块系统不是为了炫技，而是为了回答三件事：

1. 代码分到哪里
2. 谁能访问谁
3. 别处怎么引用它

## 3. 先分清几个词

### 3.1 package

你可以先把 package 理解成：

- 一个 Cargo 项目
- 通常由 `Cargo.toml` 描述

### 3.2 crate

crate 可以先粗略理解成：

- Rust 编译的一个单元

常见两种：

- 二进制 crate
- 库 crate

### 3.3 module

module 是代码内部的组织单元。

你可以先理解成：

- 一个命名空间
- 一个逻辑分组

## 4. `src/main.rs` 和 `src/lib.rs`

### 4.1 `main.rs`

如果项目是可执行程序，常见入口是：

- `src/main.rs`

这对应二进制 crate。

### 4.2 `lib.rs`

如果项目要提供可复用代码，常见入口是：

- `src/lib.rs`

这对应库 crate。

很多初学阶段项目只有 `main.rs` 就够了，但理解 `lib.rs` 的存在会帮你更早建立工程感。

## 5. 最简单的模块

先看一个单文件模块例子：

```rust
mod math {
    pub fn add(a: i32, b: i32) -> i32 {
        a + b
    }
}

fn main() {
    let result = math::add(2, 3);
    println!("{}", result);
}
```

这里有 3 个关键点：

1. `mod math` 定义了一个模块
2. `pub fn add` 表示这个函数对模块外可见
3. 调用时用 `math::add`

## 6. `pub` 是什么

Rust 默认是偏保守的，很多东西默认不对外开放。

如果你写：

```rust
mod math {
    fn add(a: i32, b: i32) -> i32 {
        a + b
    }
}
```

那么 `add` 默认只在模块内部可见。

如果你想让模块外能用，就要显式写：

- `pub`

这和 Rust 一贯的风格很一致：

- 默认收紧
- 需要开放时显式说明

## 7. `use` 是什么

如果路径太长，可以把它引入当前作用域：

```rust
mod math {
    pub fn add(a: i32, b: i32) -> i32 {
        a + b
    }
}

use math::add;

fn main() {
    println!("{}", add(2, 3));
}
```

这里 `use` 的作用可以先理解成：

- 把某个路径下的名字带进当前上下文

这样调用时就不用每次写完整路径。

## 8. 拆到单独文件

单文件足够小时很方便，但代码一多就应该拆文件。

例如项目结构：

```text
src/
├── main.rs
└── tools.rs
```

### `src/main.rs`

```rust
mod tools;

fn main() {
    tools::greet();
}
```

### `src/tools.rs`

```rust
pub fn greet() {
    println!("Hello from tools!");
}
```

这里要注意：

1. `main.rs` 里写 `mod tools;`
2. Rust 会去找同名文件 `tools.rs`
3. 里面如果有想对外用的函数，要写 `pub`

## 9. 路径的第一轮理解

看这段代码：

```rust
mod tools {
    pub fn greet() {
        println!("Hello!");
    }
}

fn main() {
    crate::tools::greet();
}
```

这里 `crate::` 可以先理解成：

- 从当前 crate 根开始找

你现在不需要把路径系统学到很细，但至少先记住：

- `模块名::项目名`
- 是 Rust 常见的访问风格

## 10. 为什么要尽早学模块组织

因为一旦代码超过一个文件，你就必须开始回答：

- 哪些函数该放一起
- 哪些类型该暴露出去
- 哪些实现细节应当藏起来

如果不建立这个意识，后面项目一变大就会乱。

## 11. 一个稍微真实一点的拆分思路

假设你做一个学习工具项目，可能会这样分：

- `main.rs`：程序入口
- `models.rs`：结构体和枚举
- `service.rs`：处理逻辑
- `utils.rs`：小工具函数

你现在还不需要把分层做得很复杂，但至少要知道：

- 文件拆分不是随便切
- 而是按职责组织

## 12. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过 `import/export`，会比较容易接受 Rust 里的模块拆分。

但 Rust 会更显式地要求你说明：

- 这是模块定义
- 这是可见性
- 这是导入

### 和 Python 类比

Python 里一个 `.py` 文件天然就是模块，这会给你一种很轻的组织体验。Rust 也支持按文件拆模块，但会更明确地通过 `mod` 和 `pub` 表达结构与边界。

### 和 Java 类比

Java 初学者会对“包路径 + 类组织”有熟悉感。Rust 也重视结构，只是它的模块系统和 Java package 不是一模一样的概念。

## 13. 常见错误与如何调试

### 错误 1：拆了文件，但忘了写 `mod`

你建了 `tools.rs`，不代表 Rust 自动知道要把它纳入编译。通常还要在入口或上层模块里声明：

- `mod tools;`

### 错误 2：函数没写 `pub`

如果模块外访问不到，先检查是不是默认私有了。

### 错误 3：把 `mod` 和 `use` 混了

你可以先这样记：

- `mod`：声明模块
- `use`：引入路径中的名字

### 错误 4：路径写错

如果报找不到函数、找不到模块，优先检查：

1. 文件名是否对应
2. `mod` 是否声明
3. `pub` 是否开放
4. 调用路径是否写对

## 14. 小练习

1. 在单文件里定义一个 `mod math`，里面放一个 `pub fn add`。
2. 用 `use` 引入这个函数后再调用一次。
3. 把一个模块函数拆到独立文件里。
4. 故意去掉 `pub`，观察模块外为什么访问失败。

## 15. 挑战题

做一个“小型学习项目拆分练习”，要求：

1. 新建一个 Cargo 项目
2. 至少拆成 `main.rs` 和两个额外模块文件
3. 一个模块放数据类型
4. 一个模块放处理逻辑
5. 在 `main` 里调用它们

额外要求：

1. 至少用一次 `pub`
2. 至少用一次 `use`
3. 至少自己修一次“忘写 `mod` 或忘写 `pub`”导致的错误

## 16. 小结

这一章最关键的理解是：

1. package、crate、module 是不同层次的组织概念
2. `mod` 用来声明模块
3. `pub` 控制可见性
4. `use` 用来把路径中的名字引入当前作用域
5. 代码一多，就必须开始按职责拆文件和模块
