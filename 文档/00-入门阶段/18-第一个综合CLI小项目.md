# 第一个综合 CLI 小项目

## 1. 这一章是干什么的

前面这一整段入门阶段，已经把很多关键拼图铺好了：

- Cargo 项目结构
- 变量与函数
- 所有权、借用、字符串
- 结构体、枚举、`Option`、`Result`
- 集合、`impl`、模块组织
- 测试
- 命令行参数与文件读取

这一章的目标就是把这些内容第一次真正串成一个“像样的小项目”。

你不会在这里学到一个全新语法点，重点是：

- 怎么把已经学过的东西组合起来
- 怎么从题目走到结构设计
- 怎么避免一上来把代码写乱

## 2. 项目目标

建议你做一个：

- 文本统计 CLI

它的输入是一个文本文件，输出可以包括：

- 总字符数
- 总单词数
- 总行数
- 某个关键词出现次数

这类项目为什么适合入门阶段？

因为它正好会用到：

- 参数读取
- 文件读取
- `String` 处理
- `Vec`
- `HashMap`
- `Result`
- 模块拆分
- 测试

## 3. 项目需求先别写代码，先拆题

先把问题拆成几个子任务：

1. 从命令行拿文件路径
2. 读取文件内容
3. 统计基础信息
4. 如果有关键词参数，再统计关键词频次
5. 把结果格式化输出

这一步很重要。

很多初学者一上来就写代码，结果很快把：

- 输入处理
- 统计逻辑
- 输出逻辑

全揉在一起。

Rust 学习阶段更应该养成先拆题的习惯。

## 4. 推荐项目结构

你可以先做一个很轻量的结构：

```text
src/
├── main.rs
├── stats.rs
└── output.rs
```

一个可行的职责划分是：

- `main.rs`：处理参数和总流程
- `stats.rs`：统计逻辑
- `output.rs`：格式化输出

如果你暂时不想拆太多文件，也可以先只拆成：

- `main.rs`
- `stats.rs`

重点不是拆得多，而是拆得有理由。

## 5. 推荐数据结构设计

你可以定义一个结果结构体：

```rust
struct TextStats {
    line_count: usize,
    word_count: usize,
    char_count: usize,
}
```

如果你还想做关键词频率统计，可以再加：

```rust
keyword_count: usize
```

或者把词频表设计成：

```rust
HashMap<String, usize>
```

## 6. 推荐函数划分

你可以先这样拆：

```rust
fn parse_args(...) -> ...
fn read_file(...) -> Result<String, _>
fn calculate_stats(...) -> TextStats
fn count_keyword(...) -> usize
fn print_stats(...) -> ()
```

这会让你的代码层次清楚很多。

一个很重要的原则是：

- 每个函数尽量只负责一件事

## 7. 一个可行的思路草图

### `main.rs` 负责流程

它主要做：

1. 拿参数
2. 调用读文件函数
3. 调用统计函数
4. 输出结果

### `stats.rs` 负责计算

它主要做：

1. 行数统计
2. 单词数统计
3. 字符数统计
4. 关键词计数

### `output.rs` 负责展示

它主要做：

1. 把结果格式化打印得更清晰

## 8. 为什么这个项目特别适合你现在做

因为它能非常真实地暴露你目前是否已经掌握了这些关键点：

1. 你能不能把 `main` 写得不臃肿
2. 你会不会正确处理 `Result`
3. 你能不能在函数参数里合理使用 `&str`
4. 你会不会用 `struct` 表达结果
5. 你会不会用模块拆分
6. 你能不能给核心统计逻辑补测试

也就是说，这不是“再写一个练习”，而是一次阶段验收。

## 9. 推荐最小版本

如果你怕一开始做太大，可以先只做最小版本：

1. 接收一个文件路径
2. 读取内容
3. 输出行数、单词数、字符数

先把这 3 项做对。

然后再迭代：

4. 支持额外关键词参数
5. 支持词频统计
6. 支持更好看的输出

这比一上来就追求“大而全”更稳。

## 10. 测试该怎么加

最值得测试的不是 `main`，而是“纯逻辑函数”。

例如：

- `calculate_stats`
- `count_keyword`

因为这些函数：

- 输入明确
- 输出明确
- 最容易写断言

示意：

```rust
#[test]
fn should_count_words_correctly() {
    let text = "Rust is fast";
    let stats = calculate_stats(text);
    assert_eq!(stats.word_count, 3);
}
```

这就是你前一章测试基础真正落地的地方。

## 11. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过 Node.js 小工具，这个项目会让你有熟悉感。但 Rust 会更强调：

- 错误处理显式
- 字符串和借用边界清晰
- 模块职责更明确

### 和 Python 类比

Python 写这个工具会更快，但 Rust 会更迫使你在结构和边界上更认真。前期更慢，长期更稳。

### 和 Java 类比

Java 也能做这个工具，但样板可能更重。Rust 做 CLI 小工具的体验通常更轻，同时保持编译期约束。

## 12. 常见错误与如何调试

### 错误 1：所有逻辑都堆进 `main`

这会让你很快失控。先拆函数，再考虑拆模块。

### 错误 2：统计逻辑和输出逻辑混在一起

这会让测试变难。尽量让“计算”和“展示”分开。

### 错误 3：遇到 `Result` 就一路 `unwrap()`

学习阶段可以局部这么做，但这个项目更建议你开始练习友好错误输出。

### 错误 4：一开始就追求“什么都支持”

先做最小可用版本，再迭代。

## 13. 建议你真正动手时的顺序

1. `cargo new text_stats_cli`
2. 先让程序能拿到参数
3. 再让程序能读文件
4. 再实现基础统计
5. 再把统计结果放进 `struct`
6. 再拆模块
7. 再加测试
8. 最后再加关键词统计等扩展功能

## 14. 项目挑战要求

建议你把这个项目当作一次小型毕业作业，至少满足：

1. 使用命令行参数
2. 使用文件读取
3. 使用 `Result`
4. 使用一个 `struct`
5. 使用至少一个模块
6. 使用至少一个测试

如果你能把这些完整串起来，说明你的入门阶段已经不只是“看懂语法”，而是开始具备独立写小工具的能力。

## 15. 扩展方向

如果你做完基础版，还可以继续加：

1. 输出最常见的前 5 个单词
2. 忽略大小写统计关键词
3. 支持多个文件输入
4. 用 `HashMap` 做完整词频统计
5. 给输出加更清晰的格式布局

## 16. 小结

这一章最关键的目标不是再教你一个知识点，而是把前面的知识真正串成工程动作：

1. 从需求拆解开始
2. 用函数和模块组织逻辑
3. 用 `Result` 处理失败路径
4. 用 `struct` 表达结果
5. 用测试守住核心逻辑

到这里，入门阶段已经形成了比较完整的一圈。再往下，你就可以开始进入更偏“初级 Rust 工程能力”的内容了。
