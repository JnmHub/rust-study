# 文件读写与命令行参数

## 1. 这一章是干什么的

到这里，你已经基本具备了写小型 Rust 程序的核心语言基础。

接下来很自然就要进入更真实的程序行为：

- 从命令行拿参数
- 读取文件内容
- 处理结果并输出

这一章会把这几件事串起来，因为它们正是很多 CLI 小工具的起点。

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- `String` / `&str`
- `Result`
- 模块和函数基础

这里先带着一个现实问题往下看：

如果一个程序只能打印固定内容，那它更像练习题；如果它能读参数、读文件、处理输入，它才开始更像真正的工具。

## 3. 命令行参数是什么

当你执行：

```powershell
cargo run -- hello.txt
```

这里的 `hello.txt` 就是传给程序的命令行参数。

Rust 标准库里最常见的入口是：

- `std::env::args`

## 4. 最小参数读取例子

```rust
use std::env;

fn main() {
    let args: Vec<String> = env::args().collect();
    println!("{:?}", args);
}
```

这段代码会把命令行参数收集成一个 `Vec<String>`。

你通常会看到：

- 第 0 个参数：程序名
- 后面才是你真正传入的参数

所以如果你执行：

```powershell
cargo run -- hello.txt
```

那么常见情况是：

- `args[0]` 类似程序路径
- `args[1]` 是 `hello.txt`

## 5. 为什么经常要先检查参数数量

因为用户可能没传够参数。

例如：

```rust
use std::env;

fn main() {
    let args: Vec<String> = env::args().collect();

    if args.len() < 2 {
        println!("usage: cargo run -- <file_path>");
        return;
    }

    println!("file path = {}", args[1]);
}
```

这就是非常典型的 CLI 防御式写法：

- 先检查输入够不够
- 不够就提示

## 6. 读取文件最小例子

Rust 标准库里最常见的读取整文件写法之一：

```rust
use std::fs;

fn main() {
    let content = fs::read_to_string("hello.txt");
    println!("{:?}", content);
}
```

这里 `read_to_string` 返回的不是直接的 `String`，而是：

- `Result<String, std::io::Error>`

也就是说：

- 读取可能成功
- 也可能失败

这和你上一章学的 `Result` 完全接上了。

## 7. 用 `match` 处理文件读取结果

```rust
use std::fs;

fn main() {
    match fs::read_to_string("hello.txt") {
        Ok(content) => println!("{}", content),
        Err(error) => println!("read failed: {}", error),
    }
}
```

这段代码非常值得多看两遍，因为它把：

- 标准库函数
- `Result`
- `match`

很自然地连在一起了。

## 8. 把参数和文件读取串起来

现在我们可以把前两部分合并：

```rust
use std::env;
use std::fs;

fn main() {
    let args: Vec<String> = env::args().collect();

    if args.len() < 2 {
        println!("usage: cargo run -- <file_path>");
        return;
    }

    let file_path = &args[1];

    match fs::read_to_string(file_path) {
        Ok(content) => println!("{}", content),
        Err(error) => println!("read failed: {}", error),
    }
}
```

这已经是一个真正有输入、有处理、有结果的小工具雏形了。

## 9. 为什么 `cargo run --` 里要有 `--`

很多初学者第一次都会疑惑这个。

在：

```powershell
cargo run -- hello.txt
```

里，`--` 的作用可以先简单理解成：

- 前面的是给 Cargo 的
- 后面的是给你的程序的

如果没有它，Cargo 可能会把后面的内容当成自己的参数来理解。

## 10. 用函数整理读取逻辑

随着代码变大，最好别把所有逻辑都塞进 `main`。

例如：

```rust
use std::fs;

fn read_file(path: &str) -> Result<String, std::io::Error> {
    fs::read_to_string(path)
}
```

然后在 `main` 里处理结果：

```rust
use std::env;

fn main() {
    let args: Vec<String> = env::args().collect();

    if args.len() < 2 {
        println!("usage: cargo run -- <file_path>");
        return;
    }

    match read_file(&args[1]) {
        Ok(content) => println!("{}", content),
        Err(error) => println!("read failed: {}", error),
    }
}
```

这样结构会更清晰：

- `main` 负责流程
- 其他函数负责细节

## 11. 命令行参数和所有权的小提醒

这里：

```rust
let file_path = &args[1];
```

拿到的是借用，不是把那个参数字符串 move 出来。

这类细节你现在已经能读懂了，这说明你前面的所有权和借用内容已经开始真正服务真实代码。

## 12. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你写过 Node.js CLI，会觉得“拿参数 -> 读文件 -> 输出结果”这个流程很熟悉。Rust 的不同点在于：

- 读取结果是 `Result`
- 借用和所有权会更明确

### 和 Python 类比

Python 里常用 `sys.argv` 和 `open(...)`，概念上很接近。Rust 的不同点是：

- 你要更明确地处理错误分支
- 类型信息更早暴露在函数签名里

### 和 Java 类比

Java 也可以读 `args` 和文件，但写法通常更重。Rust 标准库做这件事相对直接，同时保留了很强的类型和错误边界表达。

## 13. 常见错误与如何调试

### 错误 1：忘了检查参数数量

如果直接访问：

```rust
args[1]
```

而用户没传参数，就可能 panic。

### 错误 2：路径不存在

`read_to_string` 失败不一定是代码坏了，很多时候只是：

- 文件名错了
- 路径不对
- 当前工作目录不对

### 错误 3：一上来就 `unwrap()`

学习时可以，但真实工具里更应该给用户更明确的错误提示。

### 错误 4：把所有逻辑都塞进 `main`

代码一长就难读。尽量开始把流程和细节拆开。

## 14. 小练习

1. 写一个程序，打印出所有命令行参数。
2. 写一个程序，如果没传参数就打印 usage。
3. 写一个函数，读取指定文件并返回 `Result<String, _>`。
4. 在 `main` 中用 `match` 打印读取成功或失败的结果。

## 15. 挑战题

做一个“文件预览器”CLI，要求：

1. 接收一个文件路径参数
2. 读取文件内容
3. 打印文件前几行或前一部分内容
4. 如果路径不存在，给出友好提示

额外要求：

1. 至少写一个返回 `Result` 的函数
2. 至少用一次 `match`
3. 至少自己试一次“少传参数”和“传错路径”的情况

## 16. 小结

这一章最关键的理解是：

1. `std::env::args` 可以获取命令行参数
2. `fs::read_to_string` 是基础文件读取入口
3. 文件读取通常返回 `Result`
4. CLI 程序应当先检查参数，再处理输入
5. 真实程序的结构，通常要比“所有代码都写在 main 里”更清晰
