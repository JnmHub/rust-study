# 测试基础

## 1. 这一章是干什么的

到这里，你已经能写不少 Rust 基础代码了。

但下一步很自然的问题是：

- 我怎么确认它是对的？
- 每次改完代码，难道都靠手动运行看输出吗？
- 如果我后面重构了，怎么知道有没有把功能改坏？

这一章的目标就是让你建立最基础的测试习惯。

Rust 的测试工具链起点非常统一：

- 写测试
- 跑测试
- 看断言结果

基本都围绕 `cargo test` 展开。

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- 函数返回值
- 模块基础
- `assert` 这类“检查是否满足预期”的直觉

这里先带着一句话往下看：

测试不是“把代码写完以后可选地补一下”，而是帮助你更稳定地学习和重构代码。

## 3. 最小测试长什么样

先看一个最小例子：

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_add() {
        assert_eq!(add(2, 3), 5);
    }
}
```

这里有几个关键点：

1. `#[cfg(test)]` 表示这个模块只在测试时编译
2. `mod tests` 是测试模块的常见命名
3. `use super::*;` 表示把上层作用域里的内容带进来
4. `#[test]` 标记这是一个测试函数

## 4. `cargo test`

运行测试最常见的命令就是：

```powershell
cargo test
```

它会：

1. 编译测试代码
2. 运行被标记为 `#[test]` 的测试函数
3. 汇总通过和失败结果

你可以把它理解成：

- Rust 项目的统一测试入口

## 5. 常见断言

### 5.1 `assert_eq!`

检查两个值是否相等：

```rust
#[test]
fn test_add() {
    assert_eq!(2 + 3, 5);
}
```

### 5.2 `assert_ne!`

检查两个值是否不相等：

```rust
#[test]
fn test_not_equal() {
    assert_ne!(2 + 2, 5);
}
```

### 5.3 `assert!`

检查某个条件是否为真：

```rust
#[test]
fn test_positive() {
    let value = 10;
    assert!(value > 0);
}
```

## 6. 一个稍微真实一点的例子

```rust
fn is_even(n: i32) -> bool {
    n % 2 == 0
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn even_number_should_return_true() {
        assert!(is_even(4));
    }

    #[test]
    fn odd_number_should_return_false() {
        assert!(!is_even(3));
    }
}
```

这里的测试名也开始有一点“描述行为”的味道了，这比只写 `test1`、`test2` 更清晰。

## 7. `#[should_panic]`

如果某段代码本来就应该 panic，也可以测试这一点：

```rust
fn always_fail() {
    panic!("failed on purpose");
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    #[should_panic]
    fn test_should_panic() {
        always_fail();
    }
}
```

这表示：

- 如果测试过程没有 panic，反而算失败

你现在只要先知道有这个机制，不需要一上来大量使用。

## 8. 为什么测试模块里常写 `use super::*;`

因为测试模块通常嵌套在被测试代码下面。

例如：

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::*;
}
```

这里 `super` 可以先理解成：

- 上一层模块

这样测试就能直接访问当前文件里定义的函数和类型。

## 9. 测试放在哪里

入门阶段你先掌握最常见的一种就够：

- 和代码放在同一文件里
- 用 `#[cfg(test)] mod tests`

后面你还会接触更完整的集成测试目录，但现在先别铺太开。

## 10. 为什么测试对初学者特别重要

因为你在学 Rust 时会频繁遇到：

- 改一点代码
- 修一个编译报错
- 重构一处所有权写法
- 拆一层模块

这时测试能帮你快速确认：

- 我修编译器的时候，有没有把原来的行为改坏

所以测试不是只服务于大项目，它对学习阶段也很有价值。

## 11. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你接触过 Jest 或 Vitest，就能比较快接受“写断言 + 运行统一测试命令”的思路。Rust 只是把这套入口更统一地收进了 Cargo。

### 和 Python 类比

如果你写过 `pytest`，你会很容易理解“一个个测试函数 + 断言”的风格。Rust 的不同点在于它更常通过属性如 `#[test]` 来标记测试。

### 和 Java 类比

如果你见过 JUnit，那你会对“测试方法 + 断言 + 统一运行入口”不陌生。Rust 的起步写法往往比 Java 测试栈更轻。

## 12. 常见错误与如何调试

### 错误 1：忘了写 `#[test]`

如果测试函数没有这个标记，`cargo test` 通常不会把它当作测试运行。

### 错误 2：断言方向写反

例如本来该比较“实际值”和“预期值”，结果写乱了。虽然多数时候仍然能跑，但报错信息会更难读。

### 错误 3：测试函数依赖打印而不是断言

只 `println!` 不能构成有效测试。测试的核心是：

- 明确检查某个行为是否符合预期

### 错误 4：修改了代码，但没有补或改测试

如果行为已经变了，旧测试可能不再准确。测试不是一劳永逸的，它要跟着代码和意图一起更新。

## 13. 小练习

1. 写一个 `square` 函数，并给它补一个 `assert_eq!` 测试。
2. 写一个 `is_positive` 函数，并用 `assert!` 测试。
3. 写一个故意 panic 的函数，并试试 `#[should_panic]`。
4. 运行一次 `cargo test`，观察输出格式。

## 14. 挑战题

把你前面写过的一个小练习程序重构成“函数 + 测试”的形式，要求：

1. 至少抽出两个函数
2. 至少为它们各写一个测试
3. 至少用一次 `assert_eq!`
4. 至少用一次 `assert!`

额外要求：

1. 自己故意让一个测试失败一次
2. 看清失败输出
3. 再修回来

## 15. 小结

这一章最关键的理解是：

1. Rust 的统一测试入口通常是 `cargo test`
2. `#[test]` 用来标记测试函数
3. `assert_eq!`、`assert_ne!`、`assert!` 是最常见断言
4. 测试模块常写成 `#[cfg(test)] mod tests`
5. 测试不仅服务于项目，也服务于你学习和重构的稳定性
