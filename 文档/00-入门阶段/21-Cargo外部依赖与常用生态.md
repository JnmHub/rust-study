# Cargo 外部依赖与常用生态

## 1. 这一章是干什么的

到这里，你已经能用标准库做不少事情了。

但真实项目里，你几乎不可能永远只用标准库。很快你就会需要：

- 命令行参数库
- 序列化库
- 异步运行时
- 错误处理辅助库

这一章的目标就是帮你建立第一轮生态直觉：

- Cargo 怎么管理外部依赖
- `Cargo.toml` 里依赖怎么写
- 为什么 crates.io 很重要
- 入门阶段最值得知道的几个常见库是什么

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- `Cargo.toml`
- 基本项目结构
- 模块与包的概念

这里先带着一句话往下看：

Rust 很强调统一工具链，而依赖管理正是这种统一体验里最关键的一块。

## 3. 外部依赖写在哪里

Rust 项目最常见的依赖配置位置就是：

- `Cargo.toml`

例如：

```toml
[dependencies]
serde = "1"
```

这表示：

- 项目依赖 `serde`
- 版本要求大致在 `1.x` 这个主版本范围

你现在先不用把版本语义抠得太细，先知道：

- 依赖是写在 `[dependencies]` 下面的

## 4. Cargo 加依赖的基本流程

最朴素的方式就是手动改 `Cargo.toml`：

```toml
[dependencies]
clap = "4"
```

然后执行：

```powershell
cargo build
```

Cargo 会自动：

1. 解析依赖
2. 下载依赖
3. 编译依赖
4. 把它们接进你的项目

这也是为什么 Rust 的工具链体验经常被认为比较统一。

## 5. crates.io 是什么

你可以先把 crates.io 理解成：

- Rust 生态最核心的公共包仓库

它有点像：

- JavaScript 的 npm
- Python 的 PyPI

你平时看到的很多第三方库，通常都能在 crates.io 找到。

## 6. 最小外部依赖例子

比如我们先举一个很常见的序列化库：

```toml
[dependencies]
serde = { version = "1", features = ["derive"] }
```

你现在先不用深究 `features` 细节，只先知道：

- 有些库除了版本，还会带额外功能开关

这在 Rust 生态里很常见。

## 7. 为什么 Rust 生态里常看到 `features`

因为很多库会提供：

- 核心能力
- 可选附加能力

例如：

- 只用基础功能
- 或额外启用宏、derive、JSON 支持等

这也是 Rust 生态常见的“默认尽量明确、按需开启”的风格。

## 8. 入门阶段最值得先知道的几个常见库

这里不是让你现在全学，而是先知道名字和用途。

### `clap`

用途：

- 更像样地处理命令行参数

你前面已经用过 `std::env::args`，但真实 CLI 项目经常会逐步转向 `clap`。

### `serde`

用途：

- 序列化 / 反序列化

以后你会经常在 JSON、配置文件、接口数据里看到它。

### `tokio`

用途：

- 异步运行时

现在先知道名字就够，后面学 async 会碰到。

### `anyhow`

用途：

- 更方便地组织应用层错误

现在你已经在学错误处理进阶，后面很容易遇到它。

### `thiserror`

用途：

- 更方便地写自定义错误类型

适合和你前面学的错误枚举思路接起来。

## 9. 什么时候该用第三方库

你可以先建立一个务实判断：

### 值得考虑第三方库的时候

- 标准库写起来明显麻烦
- 某类问题已经有事实标准库
- 你不是在练基础实现，而是在做工具

### 先别急着上的时候

- 你还没搞清楚标准库最小做法
- 当前任务本来就是练核心语言
- 你只是为了“显得高级”而加依赖

## 10. 一个命令行库的直觉例子

假设你用标准库读参数：

```rust
let args: Vec<String> = std::env::args().collect();
```

这没问题。

但如果你的参数开始变复杂，比如：

- 必填参数
- 可选参数
- `--help`
- 默认值

这时像 `clap` 这种库的价值就会上来。

所以你可以先把生态理解成：

- 当需求上来后，生态会帮你少踩很多重复坑

## 11. 依赖不是越多越好

这一点很重要。

Rust 虽然生态很强，但入门阶段要避免一个误区：

- 什么都想加库解决

更好的顺序通常是：

1. 先用标准库理解问题
2. 再在明确知道痛点后引入库

这样你不会变成“会拷代码但不懂在干什么”。

## 12. 和 JavaScript、Python、Java 的类比

### 和 JavaScript 类比

如果你见过 npm / pnpm，就会很容易理解 Cargo 依赖管理。但 Rust 项目通常不会像 JavaScript 生态那样那么容易碎片化，统一度更高。

### 和 Python 类比

如果你用过 pip / pyproject，你会很容易接受“项目配置里声明依赖”的思路。Rust 的不同点是编译和依赖管理结合得更紧。

### 和 Java 类比

如果你接触过 Maven / Gradle，会对“项目描述文件里写依赖”很熟悉。Rust 的上手写法通常更轻量。

## 13. 常见错误与如何调试

### 错误 1：把依赖名写错

如果 Cargo 找不到库，先检查：

- 名字是不是 crates.io 上的真实名字

### 错误 2：加了依赖但忘了 `use`

Cargo 下载到项目里，不代表代码里自动带进当前作用域。

### 错误 3：一上来把项目依赖堆太多

这样会让你分不清：

- 是语言问题
- 还是库问题

### 错误 4：根本没理解标准库做法就急着上库

这会让你的基础变得很虚。

## 14. 小练习

1. 打开任意一个你自己的 Rust 项目，看看 `Cargo.toml`。
2. 手动写一个假的 `[dependencies]` 例子，理解格式。
3. 记住 3 个常见库名：`clap`、`serde`、`tokio`。
4. 用你自己的话说清：为什么现在还不急着把它们全装进项目里。

## 15. 挑战题

做一个“生态观察笔记”，要求：

1. 记录 `clap`、`serde`、`tokio`、`anyhow`、`thiserror` 分别大概干什么
2. 给每个库写一句“我未来会在什么场景下用到它”
3. 再写一段总结：为什么入门阶段不能一上来全靠第三方库

## 16. 小结

这一章最关键的理解是：

1. 外部依赖主要写在 `Cargo.toml` 的 `[dependencies]` 下
2. Cargo 会统一处理下载、解析和构建
3. crates.io 是 Rust 最核心的公共包仓库
4. 常见生态库要先知道用途，不必一上来全部深学
5. 更好的学习顺序通常是“先懂标准库，再用生态补效率”
