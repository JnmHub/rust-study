# 所有权初识

## 1. 这一章是干什么的

这一章是你真正开始碰 Rust 核心思想的地方。

如果前面几章主要在解决“代码怎么写”，这一章开始解决的是“值在 Rust 里到底怎么活、怎么转移、为什么编译器会阻止某些写法”。

很多人学 Rust 卡住，不是因为不会 `if` 或 `for`，而是因为没有真正理解：

- 一个值归谁管
- 赋值时发生了什么
- 函数传参时发生了什么
- 为什么有些值传过去之后，原变量就不能再用了

这几个问题，都是所有权在回答。

## 2. 学这一章前，你需要知道什么

你最好已经理解：

- 变量和可变性
- 函数参数与返回值
- 基本控制流

另外先接受一个心理准备：

这一章可能是你第一次明显感觉“Rust 和我以前学的语言真的不一样”。

这不是坏事。恰恰说明你开始接触 Rust 的核心价值，而不是只停留在表面语法。

## 3. 为什么 Rust 要有所有权

先不急着背规则，先看问题。

很多语言在运行时依赖：

- 垃圾回收
- 引用计数
- 或者程序员手动管理内存

Rust 想同时尽量做到：

- 安全
- 高性能
- 不依赖 GC

那它就必须在编译期尽量搞清楚：

- 这个值什么时候创建
- 这个值由谁拥有
- 什么时候该释放
- 有没有可能出现悬垂引用或重复释放

所有权就是 Rust 用来管理这些问题的核心规则系统。

## 4. 先建立第一个心智模型

你可以先把所有权粗略理解成：

每个值在 Rust 里都要有一个主要负责人，这个负责人通常就是某个变量绑定。

当负责人换了，旧负责人就不能再假装自己还拥有那个值。

这句话现在先不用完全吃透，但后面所有例子都围着它转。

## 5. 所有权的三条入门规则

第一轮先记这 3 条：

1. Rust 中每个值都有一个所有者
2. 同一时刻，一个值只能有一个所有者
3. 所有者离开作用域后，值会被清理

这 3 条是后面借用、生命周期、智能指针等内容的地基。

## 6. 作用域与值的生命周期直觉

```rust
fn main() {
    {
        let name = String::from("Rust");
        println!("{}", name);
    }

    // 这里 name 已经离开作用域
}
```

在这个例子里，`name` 只在内层代码块有效。离开作用域后，它对应的值会被清理。

这件事在很多语言里你平时不太需要直面，但 Rust 把它变成了你必须理解的规则。

## 7. `String` 为什么比整数更能体现所有权

先看这个例子：

```rust
fn main() {
    let x = 5;
    let y = x;

    println!("x = {}, y = {}", x, y);
}
```

这没问题。

但再看：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;

    println!("{}", s1);
}
```

这里会报错，因为 `s1` 的值已经被移动给 `s2` 了。

为什么整数没事，字符串却有事？

因为像 `i32` 这样的简单类型通常是 `Copy` 的，赋值时会直接复制一份简单值；而 `String` 拥有堆上的数据，Rust 为了避免不安全和重复释放，默认会做 move，而不是隐式深拷贝。

## 8. move 到底是什么

你可以先把 move 理解成：

- 所有权转移
- 不是“复制一份”
- 是“这个值现在归新变量管了”

示例：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;

    println!("{}", s2);
}
```

这里合法，因为拥有者已经变成 `s2`。

但 `s1` 已经不能再用。

## 9. 为什么 Rust 不默认深拷贝

因为深拷贝可能：

- 成本高
- 行为不透明
- 在大型程序里让性能变得不可预测

Rust 倾向于要求你显式表达：

- 你是要转移所有权
- 还是要复制数据

如果你真的要复制 `String` 的内容，可以用 `clone()`：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();

    println!("s1 = {}, s2 = {}", s1, s2);
}
```

这时候两份数据都存在，但你也明确承担了复制成本。

## 10. 函数传参也会触发所有权变化

这一点是新手经常第一次真正“撞墙”的地方。

```rust
fn print_string(text: String) {
    println!("{}", text);
}

fn main() {
    let s = String::from("hello");
    print_string(s);

    println!("{}", s);
}
```

这里会报错，因为 `s` 传入函数时，所有权被移动进了 `print_string`。

函数结束后，`text` 离开作用域，值被清理。

也就是说，函数调用不是“只是把值拿去看看”，它可能是把拥有权交过去。

## 11. 返回值也可能转移所有权

```rust
fn make_message() -> String {
    String::from("hello")
}

fn main() {
    let msg = make_message();
    println!("{}", msg);
}
```

这里返回的 `String` 所有权被交给了 `msg`。

再看另一个例子：

```rust
fn give_back(text: String) -> String {
    text
}

fn main() {
    let s = String::from("hello");
    let s = give_back(s);

    println!("{}", s);
}
```

这里发生的是：

1. `s` 的所有权移进函数
2. 函数再把它返回出来
3. 新的 `s` 重新接住这个所有权

这也是为什么前面你已经在基础语法里接触过 shadowing，它在 Rust 里很常用。

## 12. `Copy` 和 `clone` 的第一轮区别

你现在先不要把 trait 细节学太深，只要先知道：

### `Copy`

像整数、布尔、字符这类简单类型，赋值时通常直接复制，不会让原变量失效。

```rust
fn main() {
    let a = 10;
    let b = a;

    println!("{} {}", a, b);
}
```

### `clone()`

对于像 `String` 这种拥有堆数据的类型，如果你想保留两份独立数据，通常要显式调用 `clone()`。

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();

    println!("{} {}", s1, s2);
}
```

你现在先只记住：

- `Copy` 通常是轻量复制
- `clone()` 通常是显式复制
- move 是所有权转移

## 13. 和 JavaScript、Python、Java 的类比

### 和 Python 类比

Python 里变量名更像“名字绑定到对象”，重新赋值通常只是换个绑定，不会让你明确感知“所有权失效”。

比如：

```python
a = "hello"
b = a
print(a)
```

这没什么特别的心理负担。

Rust 不一样。对于拥有资源的值，它会认真追问：

- 现在到底谁负责这份数据？
- 能不能同时有多个独立拥有者？

### 和 JavaScript 类比

JavaScript 里对象赋值更像“多个变量指向同一个对象引用”，但语言不会在编译期拦你：

```js
const a = { name: "rust" };
const b = a;
console.log(a.name);
```

Rust 不走这条默认路线。它更强调责任归属和资源释放规则。

### 和 Java 类比

Java 里对象变量本质上更像引用，底层资源管理交给 GC。你通常不需要想“这个对象什么时候被释放”。Rust 则要求这件事在设计上更清晰，所以你会比写 Java 时更早接触“值流动”的概念。

## 14. 常见错误与如何调试

### 错误 1：值 move 之后还继续用

```rust
let s1 = String::from("hello");
let s2 = s1;
println!("{}", s1);
```

如果你看到报错里提到：

- moved value
- borrow of moved value

先别急着看一大段英文。先问自己：

1. 这个值是不是已经交给别人了？
2. 我现在是不是还想继续用旧名字？

### 错误 2：函数拿走了所有权

```rust
fn print_it(s: String) {
    println!("{}", s);
}
```

如果你把 `String` 按值传进去，通常就是把所有权交进去了。

### 错误 3：乱用 `clone()`

很多新手一报错就疯狂加 `clone()`。这能暂时过编译，但会让你错过真正理解所有权的机会。

更好的顺序是：

1. 先弄清楚为什么 move 了
2. 再判断是该转移、该返回、还是该借用
3. 最后才决定是否真的要 `clone()`

## 15. 小练习

1. 写一个程序，对 `i32` 做赋值复制，观察原变量还能不能用。
2. 写一个程序，对 `String` 做赋值，观察原变量会不会失效。
3. 把一个 `String` 传进函数，再尝试在函数调用后继续使用它，读懂报错。
4. 用 `clone()` 修正上一题，再理解为什么它能工作。

## 16. 挑战题

做一个“消息传递实验”程序，要求你分别实现下面 3 种情况：

1. 把一个 `String` 直接 move 给另一个变量
2. 把一个 `String` clone 一份给另一个变量
3. 把一个 `String` 传进函数，再从函数里返回回来

每种情况都打印结果，并在代码注释里写一句话说明：

- 为什么原变量还能用
- 或为什么原变量不能用了

额外要求：

1. 自己故意写出一次 move 之后继续使用旧变量的错误代码
2. 读一遍编译器报错
3. 用你自己的话总结“move 到底是什么意思”

## 17. 小结

这一章你最需要拿走的不是“会背三条规则”，而是开始建立值流动直觉：

1. Rust 中每个值都有所有者
2. `String` 这类拥有资源的值，赋值和传参时常发生 move
3. move 不是复制，而是所有权转移
4. 简单类型常是 `Copy`
5. `clone()` 是显式复制，不该拿来无脑消灭报错
