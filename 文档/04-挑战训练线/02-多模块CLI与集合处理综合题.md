# 多模块 CLI 与集合处理综合题

## 1. 这一章是干什么的

这一组题开始不再只练单点，而是把几个已经学过的能力组合起来：

1. 多模块组织
2. CLI 参数
3. 文件读写
4. 集合统计
5. 错误处理
6. 简单测试

如果说热身题是在练：

- 你能不能过 Rust 基本边界

那综合题就在练：

- 你能不能把几个知识点拼成一个小工具

## 2. 学这一章前，你需要知道什么

建议你已经阅读过：

- [第一个多模块命令行项目](../01-初级阶段/03-第一个多模块命令行项目.md)
- [错误处理与项目边界设计](../01-初级阶段/04-错误处理与项目边界设计.md)
- [迭代器、闭包与集合组合训练](../01-初级阶段/05-迭代器闭包与集合组合训练.md)

## 3. 这一组题怎么做

这组题非常适合按“先设计再写代码”的方式来做：

1. 先定项目结构
2. 先定义输入输出
3. 先设计数据模型
4. 再拆模块
5. 最后补测试

不要再把所有逻辑直接塞进 `main.rs`。

## 4. 综合题主题：学习记录统计 CLI

题目目标：

做一个命令行工具，读取一份学习记录文本，输出统计结果。

输入样例可以是：

```text
rust,95,3.5
python,88,2.0
rust,91,4.0
```

输出至少包含：

1. 总记录数
2. 总学习时长
3. 平均分
4. Rust 相关记录数量

## 5. 约束条件

要求至少拆成这些模块：

1. `main`
2. `cli`
3. `parser`
4. `analyzer`
5. `error`

如果你愿意，也可以继续拆：

1. `model`
2. `output`

这里最重要的是建立：

- 模块职责边界

## 6. 题目 1：先做最小可跑版本

要求：

1. 从命令行接收文件路径
2. 读取文件内容
3. 逐行解析
4. 输出基础统计

提示：

1. 第一版先别急着做很复杂的容错
2. 先把主流程跑通

## 7. 题目 2：补错误边界

要求：

1. 文件不存在时要有明确错误
2. 某行格式错误时要能定位到第几行
3. 分数字段非法时要能区分出来

提示：

1. 不要所有错误都塞成一个 `String`
2. 想一想哪些错误属于 I/O，哪些属于解析或业务规则

## 8. 题目 3：补集合与迭代器分析

要求：

1. 统计每个课程出现次数
2. 找出最高分记录
3. 找出学习时长超过某个阈值的记录

提示：

1. 这部分很适合练 `iter`、`filter`、`map`、`fold`
2. 同时思考什么时候简单 `for` 反而更清楚

## 9. 题目 4：补测试

要求：

1. 至少给解析逻辑写测试
2. 至少给统计逻辑写测试
3. 至少覆盖一条错误路径

提示：

1. 先测纯函数最稳
2. 别让测试全都依赖真实文件 I/O

## 10. 你最该练的不是代码量，而是这 4 个能力

做完这题后，最值得复盘的通常是：

1. 你有没有把职责拆干净
2. 你有没有把错误边界讲清楚
3. 你有没有为了快把所有结构都做成 `String`
4. 你有没有用最小测试覆盖关键逻辑

## 11. 常见错误与修正方向

### 错误 1：所有逻辑都写进 `main.rs`

这会让综合题失去训练意义。

### 错误 2：解析、分析、输出混在一起

模块边界要先拆。

### 错误 3：错误信息不区分来源

调用方和使用者都会更难排查。

### 错误 4：集合分析全靠临时变量乱堆

先定义想要的输出结构。

## 12. 小练习

1. 先不写代码，只画出你准备怎么拆模块。
2. 为“单行解析失败”设计一个错误类型。
3. 用一句话解释为什么统计逻辑适合单独抽成 `analyzer`。

## 13. 挑战题

把这道综合题再升级一版，要求：

1. 支持按课程名过滤
2. 支持输出为纯文本或 Markdown
3. 支持跳过错误行并给出告警统计
4. 保持模块结构清晰
5. 给核心逻辑补测试

额外要求：

1. 至少解释一次：为什么这里值得拆 `output` 模块
2. 至少解释一次：如果以后继续扩展，这个 CLI 最容易先乱掉的是哪一层

## 14. 小结

这一组综合题最重要的是把“会语法”推进到“会组织一个小工具”：

1. 先拆模块
2. 再理数据
3. 再定错误边界
4. 最后用测试保护核心逻辑
