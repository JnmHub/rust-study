# 挑战训练线导读

## 1. 这一章是干什么的

主线教程解决的是：

- 你把概念学明白

挑战训练线解决的是：

- 你能不能真的自己下手

也就是说，这条线的重点不是“再讲一遍知识点”，而是训练你：

1. 看到题目时如何拆解
2. 看到编译报错时如何定位
3. 看到所有权冲突时如何重构
4. 看到工程需求时如何分模块
5. 看到性能或并发问题时如何做第一轮判断

如果把主线比作“系统讲课”，那挑战线更像：

- 带任务的实战训练

## 2. 学这一章前，你需要知道什么

建议你至少已经阅读过：

- [学习路线总览](../../学习路线总览.md)
- [学习指南](../../学习指南.md)

如果你刚学 Rust，不需要等“全学完”再来挑战。

更合适的方式是：

1. 入门阶段后做热身题
2. 初级阶段中后段做综合题
3. 高级或精通阶段开始做更真实的挑战题

## 3. 挑战训练线和主线教程的区别

主线教程更偏：

1. 讲概念
2. 讲模型
3. 讲最小例子
4. 讲常见错误

挑战训练线更偏：

1. 给需求
2. 给约束
3. 给提示
4. 要你自己做设计和取舍

所以你会明显感觉到阅读方式不同：

- 主线是先理解
- 挑战线是先动手，再回头对照理解

## 4. 这一条训练线怎么分层

这条线分成三档：

1. 热身题
2. 综合题
3. 挑战题

### 热身题

重点是：

- 单点强化

例如：

- 所有权
- 借用
- `Result`
- 集合遍历

### 综合题

重点是：

- 组合多个知识点

例如：

- 多模块 CLI
- 文件读写
- 统计逻辑

### 挑战题

重点是：

- 更接近真实项目

例如：

- 并发任务
- async 流程
- 配置与日志

## 5. 做题时最重要的不是“立刻写出来”

Rust 初学者最容易犯的一个错是：

- 题目一来就直接开始写

更成熟的顺序通常是：

1. 先确定输入输出
2. 先拆数据结构
3. 先判断哪些地方会涉及所有权
4. 先决定是否需要多模块
5. 最后才开始编码

这条顺序很值钱，因为 Rust 的很多报错，本质上都来自：

- 设计边界一开始没想清楚

## 6. 这条训练线应该怎么用

最推荐的做法通常是：

1. 先独立做
2. 卡住后只看提示，不先看完整答案
3. 做完后写一份复盘
4. 回头再对照主线教程查漏补缺

你可以把每道题都当成一次小型工程练习，而不是：

- 只要结果能跑就行

## 7. 复盘时要重点写什么

每次练完，最值得记录的通常是：

1. 你最开始怎么想
2. 真正卡住的是哪条借用或边界
3. 最后是靠改数据结构、改返回值还是改模块边界解决的
4. 哪个编译报错你现在已经能看懂了
5. 哪个地方你下次会提前避免

这会比单纯记“题做过了”更有价值。

## 8. 和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

JavaScript 里很多题你可以先“写出来再说”，后面慢慢修。

Rust 更常要求你：

- 设计先过一轮，再写

### 和 Python 类比

Python 小脚本常常可以边写边改。

Rust 则更容易把“边界设计问题”提前暴露给你，这也是训练线价值很大的原因。

### 和 Java 类比

Java 背景会让你对：

- 模块拆分
- 类型设计
- 工程边界

这些事情不陌生。

Rust 的不同点是：

- 所有权和借用会把这些设计问题推得更显式

## 10. 小练习

1. 写下你现在最怕的 3 类 Rust 题目。
2. 写下你做题时最容易直接跳过的设计步骤。
3. 选一个你已经做过的小项目，回忆其中最卡的一次编译报错。

## 11. 挑战题

为你自己设计一份“训练线使用规则”，要求：

1. 规定每周做题频率
2. 规定每次做题后必须写的复盘项
3. 规定卡住多久后才看提示
4. 规定什么时候回头补主线教程

额外要求：

1. 至少解释一次：为什么 Rust 做题不能只靠“先写出来再改”
2. 至少解释一次：为什么复盘比单纯刷题更重要

## 12. 小结

挑战训练线的价值，不是额外堆题，而是把主线知识真正逼成能力：

1. 主线负责讲清楚
2. 训练线负责逼你自己动手
3. 热身题练单点
4. 综合题练组合
5. 挑战题练真实工程判断
