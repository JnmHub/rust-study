# 并发、async 与工程化挑战题

## 1. 这一章是干什么的

这一组题进入训练线里最接近真实项目的一档。

它不再只要求你：

- 把函数写对

而是要求你开始处理：

1. 并发任务
2. async 流程
3. 配置和日志
4. 错误边界
5. 工程结构
6. 性能与观测直觉

这类题的目标是训练你从“会写 Rust”逐步走向：

- 能处理真实复杂度

## 2. 学这一章前，你需要知道什么

建议你已经阅读过：

- [多线程、共享状态与 channel](../02-高级阶段/04-多线程共享状态与channel.md)

## 3. 这一组题怎么做

这组题不适合一上来就直接写完所有功能。

更稳的推进顺序通常是：

1. 先做同步最小版本
2. 再引入 async
3. 再引入并发
4. 再补日志和错误边界
5. 最后再考虑观测与性能

也就是说：

- 先跑通，再升级复杂度

## 4. 挑战题主题：异步学习数据聚合器

题目目标：

实现一个小型工具，从多个数据源并发获取学习数据，最后汇总成报告。

你可以先假设数据源都是 HTTP 接口，返回 JSON。

最终输出至少包含：

1. 每个数据源抓取状态
2. 成功获取的数据条数
3. 总学习时长
4. 平均分
5. 失败摘要

## 5. 第一阶段：先做结构设计

开始编码前，先回答这些问题：

1. 配置长什么样
2. 数据模型长什么样
3. 抓取层和分析层怎么分开
4. 错误边界怎么分层
5. 输出层是否单独模块化

建议至少拆成：

1. `config`
2. `domain`
3. `infra`
4. `app`
5. `interface`

## 6. 第二阶段：补 async 流程

要求：

1. 使用异步运行时
2. 支持同时抓取多个数据源
3. 不要把所有任务逻辑塞进 `main`

提示：

1. 先把“抓一个源”设计成清晰函数
2. 再考虑“抓多个源”的编排

## 7. 第三阶段：补工程化错误边界

要求：

1. 配置错误、网络错误、解析错误、业务错误要有分层
2. 入口层要能输出最终可读错误
3. 内部层要保留尽量清楚的上下文

提示：

1. 入口层和领域层错误不一定要长一样
2. 想一想什么时候适合转换错误

## 8. 第四阶段：补日志与可观测性

要求：

1. 记录每个数据源抓取开始和结束
2. 记录失败数据源
3. 记录总耗时
4. 给至少一个关键路径设计 tracing 字段

提示：

1. 不要只打一句“failed”
2. 先定义你最关心的关键事件

## 9. 第五阶段：补性能思考

要求：

1. 说出你怀疑的 2 个热点
2. 解释它们更像 I/O、解析还是同步等待问题
3. 说明如果要继续优化，你会先测哪里

提示：

1. async 项目慢，不一定是 CPU 慢
2. 很多时候是等待、调度或外部依赖慢

## 10. 第六阶段：补恢复与降级策略

要求：

1. 某个数据源失败时，其余数据源仍然继续
2. 输出里要明确哪些源失败
3. 允许最终报告部分成功

提示：

1. 不要默认“一处失败，整体全部失败”
2. 这会训练你思考工程里的韧性边界

## 11. 这组挑战最该练的判断

做完之后，最值得复盘的不是“我用了多少 crate”，而是：

1. 你有没有把异步流程拆清楚
2. 你有没有把错误边界分层
3. 你有没有区分局部失败和整体失败
4. 你有没有把日志做成可观察信号
5. 你有没有凭感觉谈性能

## 12. 常见错误与修正方向

### 错误 1：一上来就把并发、async、日志全混进一个文件

先拆层，再加能力。

### 错误 2：遇到网络错误就直接 `unwrap`

这会让工程题失去意义。

### 错误 3：并发了就以为一定更快

先分清瓶颈类型。

### 错误 4：日志只有一句成功或失败

真实排障需要上下文和字段。

### 错误 5：所有失败都等价处理

工程里经常要区分“可降级失败”和“致命失败”。

## 13. 小练习

1. 在不写代码的前提下，先画出模块结构图。
2. 为“单个源失败但整体继续”的场景设计返回类型。
3. 列出你要打的 5 条关键日志或 tracing 事件。

## 14. 挑战题

把这道题继续升级成一版“小型工程挑战”，要求：

1. 支持配置文件
2. 支持并发抓取
3. 支持部分失败
4. 支持文本与 Markdown 两种输出
5. 支持至少一条性能分析思路和一条观测设计

额外要求：

1. 至少解释一次：为什么这里的复杂度主要不在语法，而在边界设计
2. 至少解释一次：如果以后继续扩展，你最怕哪一层先变成耦合泥团

## 15. 小结

这一组挑战题最重要的是逼你进入真实工程视角：

1. async 不是目的，边界清晰才是目的
2. 并发不是越多越好
3. 错误边界、日志、输出、恢复策略都属于工程设计
4. 性能和观测必须靠证据，不靠感觉
