# FFI 与跨语言交互

## 1. 这一章是干什么的

到了精通阶段，Rust 已经不只是“我自己写一个程序”这么简单了。

你会越来越多地遇到这些问题：

1. 我要调用已有的 C 库怎么办
2. 我要把 Rust 写成库给别的语言用怎么办
3. 数据结构跨语言后，内存由谁管理
4. 字符串、错误、回调、线程边界怎么约定

这时你就进入了：

- FFI，Foreign Function Interface

这一章的目标不是把所有跨语言细节一次讲完，而是先建立一套非常实用的边界直觉：

- FFI 到底在解决什么问题
- `extern "C"`、`#[repr(C)]` 分别在做什么
- 为什么“谁分配谁释放”是第一原则
- 为什么跨语言边界常常和 `unsafe` 强绑定

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 所有权、借用、生命周期
- `unsafe` 的边界与原则
- 结构体和枚举的基本内存直觉
- 模块与公共 API 设计

最好已经看过：

- [unsafe 的边界与原则](./04-unsafe的边界与原则.md)
- [生命周期进阶](./01-生命周期进阶.md)

## 3. FFI 的本质到底是什么

先别把 FFI 想得太神秘。

它本质上是在解决：

- 不同语言、不同编译器模型、不同运行时之间，如何安全而稳定地交换函数调用和数据

Rust 自己的类型系统很强，但一旦跨语言边界，很多保证都不能自动继续成立。

例如：

1. 对方可能不理解 Rust 的所有权模型
2. 对方可能不认识 Rust 默认的数据布局
3. 对方可能不会遵守 Rust 的错误传播方式
4. 对方甚至没有和 Rust 一样的字符串表示

所以 FFI 的第一反应不该是：

- “怎么尽快调通”

而应该是：

- “边界契约是什么”

## 4. 为什么 FFI 会和 unsafe 紧密绑在一起

因为跨语言边界时，编译器已经无法替你完整验证：

1. 对方传进来的指针是否有效
2. 对方是否遵守长度和编码约定
3. 这块内存应该由谁释放
4. 回调时线程和生命周期是否还成立

所以 FFI 几乎天然属于：

- 需要你自己证明前提的领域

这也是为什么你会经常看到：

- `extern "C"`
- 裸指针
- `unsafe`
- `#[repr(C)]`

一起出现。

## 5. 先理解 `extern "C"`：它主要在约定 ABI

看一个最小例子：

```rust
unsafe extern "C" {
    fn abs(input: i32) -> i32;
}

fn main() {
    let value = unsafe { abs(-7) };
    println!("{value}");
}
```

这里最关键的不是函数体，而是：

- `"C"`

它在说明：

- 这个函数调用要使用 C ABI 约定

你可以先把 ABI 粗略理解成：

- 参数怎么传
- 返回值怎么传
- 调用约定怎么约定

如果 ABI 不一致，即使函数名一样，也可能直接出问题。

## 6. `#[repr(C)]` 为什么重要

Rust 默认不会承诺所有自定义类型都有和 C 一样稳定的内存布局。

所以当一个结构体要跨 FFI 边界传递时，通常需要显式写：

```rust
#[repr(C)]
pub struct StudyRecord {
    pub score: u32,
    pub hours: f32,
}
```

`#[repr(C)]` 的核心意义不是“更底层更高级”，而是：

- 给跨语言边界一个更稳定、可预期的数据布局约定

如果你忘了这一步，对方语言看到的字段布局可能和你想的不一样。

## 7. 先记住一个总原则：边界上尽量用简单类型

FFI 边界最稳的做法通常不是直接暴露复杂 Rust 类型，而是尽量使用：

1. 整数
2. 浮点数
3. 指针
4. 长度
5. `#[repr(C)]` 结构体

而不是直接把这些东西扔给对方：

1. `String`
2. `Vec<T>`
3. `HashMap<K, V>`
4. trait object
5. 带复杂生命周期关系的借用结构

因为这些高级 Rust 类型往往隐含了：

- 布局约定
- 释放逻辑
- 所有权语义

跨语言方通常并不知道这些规则。

## 8. “谁分配谁释放”为什么是第一原则

这是 FFI 里最值钱的一条原则之一：

- 谁负责分配内存，通常就应该由谁负责释放

如果这条规则不清楚，就非常容易出现：

1. double free
2. memory leak
3. 用错 allocator
4. 悬垂指针

例如：

- Rust 返回了一个字符串指针
- C 端却用自己的 `free` 去释放

这就可能直接出问题。

所以 FFI 设计时一定要明确：

1. 谁创建
2. 谁持有
3. 谁释放
4. 释放函数叫什么

## 9. 字符串边界是第一批高频坑

Rust 的 `String` 和 `&str` 很方便，但跨 FFI 后通常不能直接这么传。

在和 C 风格接口交互时，你经常会看到：

- `CString`
- `CStr`

一个非常常见的方向是：

1. Rust 要传字符串给 C 时，先变成 `CString`
2. Rust 从 C 接收字符串时，用 `CStr` 解释

例如：

```rust
use std::ffi::{CStr, CString};
use std::os::raw::c_char;

fn main() {
    let message = CString::new("hello ffi").unwrap();
    let ptr: *const c_char = message.as_ptr();

    let restored = unsafe { CStr::from_ptr(ptr) };
    println!("{}", restored.to_str().unwrap());
}
```

这里你要建立一个关键意识：

- 指针只是边界表示
- 真正难的是所有权和有效期

## 10. 为什么不能随便把 `String` 的内部指针交出去就完事

因为你要立刻问：

1. 这块内存在 Rust 侧还能活多久
2. 对方会不会缓存这个指针
3. Rust 侧会不会提前释放或移动它

也就是说，FFI 里很多问题本质上还是：

- 生命周期和所有权问题，只是换了语言边界表达

## 11. 一个“Rust 暴露给外部调用”的最小例子

例如：

```rust
#[unsafe(no_mangle)]
pub extern "C" fn add(a: i32, b: i32) -> i32 {
    a + b
}
```

你可以先把这类函数理解成：

- 给外部语言一个非常简单、ABI 明确的入口

这里有几个点要注意：

1. `extern "C"` 指定 ABI
2. 导出函数应该尽量简单稳定
3. 参数和返回值尽量先用基础类型

## 12. 暴露结构体时为什么更要保守

如果你要跨语言暴露一个结构体，第一反应应该是：

- 我能不能先暴露“句柄 + 函数”模式，而不是整个复杂对象

例如很多库不会直接把完整内部对象暴露给 C，而是：

1. 返回一个不透明指针
2. 再提供 `create`
3. `use`
4. `destroy`

这种设计的价值在于：

- 内部实现可以继续保留 Rust 自己的抽象
- 对外边界更简单

## 13. 一个更像工程代码的句柄模式

例如，你可以把复杂对象放在 Rust 内部，只对外暴露指针句柄：

```rust
#[repr(C)]
pub struct StudySession {
    total_hours: f32,
}

#[unsafe(no_mangle)]
pub extern "C" fn session_new(total_hours: f32) -> *mut StudySession {
    Box::into_raw(Box::new(StudySession { total_hours }))
}

#[unsafe(no_mangle)]
pub unsafe extern "C" fn session_free(ptr: *mut StudySession) {
    if ptr.is_null() {
        return;
    }

    // SAFETY: ptr 必须来自 session_new，且只释放一次。
    unsafe {
        drop(Box::from_raw(ptr));
    }
}
```

这里最关键的不是记住 `Box::into_raw` 和 `Box::from_raw`，而是理解：

1. 创建和释放函数必须成对设计
2. 调用方必须知道释放规则
3. 这就是“谁分配谁释放”的具体落地

## 14. 错误处理为什么不能直接把 Rust 异常思维搬过来

Rust 没有 Java、Python 那种异常边界可以直接跨语言传播。

FFI 边界上更稳的做法通常是：

1. 返回状态码
2. 返回空指针或布尔值表示失败
3. 把详细错误放到额外输出参数或日志里

也就是说，跨语言边界时你经常需要主动把：

- Rust 风格的错误模型

转换成：

- 对方语言更容易稳定接住的边界模型

## 15. 回调为什么容易变复杂

一旦 FFI 进入回调，你就要同时考虑：

1. 函数指针 ABI
2. 上下文指针生命周期
3. 回调发生时线程是否还正确
4. Rust 侧对象是否还活着

很多 FFI 回调的真正难点，不是“语法怎么传函数指针”，而是：

- 这次回调发生时，那份上下文是否仍然有效

所以回调一旦涉及长期持有、异步、跨线程，就要非常谨慎。

## 16. 为什么 FFI API 往往故意写得“丑一点”

很多初学者会觉得：

- 为什么 FFI API 这么不优雅

原因常常不是作者不会设计，而是：

- 跨语言稳定性比 Rust 内部优雅抽象更重要

所以你会经常看到：

1. 明确的 create/destroy
2. 明确的长度参数
3. 明确的错误码
4. 很少直接暴露复杂泛型
5. 很少直接暴露 trait

这不是退步，而是边界收缩。

## 17. 和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

如果你写过 Node.js native addon，或者见过 WebAssembly / N-API 这种边界，你会感觉到：

- JS 世界的动态对象模型，到了原生边界就必须重新收紧成稳定接口

Rust FFI 也一样。

### 和 Python 类比

Python 的 `ctypes`、C 扩展、`cffi` 也会让你面对：

- 指针
- 内存释放
- 字符串编码

这些问题。

Rust 的区别在于：

- 它会更强烈地要求你把边界契约写清楚

### 和 Java 类比

Java 的 JNI 会让你立刻感受到：

- 一旦跨出 JVM 的舒适区，很多安全感都不再自动存在

Rust FFI 也很像这样，只不过 Rust 内部本来就更显式，所以它会更早要求你面对这些边界。

## 18. FFI 设计的一套实用检查表

你以后设计 FFI 边界时，可以先按这个顺序检查：

1. ABI 是否明确
2. 数据布局是否明确
3. 字符串编码是否明确
4. 谁分配谁释放是否明确
5. 错误返回方式是否明确
6. 回调和线程边界是否明确
7. 调用方最容易误用的点是否已被文档写清楚

如果这里有一条答不清楚，接口就还不够稳。

## 19. 常见错误与修正方向

### 错误 1：直接把复杂 Rust 类型暴露给外部

跨语言边界应该优先简单、稳定。

### 错误 2：忘记 `#[repr(C)]`

一旦跨结构体布局边界，这很容易变成隐患。

### 错误 3：释放规则不清楚

这会直接引出泄漏或 double free。

### 错误 4：把临时字符串指针交给对方长期保存

这通常会变成悬垂指针问题。

### 错误 5：回调只看语法，不看生命周期和线程

真实复杂度往往都在这里。

## 20. 小练习

1. 写一个 `extern "C"` 的最小 `add` 函数。
2. 写一个 `#[repr(C)]` 结构体，并解释为什么这里需要它。
3. 用自己的话解释：为什么 `String` 不适合直接作为通用 FFI 边界类型。
4. 解释“谁分配谁释放”为什么比“我已经调通了”更重要。

## 21. 挑战题

设计一个“学习会话” FFI 边界，要求：

1. Rust 暴露 `create`
2. 暴露读取总时长函数
3. 暴露销毁函数
4. 至少有一个字符串输入或输出边界
5. 写清楚释放规则

额外要求：

1. 至少解释一次：为什么这里不直接把内部复杂结构体完全暴露出去
2. 至少解释一次：如果以后要让 Python 或 Java 调这个接口，哪一条约定最容易被误用

## 22. 小结

这一章最重要的是把 FFI 看成“边界契约设计”，而不是“把函数连起来就行”：

1. `extern "C"` 主要是在约定 ABI
2. `#[repr(C)]` 主要是在约定布局
3. FFI 边界应该尽量简单、稳定、可文档化
4. “谁分配谁释放”是第一原则
5. 字符串、回调、错误和线程边界是高频难点
