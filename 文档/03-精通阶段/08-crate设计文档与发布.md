# crate 设计、文档与发布

## 1. 这一章是干什么的

到了精通阶段，你要开始面对一个和前面很不一样的问题：

- 代码不再只是你自己用

一旦一个 Rust 项目要作为 crate 提供给别人使用，关注点就会明显变化：

1. 公共 API 是否稳定
2. 模块边界是否清楚
3. 错误类型是否易理解
4. 文档是否足够让别人快速上手
5. 发布和版本策略是否可靠

这一章的目标就是把“写项目”推进到“做可复用库”的层次。

建议你已经理解：

- trait、泛型、生命周期的基础与进阶
- 工程化错误处理
- 性能与观测的第一轮判断
- Cargo 与模块系统

最好已经看过：

- [高级 trait 技巧](./02-高级trait技巧.md)
- [工程化错误处理与项目结构设计](../02-高级阶段/08-工程化错误处理与项目结构设计.md)
- [基准测试与可观测性](./07-基准测试与可观测性.md)

## 3. crate 设计和写一个程序最大的区别

写一个应用程序时，你可以默认：

- 入口由你控制
- 调用方式由你控制
- 使用者就是你自己

但 crate 不一样。

一旦你把代码做成库，就等于在承诺：

1. 别人会按你的 API 理解它
2. 别人会依赖你的类型和行为
3. 以后你每次改动都可能影响别人

所以 crate 设计的核心已经不是：

- 我能不能实现功能

而是：

- 我暴露出去的边界是不是长期可用

## 4. 先记住一个重要原则：公共 API 越少越稳

很多人第一次做库时，容易把大量内部实现也一起 `pub` 出去。

这会带来问题：

1. 未来重构空间变小
2. 调用方开始依赖你本不想承诺的细节
3. 文档面也会迅速膨胀

所以精通阶段非常重要的一条原则是：

- 默认收紧公开面，只暴露真正想承诺的边界

## 5. 模块设计要开始区分“公开层”和“内部层”

例如一个学习报告格式化 crate，可能会分成：

```text
src/
├─ lib.rs
├─ report.rs
├─ formatter.rs
├─ error.rs
└─ internal/
   ├─ mod.rs
   └─ parse.rs
```

这里的重点不是目录长什么样，而是：

- `lib.rs` 决定对外公开什么
- `internal` 可以保留更自由的实现调整空间

也就是说，crate 设计本质上是在做：

- 承诺管理

## 6. `lib.rs` 为什么很重要

对库来说，`lib.rs` 往往就是第一层门面。

它通常负责：

1. 重导出核心类型
2. 隐藏内部实现细节
3. 组织对外阅读顺序

例如：

```rust
pub mod error;
pub mod formatter;
pub mod report;

pub use error::ReportError;
pub use formatter::ReportFormatter;
pub use report::StudyReport;
```

这类门面设计的价值在于：

- 使用者不必先理解你的整个目录树

## 7. API 设计时要主动问的几个问题

以后你设计 crate API 时，可以反复问自己：

1. 这个类型真的要公开吗
2. 这个字段真的要公开吗
3. 这个 trait bound 会不会把调用方绑太死
4. 这里返回借用是否会让使用者更难用
5. 这里的错误类型是否足够清晰

这些问题都和“我会不会写代码”无关，而是：

- 我会不会设计边界

## 8. 文档为什么不是附属品

对 crate 来说，文档不是“以后有空再补”。

很多时候：

- 文档本身就是 API 的一部分

因为使用者首先看到的，往往不是你的源代码，而是：

1. crate 名称
2. README
3. Rustdoc
4. 最小示例

如果这里不清楚，再好的实现也很难被正确使用。

## 9. 什么样的示例最有价值

最有价值的例子通常不是：

- 覆盖所有功能的大而全 demo

而是：

- 最小但完整的上手路径

例如一个报告格式化 crate，最先应该给出的可能是：

```rust
use study_report::{ReportFormatter, StudyReport};

let report = StudyReport::new("Rust", 95);
let text = ReportFormatter::default().format(&report);
println!("{text}");
```

这类例子的目标是：

- 让用户 30 秒内看懂“入口在哪里”

## 10. Rustdoc 思维：文档和代码一起设计

Rust 的文档生态很适合把说明贴近 API：

```rust
/// Formats a study report into plain text.
pub struct ReportFormatter;
```

真正重要的不是注释语法，而是你要开始有这种意识：

1. 这个类型是干什么的
2. 调用者最容易误解什么
3. 参数和返回值的边界是什么
4. 失败条件是什么

crate 设计成熟与否，很多时候从文档就能看出来。

## 11. 错误类型是库体验的一部分

如果一个库返回：

```rust
Result<T, String>
```

调用方通常会很难受，因为：

1. 错误分类不清楚
2. 无法稳定匹配特定错误
3. 上层很难写出有边界感的处理逻辑

所以公共库更适合：

- 清晰、可枚举、可理解的错误类型

这和应用里“入口快速汇总错误”是不同思路。

## 12. trait 和泛型在 crate 设计里要更克制

很多精通阶段学习者会出现另一个误判：

- 既然我已经学会高级 trait 和泛型，那库接口越抽象越高级

这不对。

过度泛型化会带来：

1. 文档理解难度上升
2. 编译错误更复杂
3. 扩展边界反而模糊

所以 crate API 要问的不是：

- 能不能抽象

而是：

- 这份抽象对使用者有没有真实收益

## 13. feature 设计为什么重要

成熟 crate 往往不会把所有能力都硬塞成一个固定包。

它们会通过 feature 控制：

1. 可选依赖
2. 额外功能
3. 特定平台支持
4. 与某些生态的集成

这类设计的价值在于：

- 让用户按需启用能力

但 feature 也会带来复杂度，所以要避免：

- feature 组合失控

## 14. 版本语义为什么会变成责任问题

一旦 crate 发布出去，版本号就不是装饰了。

你需要开始认真面对：

1. 哪些改动是兼容的
2. 哪些改动会破坏现有调用方
3. 什么时候应该升大版本

这背后本质上是：

- 你对公共 API 稳定性的承诺

## 15. 发布前应该先想什么

真正成熟的发布动作，顺序通常不是：

1. 先发再说

而更像：

1. API 是否已收敛
2. 文档是否够用
3. 示例是否能跑
4. 错误边界是否清楚
5. 测试是否覆盖关键路径
6. feature 组合是否合理

也就是说，发布不是“上传一下”，而是：

- 对外承诺一次边界

## 16. README 和 Rustdoc 的分工

这两个都重要，但角色不同。

### README

更适合：

1. 项目简介
2. 适用场景
3. 快速开始
4. 版本和特性概览

### Rustdoc

更适合：

1. 具体类型
2. 方法行为
3. 参数与返回值
4. 错误与示例

如果你把这两者分工搞清楚，库的可读性会立刻提升很多。

## 17. crate 设计常见的“句柄化”思路

很多库的对外 API 都不会把内部所有细节完全暴露。

而是倾向于：

1. 暴露一个核心类型
2. 暴露少量构造入口
3. 暴露少量稳定行为
4. 把复杂逻辑压在内部

这和前面 FFI 的边界思路其实很像：

- 好边界通常都更克制

## 18. 发布不是终点，而是维护开始

一旦 crate 被别人用起来，你后面要考虑的就是：

1. 兼容性
2. 迁移说明
3. 废弃策略
4. 文档更新
5. issue 反馈

所以精通阶段真正该建立的心态是：

- 发布意味着维护责任开始

## 19. 和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

如果你发过 npm 包，你会很容易理解：

- 包名
- 对外 API
- README
- 版本语义

这些都很重要。

Rust crate 的不同点在于：

- 类型系统更强，所以 API 一旦暴露，约束会更硬

### 和 Python 类比

Python 包发布时你也会关心：

- 文档
- 依赖
- 版本
- 公开接口

Rust crate 的区别是：

- 借用、trait、feature、错误类型这些设计会更直接影响使用体验

### 和 Java 类比

Java 背景会让你对：

- 公共 API 稳定性
- 版本兼容
- 文档和示例

这些责任不陌生。

Rust 的特别之处在于：

- 类型签名本身就承载了更多契约信息

## 20. 常见错误与修正方向

### 错误 1：把太多内部实现直接 `pub`

公开面应该克制。

### 错误 2：只写代码，不写最小示例

很多用户根本不知道入口怎么用。

### 错误 3：错误类型太随意

公共库更需要清晰边界。

### 错误 4：过度泛型化

抽象不是越多越好。

### 错误 5：版本号随便改

这会直接破坏使用者预期。

## 21. 小练习

1. 选一个你写过的小项目，列出其中哪些类型其实不应该公开。
2. 给一个公共函数补一段最小文档注释和用例。
3. 解释 README 和 Rustdoc 的分工差异。
4. 用自己的话说明：为什么“公共 API 越少越稳”。

## 22. 挑战题

设计一个“学习报告格式化 crate”，要求：

1. 列出对外公开的核心类型
2. 列出不想公开的内部实现
3. 设计一个最小 README 目录
4. 设计一段 Rustdoc 示例
5. 设计一个错误类型边界

额外要求：

1. 至少解释一次：为什么某个类型不应该直接 `pub`
2. 至少解释一次：如果以后加功能，你最怕破坏哪条 API 契约

## 23. 小结

这一章最重要的是把 crate 看成“对外承诺”而不是“打包一下代码”：

1. 公共 API 越少越稳
2. `lib.rs` 是重要门面
3. 文档和示例本身就是库体验的一部分
4. 错误类型、feature 和版本策略都会影响长期可用性
5. 发布不是结束，而是维护责任开始
