# unsafe 的边界与原则

## 1. 这一章是干什么的

很多人第一次看到 `unsafe`，会立刻产生两种极端反应：

1. `unsafe` 很危险，所以我永远不要碰
2. `unsafe` 很底层，所以它一定更快、更高级

这两种理解都不够准确。

`unsafe` 真正意味着的是：

- 你正在进入编译器不能完全替你证明安全性的边界

也就是说，Rust 并不是在 `unsafe` 里“关闭一切规则”，而是在说：

- 有些地方我不再替你做完整担保，你自己必须承担证明责任

这一章的目标是帮你建立一套精通阶段必须有的判断：

- `unsafe` 到底是什么
- 它具体允许你做哪些事
- 为什么“能写 unsafe”不等于“应该写 unsafe”
- 如何把 unsafe 封装进更小、更稳的安全抽象

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 所有权、借用、生命周期
- 指针与引用的基本区别
- 模块与 API 边界
- 宏系统和 trait 抽象的第一轮直觉

最好已经看过：

- [高级 trait 技巧](./02-高级trait技巧.md)

## 3. 先打掉一个误解：unsafe 不是“性能模式”

很多初学者会误以为：

- safe Rust 比较慢
- `unsafe` 会让程序自动变快

这不对。

`unsafe` 本身不等于优化，它只是允许你做一些：

- 编译器无法自动验证为安全的操作

有些高性能实现会用 `unsafe`，是因为：

- 它们为了减少边界检查、表达底层内存布局、对接外部 ABI，必须进入这个边界

但“用了 unsafe”本身并不是性能证据。

精通阶段要建立的直觉是：

- 性能靠测量证明
- `unsafe` 靠不变量证明

## 4. unsafe 具体允许你做哪些事

Rust 官方常见的五类 `unsafe` 能力，你先记住这份清单：

1. 解引用裸指针
2. 调用 `unsafe fn` 或 `unsafe` 方法
3. 访问或修改 `static mut`
4. 实现 `unsafe trait`
5. 访问 `union` 字段

你可以把它们粗略理解成：

- 这些地方一旦做错，编译器很难继续替你兜底

## 5. safe Rust 和 unsafe Rust 的边界到底差在哪

safe Rust 的核心承诺是：

- 只要代码能通过安全规则，编译器会尽量阻止一大类内存安全错误

而 `unsafe` 的意思更像：

- “下面这些操作，我默认信你已经自己验证过了”

这就像你在签一份内部契约：

- 我知道这里有哪些前提条件
- 我保证调用时满足这些前提

## 6. `unsafe fn` 和 `unsafe {}` 不是一回事

这是非常容易混淆的一点。

### `unsafe fn`

它更像在声明：

- 调用这个函数需要满足额外前提

### `unsafe { ... }`

它更像在声明：

- 我现在正在执行一段需要我自己负责证明安全性的代码

一个很实用的理解是：

- `unsafe fn` 是“把前提写进接口”
- `unsafe block` 是“在具体实现里承担责任”

## 7. 为什么 unsafe 其实很依赖文档和注释

因为一旦进入 `unsafe`，你就不能只靠类型系统让别人自动理解前提了。

所以成熟 Rust 代码里，`unsafe` 周围通常会有：

1. Safety 注释
2. 不变量说明
3. 前置条件说明
4. 封装边界说明

例如：

```rust
pub fn first_byte(slice: &[u8]) -> Option<u8> {
    if slice.is_empty() {
        return None;
    }

    // SAFETY: 上面已经检查过 slice 非空，因此索引 0 一定有效。
    let value = unsafe { *slice.get_unchecked(0) };
    Some(value)
}
```

这里最重要的不是语法，而是：

- 你必须清楚写出为什么这段 unsafe 成立

## 8. 一个常见例子：为什么裸指针比引用危险

看一个最小例子：

```rust
let x = 42;
let ptr = &x as *const i32;

unsafe {
    println!("{}", *ptr);
}
```

`*const i32` 是裸指针。

和 `&i32` 不同，裸指针本身不自动保证：

1. 一定非空
2. 一定指向有效内存
3. 一定满足别名规则
4. 一定满足生命周期要求

所以一旦解引用裸指针，责任就转到你身上了。

## 9. undefined behavior 为什么是核心问题

学习 `unsafe` 时，真正应该怕的不是：

- 程序一定立刻崩

而是：

- 程序可能在某个条件下产生未定义行为

未定义行为的可怕之处在于：

1. 它可能今天看起来能跑
2. 明天换个编译器优化级别就出问题
3. 问题位置和真实原因可能离得很远

所以 `unsafe` 不是“我试了一次没崩就说明没问题”。

## 10. 一个成熟思路：把 unsafe 缩到最小

精通阶段写 `unsafe`，最重要的工程原则之一是：

- 把不安全区域压缩到最小

这通常意味着：

1. 先在外层做完整校验
2. 只把那一小段必须不安全的操作放进 `unsafe`
3. 立刻再回到安全 API

还是上面的例子：

```rust
pub fn first_byte(slice: &[u8]) -> Option<u8> {
    if slice.is_empty() {
        return None;
    }

    // SAFETY: slice 非空，因此索引 0 有效。
    Some(unsafe { *slice.get_unchecked(0) })
}
```

这种写法比“整段函数都包在 unsafe 里”成熟得多。

## 11. 为什么优秀 Rust 库常常“内部有 unsafe，对外仍是 safe”

这是 Rust 工程里很重要的一种设计风格。

例如很多标准库或成熟 crate 会：

1. 在内部用 `unsafe` 实现高性能或底层能力
2. 对外暴露安全接口
3. 用清晰不变量保护外部调用者

你可以把它理解成：

- 把危险集中封装，换来更好的整体可用性

这和初学者常见的“哪里报错就哪里加 unsafe”完全不是一回事。

## 12. `unsafe trait` 为什么也很重要

有些 trait 表示的不是普通行为，而是：

- 一种必须由实现者保证的底层安全契约

例如标准库里的某些底层 trait，就要求：

- 你一旦实现，就要保证某些内存或线程语义前提成立

这说明 `unsafe` 不只出现在代码块里，也会出现在：

- 类型系统契约层

## 13. `unsafe` 和生命周期、别名规则关系很深

很多 `unsafe` 的真正难点，不在于语法，而在于你是否真的理解：

1. 这块内存当前是否仍然有效
2. 这里是否违反了独占可变访问规则
3. 这个指针是否越过了原本借用边界

所以如果你前面的：

- 所有权
- 借用
- 生命周期

理解不稳，那么 `unsafe` 很容易变成：

- 只是暂时把编译器静音

这是最危险的用法。

## 14. 一个很常见的误区：把 unsafe 当成“跳过借用检查”的捷径

例如你可能会想：

- 这段借用过不去，我先转成裸指针再说

这通常是坏信号。

因为它往往意味着：

- 你没有解决设计问题
- 只是绕开了编译器提醒

精通阶段更成熟的反应应该是：

1. 先重看数据流
2. 先问是否应该改所有权边界
3. 只有在确实需要底层表达能力时，才考虑 unsafe

## 15. 一个比“能不能写”更重要的问题：前提是否可证明

写 `unsafe` 前，你应该先问：

1. 这段代码依赖哪些前提
2. 这些前提能不能在当前 API 边界被稳定保证
3. 调用者是否容易误用
4. 将来别人改代码时，这些前提会不会被悄悄破坏

如果这些问题答不清楚，那通常不是“注释补一补”就能解决。

## 16. 一个更真实的封装思路例子

假设你已经检查了一段字节流确实是 UTF-8：

```rust
pub fn bytes_to_str(bytes: &[u8]) -> Option<&str> {
    if std::str::from_utf8(bytes).is_err() {
        return None;
    }

    // SAFETY: 上面已经验证 bytes 是合法 UTF-8。
    Some(unsafe { std::str::from_utf8_unchecked(bytes) })
}
```

这里的重点不是“我要炫 unsafe”，而是：

1. 前置校验先发生
2. `unsafe` 只保留最小片段
3. 对外仍暴露安全返回值

当然，这个例子在真实工程里还要继续问：

- 这里到底有没有必要用 unchecked 版本

如果没有明确性能证据，直接用安全版本通常更稳。

## 17. 和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

JavaScript 日常开发几乎没有裸指针概念。

如果一定要类比，`unsafe` 更像：

- 你从普通 JS 代码跨到 Node.js native addon 或底层宿主接口

一旦跨过去，语言本身对你的保护就少很多。

### 和 Python 类比

Python 里大多数代码都在解释器安全边界内跑。

但当你进入：

- C 扩展
- `ctypes`
- 底层二进制内存操作

你也开始承担更多自己证明正确性的责任。

### 和 Java 类比

Java 平时很少直接碰内存边界。

但当你进入：

- JNI
- off-heap memory
- 更底层的运行时接口

你同样会明显感觉到：

- 类型系统和运行时对你的兜底减少了

Rust 的 `unsafe` 和这些场景有相似点，但更显式，也更常被封装进库内部。

## 18. 写 unsafe 时的一套实用原则

你可以先记下面这套顺序：

1. 没有必要就不用
2. 能用安全抽象就不用不安全抽象
3. 必须用时，把范围压到最小
4. 写清楚 Safety 注释和不变量
5. 优先把 unsafe 封装到模块内部
6. 给封装边界补测试
7. 不用“看起来没崩”当证据

这套原则比记一堆危险故事更有用。

## 19. 常见错误与修正方向

### 错误 1：把 unsafe 当性能优化按钮

性能要靠测量证明，unsafe 只是在扩大能力边界。

### 错误 2：把整段函数都包进 unsafe

应该尽量缩小 unsafe 范围。

### 错误 3：没有 Safety 注释

以后你自己回来都可能看不懂前提。

### 错误 4：用裸指针绕开借用检查

这通常是在掩盖设计问题。

### 错误 5：对外暴露脆弱前提，却没有封装

成熟设计通常是“内部不安全，对外安全”。

## 20. 小练习

1. 写一个安全函数，在检查非空后用 `get_unchecked` 读取第一个元素。
2. 用自己的话解释：`unsafe fn` 和 `unsafe {}` 的区别。
3. 解释为什么“程序没崩”不能证明 unsafe 正确。
4. 找一个标准库或成熟 crate 的安全 API，思考它内部为什么可能需要 unsafe。

## 21. 挑战题

做一个“安全封装练习”：

1. 设计一个小型字节缓冲区工具
2. 在内部允许自己使用极少量 unsafe
3. 对外只暴露 safe API
4. 为每一段 unsafe 写出 Safety 注释
5. 解释你的不变量是什么

额外要求：

1. 至少解释一次：为什么这里 unsafe 范围已经被压到最小
2. 至少解释一次：如果别人以后维护这段代码，最容易破坏哪条前提

## 22. 小结

这一章最重要的不是让你马上大量写 unsafe，而是建立边界意识：

1. `unsafe` 不是性能模式，而是证明责任转移
2. 真正关键的是前提、不变量和封装
3. 很多优秀 Rust 库是“内部不安全，对外安全”
4. 缩小 unsafe 范围，比“我会写 unsafe 语法”重要得多
5. 不能把 unsafe 当成绕过设计问题的捷径
