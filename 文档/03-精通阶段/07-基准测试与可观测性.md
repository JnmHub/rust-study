# 基准测试与可观测性

## 1. 这一章是干什么的

上一章解决的是：

- 你为什么不能凭感觉优化

这一章要继续往前一步，解决的是：

1. 你怎么稳定地测一段代码
2. 你怎么把“快一点”变成可比较的结果
3. 你怎么在运行时看到程序真实行为

这就是：

- 基准测试
- 可观测性

它们经常一起出现，因为一个偏：

- 精细测量局部实现

另一个偏：

- 观察真实运行中的整体行为

精通阶段如果缺这两块，很容易出现两种问题：

1. 你写了很多“优化”，但没有稳定对比基线
2. 你只盯局部 benchmark，却看不到线上或长流程里的真实延迟分布

## 3. benchmark 和普通测试不是一回事

普通测试主要回答的是：

- 结果对不对

benchmark 主要回答的是：

- 在可比较条件下，这段代码表现怎么样

所以你不能把下面两件事混在一起：

1. 代码能跑通
2. 代码性能有改善

前者靠测试保证，后者靠基准和测量保证。

## 4. 什么叫“可重复基线”

基准测试最核心的价值，不是跑出一个绝对数字，而是：

- 让你在相对稳定条件下比较实现 A 和实现 B

这也是为什么 benchmark 很强调：

1. 固定输入
2. 固定测量方式
3. 尽量减少环境波动

如果这些前提都不稳，你看到的差异可能只是：

- 机器状态不同
- 测量方式不同
- 输入规模不同

而不是代码真有变化。

## 5. Rust 里 benchmark 常在看什么

基准测试常见关注点包括：

1. 某个函数平均耗时
2. 不同实现的吞吐差异
3. 输入规模变化后的趋势
4. 分配次数或延迟分布

例如：

- 文本解析器 A 和 B 哪个更快
- 不同去重策略在 1 万条、10 万条数据下差异如何
- 是否减少了不必要分配

## 6. 一个最小 benchmark 心智模型

假设你要比较两种标题归一化写法。

第一版：

```rust
fn normalize_a(input: &str) -> String {
    input.trim().to_lowercase()
}
```

第二版：

```rust
fn normalize_b(input: &str) -> String {
    let trimmed = input.trim();
    trimmed.chars().flat_map(|c| c.to_lowercase()).collect()
}
```

哪种更快，不能靠看代码猜完就算。

你应该进入的思路是：

1. 准备固定输入
2. 重复执行
3. 比较稳定结果
4. 判断收益是否值得保留

## 7. 为什么 benchmark 容易写假

基准测试最容易踩的坑之一是：

- 你以为自己在测业务代码，实际上在测别的东西

例如：

1. 输入准备成本混进了测量
2. 输出被优化器消掉
3. 每轮分配环境不一致
4. 文件 I/O 混进了你本来想测的纯函数

所以精通阶段一定要知道：

- benchmark 也需要设计

## 8. 可观测性到底是什么

你可以先把可观测性理解成：

- 当程序运行时，你能不能从外部或内部信号推断出它现在发生了什么

最常见的信号通常是：

1. 日志
2. 指标
3. tracing span
4. 错误上下文

也就是说，可观测性关心的不是：

- 代码看起来清不清楚

而是：

- 程序跑起来后，问题发生时你能不能定位

## 9. 日志不等于可观测性全部

很多人第一次接触可观测性，会把它直接等同于：

- 多打一点日志

日志当然重要，但远远不够。

因为你还会关心：

1. 这次请求用了多久
2. 哪个步骤最慢
3. 失败比例是否上升
4. 某个异步任务是否卡住
5. 哪条路径触发频率异常

这些信息往往更适合通过：

- 指标
- tracing
- 结构化事件

来观察。

## 10. 结构化日志为什么比随手 `println!` 更成熟

例如：

```rust
println!("fetch report failed");
```

这在小练习里可以，但在真实工程里价值有限，因为你缺少：

1. 哪个数据源失败
2. 花了多久
3. 当前请求上下文是什么
4. 失败次数是不是在升高

更成熟的日志思路通常是：

- 事件名
- 关键字段
- 上下文

这也是为什么 Rust 生态里很多项目会用：

- `tracing`

而不是只靠散落的 `println!`

## 11. tracing 在解决什么问题

你可以先把 tracing 理解成：

- 比普通日志更适合描述跨函数、跨异步边界的执行上下文

例如一个异步学习报告服务里，你可能关心：

1. 哪个数据源开始抓取
2. 花了多久
3. 哪个环节失败
4. 失败是否和某个配置有关

如果只是零散打印，很容易丢上下文。

而 tracing 更强调：

- span
- event
- 字段化上下文

## 12. benchmark 和 observability 为什么要一起学

因为它们分别解决两类很容易混淆的问题：

### benchmark

更适合回答：

- 局部实现 A 和 B 谁更快

### observability

更适合回答：

- 系统真实运行时，慢和错是在哪一段路径上发生的

如果你只会 benchmark，不会可观测性，就可能出现：

- 局部函数优化了，但系统整体仍然慢

如果你只会打日志，不会 benchmark，就可能出现：

- 你知道慢，但不知道某个实现是否真的更优

## 13. 一个实际味道很强的例子

假设你有一个异步报告生成流程：

1. 读取配置
2. 并发抓取多个数据源
3. 解析 JSON
4. 汇总统计
5. 输出报告

这里可以这样分工：

1. benchmark 单独测“解析 JSON + 汇总”这类局部热点
2. tracing 观察整体一次任务里各阶段耗时
3. 指标记录成功率、失败率、平均延迟

这就是精通阶段更成熟的性能工作方式：

- 局部和整体分开观测

## 14. benchmark 的结果不是越小越神圣

一个非常现实的问题是：

- 实现 A 比实现 B 快 3%

这时真正该问的是：

1. 这 3% 是否稳定
2. 这 3% 是否在真实热点路径上
3. 它换来的代码复杂度值不值得

精通阶段最重要的不是看到任何优势都保留，而是：

- 用收益和复杂度一起做判断

## 15. 可观测性设计应该尽早进入边界

很多项目的问题不是：

- 后面补不上日志

而是：

- 一开始根本没有把“如何观察系统”当成设计内容

例如你在模块边界上就应该开始想：

1. 哪些事件值得记录
2. 哪些耗时需要打点
3. 哪些错误需要保留上下文
4. 哪些关键路径需要 trace id 或任务 id

这会直接影响后续排障效率。

## 16. 一个很实用的设计习惯：先定义关键事件

对一个中小型 Rust 项目，你可以先定义几类关键事件：

1. 入口开始和结束
2. 外部依赖调用开始和结束
3. 核心统计步骤开始和结束
4. 错误分支
5. 异常慢路径

这样做的价值是：

- 你不再是想到哪打到哪

而是在有意识地建立程序运行画像。

## 17. 和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

JavaScript 项目里你可能会做：

- 性能打点
- request tracing
- 日志聚合

Rust 里并没有本质不同，只是 Rust 更常把这些边界和：

- 异步任务
- 所有权
- 类型化错误

一起设计。

### 和 Python 类比

Python 工程里你也会分：

- 单元测试
- profile
- logging
- metrics

Rust 这里同样需要这套分层，只不过你会更频繁地关心：

- 分配
- 零拷贝边界
- 并发和锁等待

### 和 Java 类比

Java 背景会让你对：

- benchmark
- metrics
- tracing
- application diagnostics

这些概念更容易上手。

Rust 的特点在于：

- 你通常能更贴近底层成本，但也更需要自己主动把观察点设计好

## 18. 常见错误与修正方向

### 错误 1：把 benchmark 当成普通测试

它们回答的问题不同。

### 错误 2：只跑一次就下结论

性能测量需要稳定基线。

### 错误 3：只会 `println!`

真实工程更需要结构化信号。

### 错误 4：局部函数优化了，就以为系统整体变快了

整体路径仍然需要 tracing 和指标来观察。

### 错误 5：出了问题才想起可观测性

更成熟的做法是提前设计关键事件和耗时边界。

## 19. 小练习

1. 选一个你写过的纯函数，说明你会如何为它设计 benchmark。
2. 选一个 CLI 或异步项目，列出其中 5 个最值得记录的关键事件。
3. 用自己的话解释：为什么日志不是可观测性的全部。
4. 比较 benchmark 和 profiling 的关注重点。

## 20. 挑战题

给“异步学习报告服务”做一版观测设计，要求：

1. 列出至少 5 个 tracing 事件或 span
2. 列出至少 3 个指标
3. 选一个核心函数，设计 benchmark 思路
4. 说明你会怎样划分“局部测量”和“整体观测”
5. 说明你最担心的性能误判是什么

额外要求：

1. 至少解释一次：为什么结构化字段比一句模糊日志更有用
2. 至少解释一次：为什么 benchmark 结果要和真实业务路径一起看

## 21. 小结

这一章最重要的是把“测量”和“观察”区分清楚：

1. benchmark 主要解决局部实现的可比较测量
2. 可观测性主要解决真实运行时的行为推断
3. 日志、指标、tracing 是不同层次的观察手段
4. 局部优化结果不自动等于系统整体收益
5. 可观测性应该尽早进入工程设计
