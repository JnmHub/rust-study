# 宏系统入门到实战

## 1. 这一章是干什么的

到了精通阶段，宏不应该再只被理解成：

- “带感叹号的函数”

它真正重要的地方在于：

1. 宏可以在编译阶段生成代码结构
2. 宏能减少重复样板
3. 宏能把一类固定模式提炼成统一写法
4. 宏会直接影响 API 可用性和可读性

这一章的目标不是把所有宏细节一次讲完，而是先帮你建立一条实用主线：

- 宏是什么
- 宏和函数、泛型、trait 有什么本质差异
- `macro_rules!` 适合解决什么问题
- 过程宏大概在解决什么问题
- 什么时候该用宏，什么时候不该用

## 2. 学这一章前，你需要知道什么

建议你已经理解：

- 函数、泛型、trait 的基础能力
- 模块系统与多模块项目结构
- 高级 trait 技巧的第一轮直觉

最好已经看过：

- [高级 trait 技巧](./02-高级trait技巧.md)
- [工程化错误处理与项目结构设计](../02-高级阶段/08-工程化错误处理与项目结构设计.md)

## 3. 先建立一个最重要的直觉：宏不是函数

很多初学者第一次看到：

```rust
println!("hello");
vec![1, 2, 3];
```

会自然觉得：

- 这是不是只是“另一种函数调用写法”

不是。

函数处理的是：

- 已经通过语法分析和类型检查的值

宏处理的是：

- 还没有真正变成最终 Rust 代码结构前的语法片段

你可以先粗略把它理解成：

- 函数是在“运行或类型层”工作
- 宏是在“代码生成层”工作

这也是为什么宏能做到函数做不到的事，比如：

1. 接收可变数量参数
2. 生成多条语句
3. 生成结构体实现块
4. 自动派生某些样板代码

## 4. 为什么 Rust 宏在精通阶段很重要

因为你开始接触的项目，已经不只是写几行逻辑，而是会越来越多地碰到：

1. 重复样板
2. 框架式写法
3. 声明式配置
4. 派生实现
5. 编译期约束

例如：

- `serde` 用 `#[derive(Serialize, Deserialize)]`
- `clap` 用属性宏描述命令行参数
- `tokio::select!` 用宏表达并发等待分支
- `tracing::info!` 用宏收集结构化日志字段

也就是说：

- 宏不是边角料，而是 Rust 生态的重要组成部分

## 5. 先分清 Rust 里最常见的三类宏

你不需要一上来记住所有底层实现，但要先分清主要类别。

### 5.1 声明式宏：`macro_rules!`

它最适合：

- 按模式匹配输入，再展开成另一段代码

这是你最先该学会的一类。

### 5.2 派生宏：`#[derive(...)]`

它最适合：

- 给类型自动生成某类标准实现

例如：

```rust
#[derive(Debug, Clone)]
struct User {
    name: String,
}
```

### 5.3 属性宏和函数式过程宏

例如：

- `#[tokio::main]`
- `#[derive(Parser)]`
- `sqlx::query!`

这类宏更强，但理解门槛也更高。精通阶段第一轮，你先知道：

- 它们本质上也是“读一段 Rust 语法，再生成另一段 Rust 代码”

就够了。

## 6. `macro_rules!` 到底在做什么

先看一个最小例子：

```rust
macro_rules! say_hello {
    () => {
        println!("hello");
    };
}

fn main() {
    say_hello!();
}
```

这里的关键不是：

- 它打印了 hello

而是：

- `say_hello!()` 会在编译阶段被替换成对应展开结果

你可以把它先理解成：

- “看到这种输入形状，就替换成那种输出形状”

## 7. 宏最常见的能力：接收重复输入

函数当然也能接收切片、`Vec<T>`，但宏可以直接接受：

- 可变数量的源代码片段

例如：

```rust
macro_rules! sum {
    ($($value:expr),* $(,)?) => {{
        let mut total = 0;
        $(
            total += $value;
        )*
        total
    }};
}

fn main() {
    let result = sum!(10, 20, 30);
    println!("{result}");
}
```

这里几个符号很重要：

1. `$value:expr` 表示匹配一个表达式
2. `$($value:expr),*` 表示匹配逗号分隔的一组表达式
3. `$(,)?` 表示最后一个逗号可有可无

这就是声明式宏最典型的工作方式：

- 匹配模式
- 重复展开

## 8. 常见片段类型先记住几个就够了

`macro_rules!` 里经常会看到这些片段说明符：

1. `expr`：表达式
2. `ident`：标识符
3. `ty`：类型
4. `path`：路径
5. `block`：代码块
6. `tt`：token tree

你暂时不用死背全部，但要知道：

- 宏不是在处理“值”
- 它是在处理“语法片段”

## 9. 为什么宏比函数更容易让人迷路

因为函数的输入输出通常很清楚：

- 给它值
- 它返回值

宏则会带来额外复杂度：

1. 错误信息可能发生在展开后代码
2. 阅读者要先猜展开结果
3. 宏输入不是普通参数，而是语法结构
4. 一旦过度使用，代码会变得很难搜、很难跳转

所以精通阶段必须记住一个总原则：

- 宏很强，但默认不是第一选择

## 10. 什么时候应该先想函数、泛型、trait，而不是宏

如果你要解决的问题只是：

1. 复用一段逻辑
2. 抽象一类行为
3. 对不同类型做统一处理

那你应该优先考虑：

1. 函数
2. 泛型
3. trait

因为这些工具通常：

- 更容易读
- 更容易调试
- 错误信息更稳定

一个很实用的判断是：

- 如果你的需求发生在“值和类型”层面，先别急着上宏

## 11. 什么时候宏才真正值得出场

宏更适合这些场景：

1. 调用端写法需要非常接近 DSL
2. 需要减少大量重复样板
3. 需要接收不定形状输入
4. 需要在编译期生成重复结构
5. 需要配合属性、derive 形成框架式体验

例如：

- `vec![]` 比手动 `push` 更自然
- `println!` 可以接收格式化模板和可变参数
- `serde` 的 derive 宏让数据模型定义更轻

## 12. 用一个真实味道更强的声明式宏例子

假设你想快速构建学习记录：

```rust
macro_rules! study_record {
    ($subject:expr, $score:expr, $hours:expr) => {
        StudyRecord {
            subject: $subject.to_string(),
            score: $score,
            hours: $hours,
        }
    };
}

#[derive(Debug)]
struct StudyRecord {
    subject: String,
    score: u32,
    hours: f32,
}

fn main() {
    let record = study_record!("Rust", 95, 3.5);
    println!("{record:?}");
}
```

这类宏的价值在于：

- 调用端更短
- 重复样板更少

但你也要立刻追问：

- 这是不是其实用一个构造函数就够了

比如：

```rust
impl StudyRecord {
    fn new(subject: impl Into<String>, score: u32, hours: f32) -> Self {
        Self {
            subject: subject.into(),
            score,
            hours,
        }
    }
}
```

这里其实构造函数往往更稳。

这正是精通阶段判断力的重点：

- 宏能做，不代表宏值得做

## 13. 宏卫生：为什么宏里定义的名字不一定和外面冲突

Rust 宏有一个很重要的概念：

- hygiene，通常翻成“宏卫生”

它要解决的问题是：

- 宏展开后，内部名字不要随便把外部上下文污染掉

这也是为什么 Rust 宏比一些传统文本替换式宏安全很多。

你不用现在钻得太深，但要先建立一个印象：

- Rust 宏不是简单字符串拼接

## 14. `macro_rules!` 的局限也要早点知道

声明式宏虽然好用，但它不是万能的。

它比较适合：

- 基于模式的结构展开

但如果你要做更复杂的事情，比如：

1. 分析一个完整类型定义
2. 根据结构体字段自动生成实现
3. 读取属性元信息

那通常就要进入：

- 过程宏

## 15. `derive` 宏为什么在生态里特别常见

因为很多场景本质上都是：

- 对一个类型，根据它的字段结构自动生成固定模板代码

例如：

```rust
#[derive(Debug, Clone)]
struct Config {
    name: String,
}
```

`Debug`、`Clone` 这类 derive 宏让你不用重复写大量模板实现。

在生态里你还会经常见到：

- `Serialize`
- `Deserialize`
- `Parser`

这说明宏经常和：

- 类型声明
- 框架约定

一起工作。

## 16. 属性宏可以改变入口和声明语义

例如：

```rust
#[tokio::main]
async fn main() {
    println!("hello async");
}
```

这里的 `#[tokio::main]` 不只是加个标签。

它实际上会生成：

- 运行时初始化逻辑
- 同步入口到异步入口的桥接代码

这类宏常见于：

1. Web 框架路由声明
2. 测试工具
3. 异步运行时入口
4. CLI 派生配置

## 17. 宏和 JavaScript、Python、Java 怎么类比

### 和 JavaScript 类比

JavaScript 里的函数很灵活，但函数本身不会在编译前改写语法结构。

如果一定要类比，Rust 宏更像：

- Babel 插件、编译阶段转换、模板生成能力

只是 Rust 宏和语言本身集成得更深。

### 和 Python 类比

Python 的 decorator、metaclass 也会改变类或函数行为。

但大多数 Python 元编程更偏：

- 运行时改对象

Rust 宏则更偏：

- 编译期生成代码

### 和 Java 类比

Java 里你可能见过：

- 注解
- Lombok
- annotation processor

Rust 的 derive 和属性宏在“自动生成样板”这件事上，和这些工具有一点像。

但 Rust 宏通常更统一，也更贴近语言层。

## 18. 宏最常见的调试思路

宏难的地方之一，在于你看到的是：

- 调用点

而编译器真正处理的是：

- 展开后的代码

所以调试宏时，你要养成几个习惯：

1. 先写最小调用样例
2. 先猜展开结果
3. 错误出现时，先判断是“匹配失败”还是“展开后代码有问题”
4. 如果项目开始大量依赖宏，可以考虑用 `cargo expand` 看展开结果

`cargo expand` 最有价值的地方不是“炫技”，而是：

- 它能把你从“这宏看不懂”拉回到“我至少能看展开后的普通 Rust 代码”

## 19. 一个非常实用的决策问题：这里真需要 DSL 吗

宏经常会诱导人写出看起来很高级的 DSL。

例如：

- 一长串特别像配置语言的调用形式

这有时很棒，但也经常带来问题：

1. 新读者不容易猜到行为
2. IDE 跳转和补全变差
3. 错误位置离真实原因更远
4. 文档成本上升

所以在工程里要反复问自己：

- 这里是为了提高使用体验，还是只是为了“写起来像框架”

## 20. 一个适合初学精通阶段的判断顺序

以后遇到“要不要用宏”时，你可以先按这个顺序问：

1. 函数能不能解决
2. 泛型能不能解决
3. trait 能不能解决
4. 构造函数或 builder 能不能解决
5. 只有前面都不自然时，再考虑宏

这条顺序非常值钱，因为它能帮你少走很多“宏先行”的弯路。

## 21. 常见错误与修正方向

### 错误 1：把宏当成“更高级的函数”

宏和函数工作的层次不同。

### 错误 2：为了省几行代码就引入宏

省几行不一定值得换来阅读和调试成本。

### 错误 3：看到 derive 和属性宏很多，就以为要自己马上写过程宏

大多数学习阶段先学会使用和判断就够了。

### 错误 4：宏报错时只盯调用点，不去想展开后是什么

很多时候真正问题在展开结果里。

### 错误 5：把 DSL 感当成设计质量

写得像框架，不等于边界就合理。

## 22. 小练习

1. 写一个 `sum!` 宏，接收多个整数表达式并返回总和。
2. 写一个 `make_pair!` 宏，把两个表达式展开成元组。
3. 观察 `vec![]`、`println!`、`format!`，分别思考它们为什么适合用宏。
4. 选一个你已经写过的小构造函数，判断它到底更适合函数还是宏。

## 23. 挑战题

设计一个“学习报告 DSL”的最小版本，要求：

1. 写一个 `study_report!` 宏
2. 输入里至少包含标题和多条学习记录
3. 展开后得到一个普通 Rust 结构体
4. 写出一版不用宏、只用构造函数/方法链的替代方案
5. 比较两版设计的优缺点

额外要求：

1. 至少解释一次：为什么这里宏不一定比函数更成熟
2. 至少解释一次：如果以后要对外发布 API，这个宏会增加什么维护成本

## 24. 小结

这一章最重要的不是立刻学会写复杂过程宏，而是先建立判断顺序：

1. 宏处理的是代码结构，不是普通值
2. `macro_rules!` 是最先该掌握的宏类型
3. derive 和属性宏是生态里非常重要的工程工具
4. 宏不是第一选择，函数、泛型、trait 往往应优先考虑
5. 宏真正值得用时，通常是为了样板收敛、DSL 或编译期结构生成
